# 拍照导入提示词设计方法论

> 用于教育类系统（如 AI家庭教师、Edu-Hub）的拍照导入功能设计。
> 核心原则：**提示词必须基于用户实际拍摄的图片设计，而非猜测课本长什么样。**

---

## 一、总体流程

```
用户需求分析 → 收集真实图片样本 
  → 分析每张图片的排版特征 → 按内容类型分组
  → 为每种类型设计提示词模板
  → 用实际图片逐张验证 → 迭代优化
```

## 二、提示词模板的三层结构

每个内容类型的提示词模板必须包含三层：

```
Layer 1 — 任务边界
  "你是XX内容提取器。只提取XX，忽略XX。"
  → 告诉模型做什么、不做什么

Layer 2 — 输出契约（JSON Schema）
  {"field": "type"}  + 严格的格式约束
  → 告诉模型怎么输出

Layer 3 — 质量约束
  "看不清楚输出 null，不要猜测"
  → 告诉模型不要瞎编
```

## 三、同类内容的不同子格式适配

同一个内容类型（如"会认字"）在现实中可能有多种排版：

| 子格式 | 特征 | 提取方式 |
|:-------|:-----|:---------|
| 两行式 | 拼音一行，汉字一行（生字表附录） | 一一对应 |
| 单行式 | 课号+汉字，拼音在相邻行 | 跨行匹配 |
| 内联式 | 汉字(拼音) 同一行内（识字表） | 括号内提取 |
| 练习式 | 汉字+拼音+组词列表（练习册） | 全字段提取 |

**提示词策略**：在模板中加入「排版适配规则」段落，让模型自动识别并选择对应的处理方式。不需要为每种子格式建单独的模板。

## 四、学期感知的数据模型

同一个字在不同学期的属性不同（会认字→会写字），数据必须以 **(学期, 字)** 为唯一键：

```python
# 错误做法：以字为唯一键
{"晨": {"pinyin": "chén", "type": "recognize"}}  # 只存一条，跨学期冲突

# 正确做法：以(学期, 字)为唯一键
{"3下/晨": {"pinyin": "chén", "type": "recognize", "words": ["早晨"]}}
{"4上/晨": {"pinyin": "chén", "type": "write", "words": ["晨光", "晨曦"]}}
```

### 导入流程

```
拍照 → 选择学期（必选）→ 选择内容类型 → 识别 → 预览确认 → upsert 入库
                                                          ↑
                                                  (semester, char) 为键合并
```

### 出题范围策略

```
每日练习 = 
  当前学期会认字（看字写拼音）      60%
  当前学期会写字（看拼音写字）      20%
  过往学期已变会写字（复习）         15%
  预习（未来学期会写字）              5%
```

## 五、内容类型设计原则

设计内容类型时，考虑三个维度的重叠：

1. **数据粒度的重叠** — 同一个字会从不同场景反复录入（生字表→会认字→看拼音写词语），每次数据粒度不同（仅拼音/拼音+组词/完整词语）。
   → 入库用 upsert 合并：补拼音、叠词组，不覆盖。

2. **语义属性的变化** — 同一个字的类型随学期变化（会认字→会写字）。
   → 按学期隔离数据，不跨学期混淆。

3. **输出结构的归一** — 不同类型输出统一结构，便于后端统一处理。
   → 所有字词类内容输出统一的 `{char, pinyin, words, semester}` 格式。

## 七、Schema 校验（Layer 4）

在提示词三层结构之后，增加第四层：**Schema 校验**。

```python
# 每个内容类型一个校验器
VALIDATORS = {
| `cn_recog` | validate_cn_recog | 检查 char/pinyin/words 完整性 |
| `shengzibiao` | validate_shengzibiao | 检查 lesson/chars 完整性 |
| `cishu` | validate_cishu | 检查 entries 非空 |
| `cn_highlight_words` | validate_cn_highlight_words | 检查 words 非空、source_text 非空 |
}
```

校验结果作为 `warnings` 返回前端展示（不阻断导入），让家长在预览阶段修正。

## 八、学期感知的 upsert 合并

### 合并规则

以 `(student_id, semester, char)` 为唯一键，三条合并规则：

1. **补拼音** — 旧记录缺拼音时补入
2. **叠词组** — 新组词追加到旧组词后，不覆盖
3. **升格类型** — `recognize`→`write` 自动升级（会认→会写）

```python
def upsert_chinese_char(student_id, char, pinyin, recog_type, words, semester):
    existing = db.find(student_id=student_id, semester=semester, char=char)
    if existing:
        if not existing.pinyin and pinyin:    # 补拼音
            updates.append("pinyin=?")
        if new_words:                         # 叠词组
            updates.append("meaning=?" + merged_words)
        if existing.type=='recognize' and recog_type=='write':  # 升格
            updates.append("recognition_type='write'")
    else:
        db.insert(...)  # 新增
```

### 跨学期隔离

同一个字在不同学期各自存储，互不影响：
- `3下/存 → type=recognize, words=[生存,存款]`
- `4上/存 → type=write, words=[保存]`

出题时按学期比例混合抽取。输入格式规范化学期：`"三年级下" → "3下"`。

## 九、验证方法

每个提示词模板设计后，必须用至少 2-3 张**真实的用户拍摄图片**做验证：

```python
# 验证流程
for each image in real_images:
    result = call_vision_api(image, prompt)
    parsed = json.loads(result)
    errors = validate_schema(parsed, content_type)
    
    accuracy = len(parsed["chars"]) / len(expected_chars)
    if accuracy < 0.8:
        refine_prompt()  # 迭代优化
```

### 完整验证管线

完整的端到端验证包括5步：应用启动 → 身份登录 → OCR识别 → 入库 → 按学期查询。
参考 `references/verification-pipeline.md` 获取可执行的curl命令序列。

> ⚠️ **中文字符URL编码陷阱**：通过curl CLI测试API时，查询参数中的中文字符（如 `semester=3下`）必须URL编码为 `semester=3%E4%B8%8B`，否则Flask无法正确解析参数，查询返回空结果。浏览器表单提交无此问题。


## 十一、大列表输出防截断：紧凑格式

当图片中含有大量条目（如词语表每张80-100个词），JSON 输出容易因 token 上限被截断。

**对策**：改用紧凑格式——让 OCR 输出逗号分隔的纯文本，而非 JSON 数组。

```python
# ❌ 易截断：JSON 数组（每个元素间的空格/换行消耗 token）
prompt = '输出JSON数组：["乌黑","活泼","春日","轻风"]'
# ← 超出上限时只有前20个元素，后60个全丢

# ✅ 紧凑：逗号分隔的纯文本
prompt = '输出格式：只需纯文字，词语用逗号分隔。如：乌黑,活泼,春日,轻风'
# ← 同样 token 量可以容纳全部元素
```

适用场景：词语表、单词列表等纯文本/纯内容的批量提取（每张图70-120个条目）。
不适用：需要结构化关系（如拼音+答案+提示字对应）的内容（每张图5-15组）。

### 平铺 vs 分组输出

词语表类的输出有**两种结构选择**，根据用户需求决定：

| 结构 | 示例 | 适合场景 | 对应类型 |
|:-----|:-----|:---------|:---------|
| 课号分组 | `[{lesson:2, words:["乌黑","活泼"]}]` | 学生要按课复习 | 传统词语表 |
| 扁平列表 | `["乌黑","活泼","春日",...]` | 纯字库、不关心课号 | 会写字词库 |

**用户确认前不要假设用哪种结构**——问清楚"课号要不要"再选。问了就记住。本例中用户明确说"课号可以忽略"，所以用扁平列表。但同一用户的不同材料可能结构不同。

### 词语表分词分隔符

词语表的排版多种多样，OCR输出中词语可能有不同分隔符。导入时必须**检查分隔符类型**并统一拆分：

| 分隔符 | 例 | 注意 |
|:-------|:---|:-----|
| 空格（最常见） | `乌黑 活泼 春日` | 逐空格split |
| 斜杠 | `赵州桥/石匠/设计` | 逐斜杠split |
| 课号混入 | `20 肥皂泡 种类 其中 网球` | 用 `not w.isdigit()` 过滤掉课号数字 |
| 跨行续接 | `赶集 聚拢 形成 加入\n春光 平添 稻田` | 先按换行split再按空格split |

> ⚠️ 同一张图内的词语可能混合使用空格和斜杠分隔。先用 `/` 替换空格再 split 可统一处理。另外注意中文词语最少2个字，单字或纯数字要过滤。

## 十二、关键流程纪律：观察先行，代码在后
> ⚠️ **本技能最重要的规则**——这条写在这里是因为它在同一个会话中连续被违反两次，每次都被用户当场指正。
> 
> 已经踩过的坑：你看完了也没用——你得**按正确顺序做**，不是知道顺序就行。

### 硬性流程（不可调换顺序）

```
┌─────────────────────────────┐
│  用户发送图片                │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│  ① 批量化观察（全部图一起看）│  ← 不是一张一张看
│  - 逐张提取原始文字（不解释） │
│  - 归纳排版类型（不猜意图）   │
│  - 一次性列出给用户          │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│  ② 用户确认排版类型          │
│  - "我看到X类排版，请纠正"   │
│  - 等用户回复后再继续        │
│  - 用户说了"整体方向没有问题"│
│    才代表确认通过            │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│  ③ 设计提示词模板            │
│  - 基于确认的排版写三层结构   │
│  - 用实拍图测试 → 调优       │
└──────────┬──────────────────┘
           ▼
┌─────────────────────────────┐
│  ④ 最后才固化代码            │
│  - 提示词 → 校验器 → 入库    │
│  - 路由注册 → 验证全链路     │
└─────────────────────────────┘
```

**永远不要从①或②跳到④。** 你改的数据库表结构、加的字段、写的批量导入函数，都可能因为用户一句话"其实不是那个排版"而全部作废。

**特别是：当步骤②中用户的确认信息和你原来的假设冲突时，立刻停下来重新观察。不要试图「修正一点继续」。**

### 批量看图原则 — 一次看完，归纳类型，再给用户

收到用户多张图片时，**一口气看完，归纳排版类型，一次性汇报**。这是之前用户认可的做法。

```
❌ 错误做法：
  图1跑完 → "这是对的吗？" → 用户回答
  → 图2跑完 → "这个呢？" → 用户回答 → ...
  → 用户反馈：'你为什么不一口气看完？'

❌ 更糟糕的错误做法：
  用户说"开始会写" → 直接改db加字段改路由
  → 用户发图 → 发现根本不是自己想的那种排版
  → 代码全白写

✅ 正确做法：
  6张图全部跑 OCR 看完 → 归纳排版类型
  → 一次性给用户："我看到三类排版：A、B、C"
  → 用户一次纠偏 → 问清楚"课号要不要""拼音从哪里来"
  → 确认 → 才开始设计提示词
```

### 用户分享信息 ≠ 技术指令（关键区分）

用户说一个数据特征时，**首先理解它是什么特征**，而不是立刻想着怎么实现它。

| 用户说 | ❌ 错误处理（当指令） | ✅ 正确理解 |
|:-------|:---------------------|:-----------|
| "它的拼音需要你补充" | 去装 pypinyin 库 | 哦，图片上没有拼音→我需要决定谁补（OCR还是后端库） |
| "课号可以忽略" | 删掉课号字段 | 哦，输出结构从课号分组改为扁平列表 |
| "开始会写" | 加部首/结构字段 | 哦，用户要处理会写类的内容→先看图确定长什么样 |

**区分方法**：问自己「这是数据本身的属性描述，还是技术实现方案？」如果是前者，**不要执行**，先理解。

### 「我又忘记了」模式识别

如果在处理图片导入时，用户说了以下任何一句，立刻停下当前正在做的所有代码工作：

> - "你忘记了我们刚才处理的目标和步骤了吗？"
> - "我发现你又忘记了原来的处理方式"
> - "你为什么不能和之前识别会认字的处理一样聪明"

**这些句子的意思是**：你跳过了①观察和②确认，直接进入了④代码固化。**回退到①重来。**

### vision_analyze 工具缺失时的应对

`vision_analyze` 工具（直接看图并描述画面内容）在 Hermes Agent 中有时可用有时不可用。当不可用时：

```
替代方案1：用 Qwen-VL-OCR + 描述性提示词
  prompt = '请描述这张图片的排版、内容分布、标题、行结构。'

替代方案2：用 OCR 原始文字提取 + 自己归纳
  prompt = '提取这张图片中的所有文字，保留原始排版顺序。'
  → 自己从结果中识别排版规律

替代方案3：Hermes 侧全管道自动化（批量场景）
  ① Python urllib.request 代替 subprocess+curl（规避 base64 参数溢出）
  ② 两遍走查：第一遍宽泛描述所有图 → 第二遍只定向追问不清晰的
  ③ 详见 `references/hermes-side-batch-analysis.md`

不推荐：逐张问用户"这是什么排版"
  → 用户有更好的事要做，这不是他们该干的工作
```

### 「用户整理好识别结果」的输入模式

除了"用户发图片→我观察分析"的标准流程外，还有一种情况：**用户自己整理了正确识别结果（结构化JSON/清单），直接作为提示词设计的参考。**

| 输入模式 | 典型场景 | 我的工作 | 注意事项 |
|:---------|:---------|:---------|:---------|
| 发图片→我分析 | 最常见的模式 | 观察排版→归纳类型→确认→设计 | 流程纪律十二控制 |
| 发整理好的识别结果 | 用户已手动识别，展示目标输出 | 直接基于结果推理图片排版→设计提示词 | 仍需用图片验证，不假设排版的全部细节 |
| 发图片+识别结果 | 用户同时给参考 | 比对图片和结果的一致性→设计与两者匹配的提示词 | 重点检查结果里有无遗漏（如拼音） |

**关键原则**：即使有正确识别结果，如果可能，还是要看图确认排版细节（如"粉色高亮"的视觉特征仅从文本结果看不出来）。无法看图时，在提示词中描述特征让模型自适应。

**提示词设计中的特殊考量**：当结果来源是"用户整理"而非"模型提取"时，要特别注意用户的整理方式可能隐含排版假设——比如用户把"溪边"既作为组标题又作为组内词语列出，提示词需要能区分标题和内容。

| 越位行为 | 表现 | 后果 |
|:---------|:-----|:-----|
| 看到图就猜类型 | "这是词语表！" → 动手加类型 | 用户说"1-3是会写字的练习"→ 加错了 |
| 用户说关键字就跳步 | 用户说"会写字"→ 直接改字段加提示词 | 改的字段对不上实际图片 |
| 用户分享信息当指令 | 用户说"拼音需要你补充"→ 去装 pypinyin | 方向错了，应该先看图片有没有拼音 |
| 没等确认就固化 | 改db/写路由/建模板 三连发 | 全部重来 |
| 逐张零散地问不批量看 | 一张图问一次"这是对的吗？" | 用户烦了，说"你为什么不一口气看完？" |
| 默认经验替代现场查看 | 以为"会写字"=单字带部首结构 | 实际是词语表，字段全对不上 |
| 长时间沉默后忘记上下文 | session恢复后直接动手 | 用户问"你忘记了步骤吗" |

### 5. 工具集变化的适应

❌ 看到「汉字+拼音+组词」就认为是会认字，或看到「组词+单字」就认为是字理速记
✅ **先确认表格的列标题**：
   - 列标题包含「汉字、组词、标识、页码/频率」→ 这是**会认字表**（`cn_recog`）——一个汉字对应多个组词
   - 列标题包含「组词、汉字、标识、频率」→ 这是**会写字组词拆解表**（`cn_write_table`）——一个完整组词拆出多个单字
   - 没有组词列，只有汉字+拼音 → 普通会写字单字表（`cn_write`）或生字表（`shengzibiao`）
✅ 判断依据是**列标题顺序**（组词在左列还是在右列）和**每行汉字个数**（1个汉字多个组词 vs 1个组词多个汉字）

## 十三、页码驱动的课文匹配模式

### 背景

当从课文页提取高亮词语时，用户纠正了三个关键设计假设（2026-05-16）：

| 用户纠正 | 我原来的假设 | 修正后的设计 |
|:---------|:------------|:------------|
| "不是每一页都有每课的名称" | 认为每页都有标题可以分组 | 输出结构自适应——有标题则分组，无则平铺 |
| "不是每个标记都是同样的颜色" | 试图区分粉色/其他颜色 | 标记提取**不区分颜色**，只看"有无标记" |
| "拍课文的同时是否也将课文作为资料留存" | 只提取词语不保存全文 | 双轨产出：词语入库 + 全文入库 |

### 页码锚定设计

页码是比课文标题更可靠的锚点：
- 课名可能在页眉、被裁剪、跨页才有，但页码每页都有且唯一
- 方案：OCR 提取页码 → 应用层查 `page_lesson_map` 表 → 自动补全课文标题和课号

```
OCR → 提取: {page_number, words[], source_text, lesson_title?}
        │
        ▼
应用层 → 查 page_lesson_map(student_id, semester, page_start <= page <= page_end)
        │
        ├─ 命中 → 自动补全 lesson_title + lesson_number
        └─ 未命中 → 保持 OCR 原值，留空待填
```

### 双轨入库架构

一次拍照产生两种独立产出：

| 轨道 | 目标表 | 存储内容 | 用途 |
|:-----|:-------|:---------|:-----|
| 轨道1 | `chinese_words` | 高亮标记的词语列表 | 出题引擎生成练习 |
| 轨道2 | `lesson_texts` | 课文全文 + 元数据 | 资料留存、阅读理解素材 |

### 新增数据库表

```sql
CREATE TABLE lesson_texts (
    student_id, semester, lesson_number, lesson_title,
    page_number, source_text, created_at
);

CREATE TABLE page_lesson_map (
    student_id, semester,
    page_start, page_end,  -- 页码范围
    lesson_number, lesson_title,
    UNIQUE(student_id, semester, lesson_number)
);
```

## 十四、课本照片的页码→课号映射预确认

### 场景定义

当收到多张课本照片且无法通过视觉确认每张图所属课号时（课本页眉无课号标识、OCR对课文标题的识别不可靠），必须先建立**页码→课号映射**，待用户确认后再进行OCR提取。否则可能跨课号提取数据，被用户要求「重新提取」。

### 核心原则

**提取页码先于提取内容。** 页码是比课文标题更稳定的锚点——课文标题可能被裁剪、跨页、或OCR误识别，但页码每页唯一。

### 标准流程（4步骤，不可调换）

```
Step 1 — 提取页码
  对每张图片用 Qwen-VL-OCR 直接询问页码数字
  → 得到每张图的 page_number（可能为null需二次识别）

Step 2 — 预分类
  将page_number映射到已知课号范围
  对OCR页码与预期不符的标为「⚠️待确认」

Step 3 — 编制映射总表，一次性呈现给用户
  格式：图片 → page → 推断课号 → 推断内容 → 状态
  标注所有待确认项，一次性列出（不逐张问）

Step 4 — 用户确认后，才进行内容提取
  按确认后的课号归属，使用对应提示词模板
```

### 用户修正格式

用户会用以下格式一次性修正页码范围：

```
课文名：pXX~pYY；课文名2：pXX~pYY
```

> **例**：你推断 L19 到 p77-p78，用户说"剃头大师：p74~P76；肥皂泡：P77~79；我不能失信：P80~81"
> → 直接替换映射表，不需要追问细节。修正后的表即最终版本。

### 特殊页面标记规则

| 页面类型 | 处理方式 | 示例 |
|:---------|:---------|:-----|
| 课文正文 | 标记「课文正文」，仅做资料留存 | L20 p77-79 |
| 课后生字页 | **焦点提取对象**（会认字+会写字） | L18课后 p73 |
| 课后练习页 | 检查底部是否有内嵌生字表 | L23课后 p90-91 |
| **习作/作文练习** | **非生字内容**，不提取字符 | 习作 p82, 作文练习 p94 |
| 语文园地 | 需提取识字加油站（有生字） | 园地七 p95-97 |
| 口语交际 | 非生字内容 | 劝告 p93 |
| 略读课文 | 只有会认字，无会写字 | L21 p80-81 |

### 页码确认后的完整性检查

确认完映射后，**立即检查每个精读课文是否有对应的课后生字页**：

如果精读课文缺课后页（如L20肥皂泡那组照片未拍到会认字页），在映射表中标注**「⚠️课后页缺失」**，呈现给用户时醒目红色提示，询问是否需要从课本附录补充。

### 呈现给用户的映射表模板

使用HTML暗色科技风信息图呈现。列：页码 → 课号 → 课文名 → 内容类型 → 状态。待确认项用红色虚线框。课后缺失页用 ⚠️ 标记。

### 与内容提取的关系

| 阶段 | 工具 | 产出 |
|:-----|:-----|:-----|
| 预确认（当前流程） | 简单页码询问提示词 | 页码→课号映射表 |
| 内容提取（后续流程） | 各内容类型专用提示词模板 | 结构化字词数据 |

预确认阶段只用简单询问提示词提取页码，**不**在此阶段做内容级的OCR提取——那会浪费token，且用户可能纠正映射导致提取全部作废。

## 十五、批量课本练习页预审与生字词提取（特殊场景）

### 场景定义

当用户拍摄的不是独立的生字表附录，而是**课本课文后的练习页**时，生字表和词语表**嵌入在页面底部**，与练习题文本混排。这不是标准的 `shengzibiao` 或 `cn_words` 输入，需要**两步走流程**。

### 两步走流程

```
Step 1 — 批量预审（所有图一起跑 OCR）
  用宽泛的描述性提示词提取每张图的内容类型+全部文字
  → 归纳出哪些图是「课文正文」、哪些是「练习页内含生字表」
  → 一次性呈现给用户确认分类

Step 2 — 定向精提取（确认后执行）
  只对有生字词的内容做结构化提取：
  ├─ 练习页的会认字区域 → 格式：tì 剃 zhí 执 fǒu 否（拼音+字配对）
  ├─ 练习页的会写字区域 → 格式：表胆骂仇惯差刑付倍（纯字列表）
  └─ 课后词语 → 直接提取词条
  课文正文 → 只留作资料（不入 chinese_chars/chinese_words）
  语文园地/口语交际 → 有字词的才提取
```

### 练习页内嵌生字表的排版特征

| 区域 | 视觉特征 | 提取策略 |
|:-----|:---------|:---------|
| **会认字栏** | 汉字上方或旁边有拼音，通常为「拼音+空格+汉字」配对出现，如 `tì 剃 zhí 执` | OCR 识别后提取汉字+拼音配对；拼音含声调标记 |
| **会写字栏** | 无拼音的纯汉字列表，每个字独立，如 `表胆骂仇惯差刑付倍` | 直接提取汉字列表；拼音由后端补充（或存空待补） |
| **会写字扩展栏** | 会写字延伸组词或用法，如 `加倍 虽然 厘米` | 提取后判断是会写字的组词还是独立词语 |
| **课后词语栏** | 课文结束后独立的词语列表（见词语表附录） | 按词语表流程处理 |

> ⚠️ **「认识词语」表格的多字拼音陷阱**：课本课后练习页中常有「认识词语」表格（认识+读一读），排版为**一个词语占一行**，拼音标注在词语上方，但OCR容易只提取最后一个字的拼音作为整词拼音（如词语"墨水"OCR结果为"mò"而非"mò shuǐ"）。**根因**：OCR提示词未要求输出每个字独立拼音，模型自动简化为标记最后一字。**缓解**：在提示词中加入「每个词语必须输出完整拼音，每个字一个拼音段，空格分隔」约束。如识别后仍有缺失，用 `pypinyin` 从 word 字段重建全拼音。三年级下册第18课（童年的水墨画）已有 10 个词受此影响。

### 批量 OCR 描述性提示词模板

当需要预览批量图片时，使用此宽泛提示词代替结构化提示词：

```
请详细描述这张图片的所有内容（用中文）：
1. 这是什么类型？（课文正文/生字表/会认字表/会写字表/词语表/练习题/语文园地/口语交际等）
2. 课号/单元号？
3. 如果是课文：提取标题、作者、全文
4. 如果有生字/会认字：列出每个字及其拼音（精确配对）
5. 如果有会写字：列出所有会写字
6. 如果有词语：列出所有词语
7. 提取这张图上的所有可见文字，不要遗漏
```

### 课本练习页 vs 独立附录生字表 的核心差异

| 维度 | 独立附录生字表 | 练习页内嵌 |
|:-----|:--------------|:-----------|
| 排版清晰度 | 表格结构清晰，列标题明确 | 混在练习题文字中，需要手动剥离 |
| 是否含课文 | 不含 | 含课文摘要/全文 |
| 是否含练习题 | 不含 | 含完整练习题（需忽略） |
| 会认/会写是否分栏 | 通常分两栏标注 | 通常在同一块区域，先会认后会写 |
| 提取难度 | 低（直接 JSON） | 高（需人工筛选词汇区域） |

### 已知局限

1. **Qwen-VL-OCR 对拼音声调的精度**：部分声调标记识别不准确（二声/三声混淆）。批量预审阶段可接受，精提取阶段需人工校正。
2. **混排干扰**：练习题中的自然文本可能包含汉字列表被误识别为会写字。后处理需用字数阈值过滤（单字列≥3个连续单字才视为有效会写字区域）。
3. **语文园地的识字加油站**：虽是课本附录内容，但含独立生字（如 `旭屿瞭巡缆锚`），需要单独提取。见 `references/unit6-7-textbook-vocab-extraction.md` 示例数据。

- `references/unit6-7-textbook-vocab-extraction.md` — 三年级下册第6、7单元课本的全部生字词提取数据（本次会话产出）

### 设计启示

设计新的内容类型时，不要默认任何排版特征一定存在。每次先向用户确认：

1. **分组依据** — 是否有标题/课号？还是平铺？（不是每页都有课名）
2. **提取条件** — 有颜色分类意义吗？还是只看标记有无？（颜色不重要）
3. **产出范围** — 用户只想要提取的主产物？还是连带上下文也要保留？（双轨可能优于单轨）
4. **锚点选择** — 提取的信息中哪个字段是最稳定可靠的？（页码>课名>猜测）

## 十六、内容类型数据模型（含出题逻辑）

> 本章节整合自 `chinese-content-architecture` skill（已归档）。覆盖内容类型分类、数据模型字段定义、文本导入格式、出题引擎过滤逻辑。与本章节中的提示词设计构成完整的内容输入→存储→出题链路。

### 1. 内容类型分类（含考核方式）

| 编号 | 内容类型 | 考核方式 | 导入方式 | 练习题型 |
|:----:|:---------|:---------|:---------|:---------|
| 1 | 会认字(recognize) | 看字写拼音 | 拍照/文本 | `char_to_pinyin` |
| 2 | 会写字(write) | 看拼音写字 | 拍照/文本 | `pinyin_to_char` + `meaning_to_char` |
| 3 | 试卷(exam) | 综合 | 拍照 | 自动拆解分流到各类型 |
| 4 | 背诵诗词(poem) | 默写/填空 | 拍照/文本 | `poem_fill` + `poem_full` |
| 5 | 阅读理解(reading) | 文章+问题 | 拍照/文本 | 选择题/判断题/简答题 |

### 2. 数据模型（content.json 字段定义）

**characters 字段** — 核心原则：数据字段必须与提示词输出字段一一对应。

```json
{
  "char": "存",
  "pinyin": "cún",
  "recognition_type": "recognize",     // "recognize" | "write"
  "word_examples": [
    {"word": "生存", "pinyin": "shēng cún"},
    {"word": "存款", "pinyin": "cún kuǎn"}
  ],
  "source": "photo_import"
}
```

| 提示词输出 | 对应字段 | 说明 |
|:----------|:--------|:-----|
| 核心汉字 | `char` | ✓ |
| 汉字拼音 | `pinyin` | ✓ |
| 组词及拼音 | `word_examples[]` | ⚠️ 旧字段 `meaning` 只存文字，丢失拼音 |
| 会认/会写归属 | `recognition_type` | 由前端分类选择 + 提示词共同决定 |

**向后兼容**：旧 `content.json` 的无 `recognition_type` 字段的字默认视为 `"write"`（全题型出题）。

**poems 字段**：
```json
{
  "title": "静夜思",
  "author": "李白",
  "dynasty": "唐",
  "text": "床前明月光，疑是地上霜。举头望明月，低头思故乡。"
}
```

**readings 字段**：
```json
{
  "title": "金色的草地",
  "passage": "我们住在乡下，窗前是一大片草地……",
  "questions": [
    {"id": 1, "text": "草地为什么是金色的？", "type": "choice|short|judge", "answer": "..."}
  ],
  "difficulty": "⭐⭐",
  "source": "photo_import"
}
```

### 3. 文本导入格式（`import_from_text()`）

适用于手动粘贴和家长编写的结构化文本：

```
课时: Y3U1
标题: 第一课
会认字: 晨-chén-早晨(zǎo chén),绒-róng-绒毛(róng máo)
会写字: 付-fù,出-chū
生字: 旧-jiù (backward compat)
词语: 早晨-zǎo chén
古诗: 静夜思-唐-李白-床前明月光
```

**关键规则**：
- `会认字:` 和 `会写字:` 使用 **追加模式**（`+=`），同一课时内可以共存
- `生字:` 使用 **覆盖模式**（`=`），兼容旧导入行为
- 顺序：先出现的类型先处理，后出现的追加，除非碰到 `生字:` 则覆盖

### 4. 出题引擎过滤逻辑

```python
def generate_questions(characters, student_id, batch_id, count=12):
    for c in selected:
        recognition_type = c.get("recognition_type", "")

        if recognition_type == "recognize":
            qtype = "char_to_pinyin"          # 会认字→只看字写拼音
        elif recognition_type == "write":
            r = random.random()
            qtype = "pinyin_to_char" if r < 0.6 else "meaning_to_char"  # 会写字
        else:
            qtype = random.choice(["char_to_pinyin", "pinyin_to_char", "meaning_to_char"])
```

**频次模型不受影响** — 所有字符共用同一套 error_count → freq_level 逻辑，区别只在题型范围。

### 5. 旧数据标准化（兼容性）

```python
def normalize_character(char: dict) -> dict:
    result = dict(char)
    if "recognition_type" not in result or not result["recognition_type"]:
        result["recognition_type"] = "write"  # 旧数据视为会写字
    if "word_examples" not in result:
        result["word_examples"] = []
    if not result["word_examples"] and char.get("meaning"):
        result["word_examples"] = [{"word": char["meaning"], "pinyin": ""}]
    return result
```

所有读取数据的接口必须在返回前调用 `normalize_character()`。

### 6. 拍照导入不走文本解析

拍照导入直接调用 `upsert_batch(characters=characters)`，**不经过** `import_from_text()` 的文本解析层。

## 十七、考试试卷 OCR（新增内容类型）

### 与课本 OCR 的核心差异

| 维度 | 课本 OCR（shengzibiao/cn_recog 等） | 试卷 OCR（exam_paper） |
|:-----|:-----------------------------------|:----------------------|
| 内容结构 | 单一类型统一排版（生字表/词语表） | 混合多题型（选择/填空/阅读/作文） |
| 数据类型 | 结构化字词 | 题干+选项+答案+评分 |
| 主观答案 | 无（纯数据提取） | 有手写答案（需保留原样） |
| 页码关系 | 单页独立 | 双面关联（正面→背面） |
| OCR 输出 | JSON 数组，可严格校验 | 自由文本+结构化混合 |
| 幻觉风险 | 低（有格式约束） | **高**（阅读理解段落易编造） |

### 典型试卷结构

```
2025学年（下）三年级语文三四单元综合练习
姓名：___  学号：___

一、基础知识
  1. 看拼音，写词语。     → pinyin_to_word 型
  2. 选读音相同的一项    → 选择题 (A/B/C/D)
  3. 选有错别字的一项    → 选择题
  4. 选最合适的开头      → 选择题
  5. 选正确语句填入      → 选择题
  6. 课文内容填空         → fill_blank 型
  7. 修改病句              → 主观题
  8. 按要求改句子          → 主观题

二、阅读芳草地
  （一）短文标题          → 阅读理解（含短文正文+子小题）
  （二）短文标题          → 阅读理解

三、作文（如有）
```

### 试卷 OCR 的两步走策略

第一步：提取卷面结构（宽泛描述）
```
"请描述这张试卷的：
1. 试卷标题、姓名学号区域
2. 所有大题标题（一、二、三...）及包含的小题号
3. 每个小题的题型（选择题/填空题/问答题/阅读题）
4. 选择题的选项（A/B/C/D）
5. 学生手写的答案内容（保留原样）
6. 页码数字"
```

第二步：分类定向精提取（按题型分别处理）
```
对于选择题：提取题干+4个选项+学生所选答案
对于填空题：提取题干空格+学生填写内容
对于阅读理解：提取短文原文+每个问题+学生答案
对于作文：提取题目+学生全文（如有）
```

### 已知陷阱

**陷阱1：阅读理解段落幻觉（必须警惕）**
Qwen-VL-OCR 在处理阅读理解短文时，常常「补充」原文中没有的文字。表现：OCR 输出的短文内容比真实试卷长数倍，含大量编造的推理文字。例如：「耳朵上的茧」阅读题输出了一段关于莱斯·布朗生平的长篇虚构。

**缓解策略**：
- 两步提取：第一步只提取题号+题目文本（不提取短文），第二步用单独调用提取短文原文
- 长度约束：在提示词中加「只输出试卷上可见的文字，不要补充」
- 后续比对：对阅读理解短文用课文正文数据源做校对（如果课文已在 lesson_texts 中）

**陷阱2：选项提取不完整**
试卷选择题的 A/B/C/D 选项因排版紧凑，OCR 可能漏提取中间选项。缓解：在提示词中明确要求「一一列出每个选项的全部文字，包括 A. B. C. D. 前缀」

**陷阱3：手写答案覆盖题干文字**
学生手写答案可能覆盖印刷体文字，导致 OCR 混淆「题干」和「答案」。缓解：以「手写区域位置」判断哪个是答案（通常在横线/括号/空格内）。

**陷阱4：双面文档关联**
正反面照片可能独立导入，需通过「试卷标题+姓名学号」关联两面。缓解：导入时要求用户先选择正面照片，再选背面照片，由同一 paper_id 关联。

**陷阱5：批改标记干扰**
教师用红笔打的 ✓/✗/半对标记，OCR 可能误读为文字笔画。缓解：OCR 阶段不要求批改标记识别，由后续服务层从 is_correct 字段逻辑判断。

### 多模型协作模式

#### 场景：彩色背景词语检测

Qwen-VL-OCR 不识别彩色背景标记（课本高亮词语）。需要配合颜色识别模型：

| 职责 | 模型 | 特长 | 成本 |
|:-----|:-----|:-----|:----|
| 常规文字提取 | Qwen-VL-OCR | 快速文本识别（~1s/张） | ¥0.014/张 |
| 彩色标记检测 | Qwen3-VL-Plus | 可分辨粉色/黄色/蓝色背景的词语 | ¥0.0017/张 |

**架构模式**：双轨并行 — 先 Qwen-VL-OCR 提取全文，再用 Qwen3-VL-Plus 检测有背景色的词语。两条结果通过 `page_number` 字段关联。

### API 调用示例（原生 urllib）

```python
import json, urllib.request

body = {
    "model": "qwen-vl-ocr-latest",
    "messages": [{
        "role": "user",
        "content": [
            {"type": "text", "text": prompt},
            {"type": "image_url", "image_url": {
                "url": f"data:image/jpeg;base64,{compressed_b64}",
                "min_pixels": 32*32,
                "max_pixels": 32*32*2048,
            }},
        ]
    }],
    "max_tokens": 2048,
    "temperature": 0.1,
}

req = urllib.request.Request(base_url, data=json.dumps(body).encode("utf-8"),
    headers={"Authorization": f"Bearer {api_key}", "Content-Type": "application/json"},
    method="POST")
resp = urllib.request.urlopen(req, timeout=60)
result = json.loads(resp.read())
text = result["choices"][0]["message"]["content"]
```

### 关键陷阱

1. **图片压缩策略**：Qwen-VL-OCR 建议 JPEG 1024px@85%；Qwen3-VL-Plus 需要 PNG 原图（不压缩）以保留颜色细节
2. **系统提示词合并**：Qwen-VL-OCR 不支持 `system` 角色，必须将 System Prompt 拼入 User 的 text
3. **JSON 输出要求**：提示词必须说明"只输出JSON，不要额外文字"
4. **API Key 加载**：terminal() 的 Python 子进程不自动 source `~/.bashrc`，需要在代码中备选读取
5. **中文渲染测试**：PIL 默认字体不渲染中文，需指定字体（如 `wqy-zenhei.ttc`）
