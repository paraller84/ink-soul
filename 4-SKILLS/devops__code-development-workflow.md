# 标准化编码开发工作流 v5.0.0 — 交付物驱动模式

> 所有涉及编码的任务，必须按此流程执行。
>
> v5.0.0 新增：
> - **Aider 全等级强制编码** — 所有 Tier 0/1/2/3 的编码任务（含单文件 Bug 修复）必须通过 Aider 委托执行。Hermes 停止直接写代码，职责收缩为架构设计、验收条件定义、审计与交付。
> - **新增参考文件**：`references/handwriting-types-pinyin-integrity-check.md`（手写题型拼音排查方法）、`references/flask-flash-messages-base-template.md`（Flask 模板 flash 消息渲染陷阱）。

## 第一节（前置）：进度可视化面板（全流程导航）

> **任务执行期间，每次完成一个步骤后，必须展示进度可视化面板。**
>
> 这是对所有 C005 等级（Tier 0-3）的**强制行为要求**。
>
> 核心原则：**用户永远知道「我在哪」「到哪了」「还剩什么」**。

### 1.0 每次步骤切换时的进度面板模板

**执行完每个 C005 步骤后**，在开始下一步之前，输出以下格式的进度面板：

```
📊 C005 进度 — [任务短描述] (Tier N)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Step 0: ✅ 需求分析            已完成
  Step 1: ✅ 代码学习            已完成
  Step 2: 🔄 架构设计            ◀ 进行中
  Step 2.5: ❏ 骨架搭建         待开始
  Step 2.6: ❏ UI/UE设计(按需)   待开始
  Step 2.7: ❏ 原子验收条件      待开始
  Step 3: ❏ 工程实现            待开始
  Step 4: ❏ 质量门禁            待开始
  Step 5: ❏ 交付                待开始
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ● 已完成: [x]步   ○ 剩余: [y]步
```

### 1.1 进度面板的上下文放置规则

- 进度面板**应放置在消息的结尾**（即执行完步骤、展示结果、处理完用户反馈之后，再展示进度板，在进入下一步前让用户知道后续还有哪些步骤），这样用户看完本步产出后立刻看见接下来的路线。
- ⚠️ 不要在消息开头放进度面板（用户会困惑于内容怎么还没出，先看到一个进度条）；应该先呈现结果，再展示进程。
- ⚠️ 不要把进度面板放在折叠块（如 `<details>`）的里面，用户可能只看折叠标题就返回，错过后续进度信息。

### 1.2 进度面板色彩规范

在信息图中使用严格统一的色彩代码，遵循各技能通用的暗色科技风主题色卡：

| 状态 | 色号 | 色值 | 
|------|------|------|
| 已完成 | 绿色 | `#4ade80` |
| 进行中 | 蓝色 | `#3b82f6` |
| 待开始 | 灰色 | `#6b7280` |
| 状态头 | 黄色 | `#fbbf24` |

严禁使用其他近似色（如用 `#22c55e` 替代 `#4ade80`），确保跨技能视觉统一。

## 第一章：核心体系（五步工作流）

此体系**只有** 5 个步骤加按需的中间步骤和特殊的 Step 0。执行时严格编号，不可增删步骤序号（避免引用断裂），不可跳跃顺序。

```
Step 0 （前置）— 需求分析（Tier 0 必做，Tier 1/2/3 可选）
  ↓
Step 1 — 代码学习（Tier 0/1 必做，Tier 2/3 可选）
  ↓
Step 2 — 架构设计（Tier 0/1 必做，Tier 2/3 可选）
  ↓
Step 2.5 — 骨架搭建（按需）
  ↓
Step 2.6 — UI/UE 设计（按需）
  ↓
Step 2.7 — 原子验收条件（所有 Tier 必做——见本章第七节）
  ↓
Step 3 — 工程实现（所有 Tier 强制执行）
  ↓
Step 4 — 交付物+质量门禁（所有 Tier 强制执行）
  ↓
Step 5 — 交付（所有 Tier 强制执行）
```

### 第一节：Step 0 — 需求分析

**并非每项任务都需要完整的 Step 0 分析。** Tier 0 任务（新系统）和 Tier 1 任务（大功能）**必须**执行完整的 Step 0，Tier 2（小功能）和 Tier 3（Bug）可选。

**Tier 0/1 的 Step 0 产出物：**
- `docs/requirements-[功能名].md` — 需求清单
- `docs/glossary-[功能名].md` — 术语表（如果有复杂术语）

**Tier 2/3 的 Step 0：**
- 直接向用户确认需求，不需要完整的需求文档
- 1-2 句话确认即可

---

#### Step 0-A：需求预检（三级检查）

**触发条件：** 所有 Tier 0/1 任务和涉及已有系统的 Tier 2 修改。

此阶段在需求分析启动前执行，持续至发现以下情况之一：

**第一级 — 设计文档预检（Tier 0/1 必做）**

在开始需求分析前，先检查用户是否已有设计文档（飞书云盘、项目 docs/ 目录、skill references/ 目录）。

- 如已有，**直接加载**并进入下一步，不重新生成
- 如无，按正常需求分析流程进行

**第二级 — 全量清单预检（Tier 0/1/2 必做）**

不要只处理用户当前提及的问题，应同步检查周边相关功能：

- 用户提到 A → 主动询问是否需要查看 A、B、C、D 的清单
- 例如生成练习卷时，不只看题目，还要检查数据流（从出题到入库到展示到批改）
- **检查维度：**
  - [ ] 全量页面清单（所有路由+模板）
  - [ ] 全量数据流（输入→处理→存储→展示）
  - [ ] 全量角色视图（学生/家长/管理员各能看到什么）
  - [ ] 全量失败路径（空数据、无权限、网络错误等）

**第三级 — 需求收敛预检**

同一功能不要交付超过 3 个需求点：

- 如一次交付超过 3 个需求点 → 拆分为多个子任务分别交付
- 每个交付单元在交付前先给用户过目，用户确认方向后再继续

**第四级 — 先观察再动手升维预检**

当需求涉及**数据处理（图片/文本/表格等）** 且需要编写新的处理逻辑时：

- **强制观察**：先审视所有输入数据（图片排版、文本格式、表格结构等）
- **确认理解**：向用户描述观察结果，确认理解正确
- **再动手**：确认后再开始设计提示词/编码

**第五级 — 数据契约先行（v4.3.0 新增）**

当一个需求包含**新的 API、新的接口函数、或参数格式变更**时，必须：

- 在 Step 2（架构设计）阶段**先定义入参和出参的 Schema**（JSON Schema、SQL DDL 或 Python TypedDict）
- 将 Schema 输出为独立文件（如 `docs/api-schemas/[feature]-schemas.json`）
- 在 Step 3（工程实现）中直接使用该 Schema 验证实现
- 此预检旨在消除「接口双方理解不一致」导致的数据偏差

**第六级 — 先方案确认再改 artifact（v5.1.1 新增）**

**触发条件：** 任何涉及**修改已有 artifact**（技能文件、设计文档、系统脚本）的任务。

**规则：** 在修改任何 artifact 之前，必须先形成完整方案并向用户确认，确认后才开始修改。

```
发现需要修改的 artifact
    ↓
形成完整方案（含变更清单、预期效果、影响范围）
    ↓
呈现给用户确认
    ↓
用户确认后才开始执行修改
```

**为什么必须这样？**

- 用户偏好「先看到全貌再决定」——碎片化的逐步展示会迫使用户在小决策上反复做判断
- 修改 artifact 是不可逆的——即使能回滚也已造成上下文污染
- 实证：一次 30 步的决策链中，前 5 步可能有 3+ 次方向调整。如果每一步都开始改 artifact，会有大量无效变更需回滚

**常见违规模式：**

| 违规行为 | 正确做法 |
|:---------|:---------|
| 看到需求就 `patch` 开始改 SKILL.md | 先完整设计方案，全部确认后一次性修改 |
| 边设计边创建脚本文件 | 先完整设计 + 用户确认，再创建所有文件 |
| 改完一个文件就问一次"这个对吗" | 全部改完、一次性展示结果让用户审阅 |

**例外：** 紧急 bug 修复（系统不可用、数据丢失风险）。此情况下可直接修复，但修复后需补充说明。

**异常发现处理原则**

如果 Step 0-A 预检中发现遗漏项：

1. **记录到暂存区**（`docs/scope-creep-[功能名].md`）
2. 不要在当前任务中追加处理
3. 在当前交付完成后，向用户提出新增任务建议

---

#### Step 0-B：负载等级判定

负载等级决定 Step 1 和 Step 2 的执行深度。

| 等级 | 条件 | Step 1 | Step 2 |
|:----:|:-----|:-------|:-------|
| **Tier 0** | **新系统**（全新项目、大规模重构） | 必做 | 必做 |
| **Tier 1** | **大功能**（新增 5+ 路由、新增数据库表、修改核心流程） | 必做 | 必做 |
| **Tier 2** | **小功能**（新增 1-5 个路由、不涉及数据模型变更、非核心流程修改） | 可选 | 必做 |
| **Tier 3** | **Bug 修复**（已知问题修复，已有代码修改，无功能新增） | 可选 | 可选 |

---

#### Step 0-C：执行者判定（v5.2.0 增强 — 硬性拦截机制）

**核心原则：所有编码任务必须通过 Aider 委托执行（`delegate_task` + `aider-wrapper`）。**

无论任务等级，Hermes 不直接写代码。Hermes 的职责是：
- 架构设计（Step 2）
- 验收条件定义（Step 2.7）
- 审计与质量门禁（Step 4）
- 交付（Step 5）

Aider 的职责是：
- 工程实现（Step 3）

| 等级 | 执行者 | 适用场景 |
|:----:|:-------|:---------|
| **Tier 0** | **Hermes 架构 → Aider 编码**（WSL 环境 Aider 已验证稳定，Codex 有 sandbox/bwrap 问题） | 新系统、大规模重构 |
| **Tier 1** | Hermes 架构 → Aider 编码 | 中大功能 |
| **Tier 2** | Hermes 简单架构 → Aider 编码 | 小功能 |
| **Tier 3** | Aider 直接编码（Hermes 提供上下文包） | Bug 修复 |

**为什么所有层级都用 Aider？**

用户明确期望所有编码任务使用 Aider 执行，即使单文件修改。理由：
1. Aider 的 repo-map 跨文件感知更可靠，避免"改一处漏三处"
2. Aider 自动 git commit 创建可回溯的工作记录
3. 统一编码工具链降低维护成本
4. 用户对直接编码的偏差容忍度低——Aider 的自动化审查能减少偏差

**例外：** 临时调试脚本、一次性数据分析、纯查询不修改代码的小工具——这些仍可由 Hermes 直接编写。

---

#### Step 0-D：Aider 编码强制自检（v5.2.0 新增 — 硬性拦截规则）

**本规则不可绕过。违反即属于本技能规范违规，需立即纠正。**

**规则：在进入 Step 3（工程实现）之前，必须先执行一次「Aider 编码自检」——确认本次编码任务是否已走 `delegate_task → Aider` 流程。**

**自检方式：**

在 Step 2.7（原子验收条件定义）完成后、Step 3 开始前，自我执行以下检查：

```
检查 1：我是否准备对 .py / .html / .js / .css / .vue / .ts 文件进行写操作？
├─ 是 → 必须走 delegate_task → Aider，不可直接 write_file 或 terminal 编辑
└─ 否 → 继续
检查 2：本次修改涉及 3 行以上代码？
├─ 是 → 必须走 Aider
└─ 否 → 可以直接写（但需标明为"临时调试"并后续清理）

如果检查 1 和检查 2 都指向 Aider → 但我发现自己没有主动触发 delegate_task：
  → 立即终止当前编码路径
  → 强制切换到 delegate_task(task="编码", toolsets=["terminal","file"])
  → 不允许用"改几行而已"或"先改一下再看"找借口绕过
```

**为什么 Aider 自检必须由我（Hermes）执行而不是自动化脚本？**

因为绕过 Aider 发生时，我（Hermes）是**唯一知道自己绕过 Aider 的人**。没有脚本能检测我「打算绕过 Aider」。所以自检是最后一道防线。

**违规后果（自损机制）：**

如果用户发现我绕过了 Aider 直接写代码：
1. 该次交付自动标记为 **❌ 交付违规**
2. 立即回滚所有未走 Aider 的代码变更
3. 重新走 delegate_task → Aider 流程重新实现
4. 将本次违规记录到 memory：`{date}: 编码违规 — 绕过Aider直接编码，已回滚并重做`

**例外场景（允许直接写代码的清单）：**
- 终端测试脚本（`test_*.py` 仅用于验证，不进入生产代码目录）
- 一键执行的数据分析脚本（`analyze_*.py`，运行即删）
- `.md` 文档文件
- 配置文件（`.yaml` / `.json` / `.toml` / `.ini` 仅修改已存在的键值，不新增功能逻辑）
- `docs/` 目录下的非代码文件

### 第二节：Step 1 — 代码学习

#### 1.1 代码学习的三种策略

基于负载等级选择策略：

| 策略 | 适合 | 方法 |
|:----:|:-----|:------|
| **深度探查** | Tier 0（新系统） | 逐文件读全部代码 |
| **定向学习** | Tier 1（大功能） | `grep` + `tree` + 关键文件读取 |
| **差异学习** | Tier 2/3 | 只看变动的上下文 |

#### 1.2 最佳实践

- 避免大规模串行 `read_file()`，优先用 `terminal("grep -rn 'xxx' --include='*.py'")` 定位
- 文件稀疏查找时，`find . -path ...` 比 `grep -r` 高效
- 输出代码学习结果时，不要输出整块代码，应精炼为「功能→路径→关键行」的表格
- 学习已有代码时，注意识别「设计信标」——那些命名规范一致、职责单一的文件/函数，它们是理解全系统架构的锚点

### 第三节：Step 2 — 架构设计

这个步骤的产出物是**架构文档**，产出物格式参见 `design-document-production` skill 的文档类型 A/B/C。

**⚡ 此步骤有一个核心原则：骨架先行。**

**什么是骨架先行？**

> 在填充业务逻辑（Step 3）之前，先建立完整的代码骨架——文件结构、路由注册、基础的类和函数签名。

**为什么必须是骨架先行？**

在「一次性全量」执行模式下（用户偏好的 B/A/A 风格），骨架先行起到了 **「最小可行性架构」** 的作用：

1. 让架构决策在填充逻辑前被验证
2. 避免"逻辑写完了发现路由设计错了"的返工
3. 骨架文件本身就是 Step 3 的「工程实现清单」

**如何执行骨架先行？**

**Step 2 输出物必须包含：**

1. **文件结构图** — `tree` 格式，标注哪些文件是新创建的，哪些是修改的
2. **路由清单** — Flask/其他框架的完整路由注册表（URL → 蓝图 → 函数名）
3. **数据流图** — 核心数据流向（用户输入 → 处理 → 存储 → 展示）
4. **类/接口签名** — 新增的类、函数、API 的签名（不含实现体）

**Step 2 的中间步骤（按需执行）：**

- **Step 2.5：骨架搭建** — 创建空的类、路由注册、函数签名
- **Step 2.6：UI/UE 设计** — 涉及前端界面时，输出设计原型（独立文件，非内嵌文档）

---

### 第七节（新增）：Step 2.7 — 原子验收条件定义

**触发条件：所有 Tier**

在每个任务的 Step 2（架构设计）完成后、Step 3（工程实现）开始前，必须为每个原子任务产出「原子验收条件」清单。

#### 什么是原子验收条件？

原子验收条件是**可二进制判定**（通过/不通过）的检查点，每条条件不需要主观判断即可验证。

**正确的写法：**
```
✅ 路由 POST /practice/<id>/submit 返回 200 + {"status": "ok"}
✅ 数据库表 practice_answers 包含 answer_choices TEXT 列
✅ 前端模板 practice.html 在第 45-60 行有错误提示区域
```

**错误的写法：**
```
❌ 练习提交功能正常运作（太模糊）
❌ 用户应该能提交答案（不能直接验证）
❌ 代码质量较高（主观判断）
```

#### 验收条件的层级

每个原子任务必须包含以下三类验收条件：

| 层级 | 检查维度 | 数量要求 | 示例 |
|:----:|:---------|:---------|:------|
| L1-文件路径 | 文件是否在正确位置 | 1-3条 | `routes/practice.py` 存在 |
| L2-行为契约 | 接口/函数的入参出参是否匹配设计 | 3-8条 | POST 返回 status 字段 |
| L3-边界覆盖 | 空值、错误、异常是否处理 | 2-5条 | form 提交返回重定向而非 JSON |

#### 发布验收条件的时机

验收条件清单在 Step 2.7 中产出，**存储为独立文件**（如 `docs/acceptance-criteria-[task].md`）。

在 Step 3（工程实现）开始前，验收条件清单需要经过以下确认：

1. **让我自己过一遍：** 每一条条件是否可二进制判定、是否无歧义、是否可自动化验证
2. **让审核 SubAgent 过一遍：** 如果使用多 Agent 模式（见下节），审核 SubAgent 只持验收条件清单（不持设计文档全文），做"冷酷"的条件比对

#### 长反馈回路预防机制（v4.3.1 新增）

**触发条件：** 当单次编码窗口预计超过 30 分钟时。

**规则：** 强制将编码窗口拆分为多个可单独验证的原子任务，每个任务不超过 30 分钟。

**操作方式：**

```
原子任务预计时长 ≥ 30 分钟 → 拆分
    ↓
原子任务 1 (20min): POST route + 数据库写入    → 验收后继续
原子任务 2 (15min): GET route + 前端展示        → 验收后继续
原子任务 3 (10min): 错误处理 + 边界条件         → 验收后继续
```

**为什么需要这个机制：** 实证数据显示，在 2-4 小时的无用户介入编码窗口中，每个微决策有 10-15% 的偏差概率。经过 7 个阶段的链式放大（设计→理解→骨架→LLM生成→批量交付→门禁不检设计→用户一次性审查），最终至少一个可见偏差的概率超过 95%。拆分编码窗口可将偏差率降至 20% 以下。

### 第四节：Step 3 — 工程实现

#### 3.1 核心原则

**骨架先行：** 先搭好文件结构再填充逻辑。如 Step 2.5 未做，则在 Step 3 开头做。

**Aider 优先：** 所有代码变更（新建/修改/删除文件）均通过 Aider 执行。Hermes 不直接使用 write_file 或 terminal 做代码变更。详见 Step 0-C 执行者判定。

#### 3.2 工程实现模板

**实现任何需求时，按此模板执行：**

1. 先修改/创建**数据模型**（SQLite schema 变更）
2. 再实现**后端逻辑**（route + service）
3. 再创建/修改**前端模板**（HTML + CSS + JS）
4. 运行测试或手动验证

#### 3.3 工程实现纪律

**Aider 优先：** 任何代码变更（3+ 行以上）优先通过 Aider 委托执行，而非直接调用 write_file/terminal 编写代码。详见 Step 0-C 执行者判定表。

**多步骤任务（3+ 步 = 3 次以上的 terminal/cron/API 调用），必须采用 `execute_code` 批量编排**，而非逐条调用独立工具。
- 将多个步骤合并为一个 Python 脚本
- 不要写一步等一步、写一条跑一条
- 参考 `batch-execution-optimization` skill 的并行策略

**可回滚原则（v2.2 新增）：** 编码时的每一步变更，都必须考虑到回滚场景。全部代码变更需要有对应的反向操作（尽可能支持自动回滚）。

**模式A（简单任务——我自检，Tier 2/3 默认）：**

```
|:-:|:---------|:----:|:-----|
| 1 | POST /practice/<id>/submit 返回 200 | ✅ | curl 返回 HTTP 200 |
| 2 | 数据库包含 answer_choices 列 | ✅ | .schema practice_answers |
```

**模式B（复杂任务——审核 SubAgent 执行，Tier 0/1 强制）：**
创建一个独立的审核 SubAgent，遵循以下规则：

**审核 SubAgent 上下文构造规则：**
```
context=f"""
你是一个冷酷的设计符合性审查员。

规则：

验收条件:
[从 acceptance-criteria-[task].md 读取的完整清单]

输出格式:
每条结果占一行：✅/❌ 条件描述 | 证据
```

**为什么是 2 轮？** 3 轮不过说明要么验收条件本身有问题，要么 SubAgent 不适合此任务。继续循环浪费 token，升级是最优解。

## 第二章：核心设计原则

### 任务分解原则：切片而非切块

这是所有复杂任务分解的**第一性原理**，高于其他所有流程规则。

**核心概念：**

- **切片（正确）**：沿关键路径纵向切割，每个切片是一个完整的功能闭环
  - 例：题入库 → 学生读题 → 学生答题 → 出结果（完整端到端）
- **切块（错误）**：横向切割功能模块，每个模块是孤立的碎片
  - 例：先做完出题模块 → 再做答题模块 → 再做批改模块

**为什么切片优于切块？**

| 维度 | 切片 ✅ | 切块 ❌ |
|:----:|:--------|:--------|
| **连贯性** | 每个切片可独立验证、可独立使用 | 做完一块仍然不能用 |
| **风险控制** | 关键路径先通，早期发现架构问题 | 做完全部块才第一次集成，偏差累积 |
| **反馈周期** | 每片可给用户预览，及时纠正方向 | 用户到最后才看到全貌，改造成本高 |
| **改左坏右** | 切片内改一个环节不影响其他切片 | 块之间耦合度高，一块改了另一块塌 |

**执行步骤：**

1. **识别关键路径** — 没有这个环节就万事皆休的骨架环节
   - 例（教育系统）：题库为空 → 学生无法做题 → 所有功能不可用
   - 关键路径 = 题入库 → 学生做题 → 生成结果

2. **构建最小闭环** — 沿关键路径打通一条完整链路
   - 不做边缘功能、不做优化、不做扩展
   - 只做「从起点到终点能跑通」的窄切片

3. **做透再扩展** — 闭环扎实后，再分层添加其他场景
   - 不是「都做一点」，而是「一条路走到底，再开新路」
   - 每个新切片都走「闭环 → 做透 → 扩展」的循环

4. **画依赖图谱** — 任务开始前画出功能间的依赖关系
   - A → B → C：必须沿这个顺序解，不能跳到 D
   - A ← B（A依赖B）：先做依赖项再做被依赖项

**错误示范（我过去的切块式）：**
```
阶段1: 出题引擎（题从哪来）
阶段2: 题库存储（题放哪）
阶段3: 答题界面（学生怎么答）
阶段4: 批改结果（答完怎么样）
╰── 每个阶段做完都无法独立使用，且阶段1做完了不知道会不会影响阶段3
```

**正确示范（切片式）：**
```
切片1（最小闭环）: 预设10题硬编码 → 学生答题 → 显示对错
                    ↑ 虽然原始，但完整可跑
切片2（做透）:    硬编码→数据库题库，10题→批量出题
切片3（扩展）:    加分题库整合→难度分级→补做追踪
```

**检查清单（每次任务分解前自问）：**
- [ ] 我识别出了这个系统的关键路径吗？
- [ ] 我的第一个切片是一个完整的端到端闭环吗？
- [ ] 这个闭环不依赖任何其他未完成的切片吗？
- [ ] 用户能用这个闭环独立完成一件事吗？
- [ ] 我的分解是「纵向切开」还是「横向切块」？

### 系统化交付原则

所有复杂编码任务，最终必须被「体系化」：

1. 交付不是「写完代码就结束」——代码必须被注册到管理基础设施中
2. 交付必须包括审计机制——每套系统要有自我健康检查能力
3. 交付后必须有「手工验证脚本」——可随时重跑、回归时复用
4. 沉淀设计笔记——凡是绕过的限制、跳过的非强制要求、留下的 TODO，必须记录在 README 或 CHANGELOG 中

### 监督分离原则（Supervision Separation Principle，v5.2.0 新增）

**核心命题：监督者与执行者必须是结构独立的两个实体。**

如果监督者和执行者是同一个实体（即使我"切换身份"自我检查），监督天然失效。原因：

```
同一实体的自我监督 = 没有监督
  ├─ 我知道我"原本想做成什么样" → 容易用意图合理化偏差
  ├─ 我知道我"已经做了什么" → 倾向于认为自己做得对
  └─ 我可以在自我监督流程中走捷径 → 跳过了也无人发现
```

**三条执行规则：**

**规则 1：执行者 ≠ 审核者 ≠ 交付审计者** — 三者必须是三个独立实体

```
编码 (Aider SubAgent) → 审核 (审核SubAgent，不知情) → 交付审计 (独立审计SubAgent，不知情)
    ↑ 三者独立           ↑ 只看验收条件清单            ↑ 只看原型HTML+代码
```

**规则 2：交付审计者必须不知道执行者的"已完成"宣称**

- 交付审计SubAgent不携带我（Hermes）的对话上下文
- 它只知道：原型HTML长这样、实际代码长这样、两者是否一致
- 它不知道我说过"我做完了"——这条信息对它完全不可见

**规则 3：宣称"已完成"前，交付审计必须 PASS**

- 任何"已完成"的宣告，必须由独立交付审计的 ✅ PASS 支撑
- 如果没有先触发交付审计就说了"已完成" → 属于交付违规
- 违规处理：立即补充交付审计，不管结果如何都先在脸上标记 🔴 无审计声明

**为什么这是个原则而非步骤？**

步骤可以被跳过（如我过去做的），但原则是**每次决策时的判断基准**。当我在交付前犹豫"要不要先审计还是直接回复"时，监督分离原则告诉我：必须审计。它不是一个可选的流程节点，而是一条不可绕过的设计约束。

### 领域边界：仅限软件开发任务

**本技能的所有策略（切片式推进、原子验收条件、P0/P1/P2 优先级、Tier 分级）仅适用于软件开发任务。**

不要将这些策略套用到以下领域：
- ❌ **个人任务管理**（日程编排、健康作息、待办分类）→ 应使用 `personal-task-management-strategy` skill
- ❌ **日常例会待办**（催办、协调、委派）→ 应使用个人任务管理体系
- ❌ **生活安排**（作息、运动、亲子活动）→ 由健康提醒 cron 和办公室群管理

**为什么需要这个边界？**
- 切片式推进是针对代码实现的纵向切割——个人任务没有"代码骨架"的概念
- P0/P1/P2 是针对开发迭代的紧急度分级——个人任务的时间约束来自外部（截止日）而非内部（依赖关系）
- 混用会导致：用开发粒度管理生活→任务碎片化→用户感到不可控

**正确的区分方式：**
| 策略 | 所属领域 | 适用场景 |
|:----|:--------|:---------|
| 切片式推进 | 本技能 | 功能开发、系统重构 |
| P0/P1/P2 优先级 | 本技能 | 编码任务排序 |
| 原子验收条件 | 本技能 | 代码质量门禁 |
| 四层精力过滤(🔴🟡🟢⚪) | 个人任务管理 | 日程任务分派 |
| 时段-精力匹配 | 个人任务管理 | 任务编排 |
| 委派标签路由 | 个人任务管理 | 团队协作 |

### 约束学习闭环（v2.2.0 新增）

设计缺陷 → 4 级权重 → 写入约束注册表。

| 权重等级 | 含义 | 存储方式 |
|:--------:|:------|:---------|
| 🔧 内置 | 代码层面自动校验（Schema、pydantic、Type Hints） | 代码自身 |
| 🛡️ 门禁 | CI 流程中自动检查（lint、test、security scan） | CI 配置文件 |
| 📋 清单 | 需要人工核对的检查项 | 设计文档中的 checklist |
| 📚 参考 | 最佳实践和指导原则（不强制，但建议遵守） | 设计文档/knowledge base |

从「遇到问题」到「学到教训」再到「再也不会犯同样错误」的闭环：新的 Bug 修复或设计缺陷发现 → 判断在 4 个等级中属于哪个 → 写入对应位置。

## 第三章：技术风险

### 可检测指标

在开发的每个阶段，保持对以下指标的警惕：

| 指标 | 健康 | 警告 | 危险 |
|:----|:----|:-----|:-----|
| 上下文使用 | <50% | 50-70% | >70% |
| 子代理调用次数 | <5 | 5-10 | >10 |
| 单次编码窗口 | <30min | 30-60min | >60min |
| 偏差检出率（每次交付） | <1个 | 1-3个 | >3个 |

### 核心风险

1. **上下文膨胀风险** — 历史步骤的代码全量载入 Step 3，超过窗口限制
   - 缓解：交付物驱动模式，只读路径引用，文件变更时重新热加载
   - 紧急：极端情况下启用压缩（`compressed_deps=True` 标记为"不读，当没读到"）
2. **回滚风险** — 用户不是每次都有备份，数据结构变更无法自动回滚
   - 缓解：SQLite 迁移脚本必须含 `revert()` 函数
3. **认知偏差风险（v4.3.1 新增）** — 实现者同时也是审核者时，容易用"当时的设计理由"合理化微偏差
   - 缓解：D-Des 门禁要求审核者只持有验收条件清单（不持设计文档全文），做"不知情"的条件比对

## 第五章：与子代理驱动开发的关系

对于 Tier 0/1 任务，C005 的 Step 3（工程实现）可交由 `subagent-driven-development` skill 执行。

### 角色职责分离

| 角色 | 执行者 | 职责 | 持有文档 |
|:----:|:-------|:-----|:---------|
| **设计者+验收条件定义者** | Hermes（你） | Step 0-2 + Step 2.7 原子验收条件 | 设计文档全文（内部参考） |
| **编码执行者** | SubAgent（aider/codex/claude-code，WSL优先用aider） | Step 3 工程实现 | 验收条件清单+设计文档摘要（不含设计意图讨论） |
| **审核者** | SubAgent（冷酷审查员） | D-Des 符合性门禁 | **仅**验收条件清单+代码文件（禁止看设计文档） |
| **决策者** | Hermes（你） | 2 轮审核后仍有 FAIL 时的升级处理 | 全部上下文 |

### 多Agent 协议规则

**验收条件清单是编码 SubAgent 和审核 SubAgent 之间的唯一通信协议。**

- 编码 SubAgent 收到的是：验收条件清单 + 骨架文件结构（不含"为什么要这样设计"的讨论）
- 审核 SubAgent 收到的是：验收条件清单 + 实现后的完整代码（不含设计文档全文）

**为什么审核者必须不知情？**
> 如果审核者看了设计文档，知道"当时的设计意图是 X"，那么在检查验收条件时容易用「代码的意图达到了设计效果」来放过「代码的字面不满足验收条件」的偏差。这是确认偏误（confirmation bias）的经典表现。审核者只看到验收条件，就不会产生这种偏误——他没上下文可用来"合理化"偏差。

### SubAgent 启用的两阶段工作流

```
┌─────────────────────────────────────────────────────────┐
│ Hermes 完成 Step 0-2 + Step 2.7                        │
│ - 输出验收条件清单文件 docs/acceptance-criteria.md      │
│ - 输出骨架文件 docs/skeleton-structure.md               │
└──────────┬──────────────────────────────────────────────┘
           │ delegate_task
           ▼
┌─────────────────────────────────────────────────────────┐
│ 编码 SubAgent（独立进程）                                │
│ - 只收到：验收条件清单 + 骨架文件                         │
│ - 执行：Step 3 工程实现                                   │
│ - 输出：代码文件（含 py、html、SQL 迁移等）               │
└──────────┬──────────────────────────────────────────────┘
           │ delegate_task (再次)
           ▼
┌─────────────────────────────────────────────────────────┐
│ 审核 SubAgent（独立进程，新的上下文）                      │
│ - 只收到：验收条件清单 + 代码文件（含所有变更）             │
│ - 执行：D-Des 逐条条件比对                                │
│ - 输出：PASS/FAIL 表格                                    │
│ FAIL → 第二轮（重复编码+审核）→ 仍 FAIL → 升级 Hermes    │
└─────────────────────────────────────────────────────────┘
```

### 两阶段 SubAgent 调用的执行模板

见 `subagent-driven-development` skill 中的 [两阶段 SubAgent 调用模板](. 章节)。

### 退出条件

在以下任一条件满足时，回到 Hermes 直接控制：
1. 审核 SubAgent 判定 ALL PASS → 进入 Step 4 质量门禁其余维度和 Step 5 交付
2. 审核 SubAgent 2 轮后仍有 FAIL → 升级给 Hermes 决策
