# 团队自我评估与角色优化工作流

## 适用场景

- 完成一批重要任务后，评估团队角色是否需要调整
- 发现执行过程中反复出现某类问题（如测试遗漏、前端 bug）
- 任务领域发生重大变化（如从纯后端转向全栈开发）
- 定期复盘（每月/每季）时评估团队健康度
- 团队新增角色后，评估其对后续任务的实际价值

## 触发条件

| 信号 | 说明 |
|:----|:------|
| 批量任务完成 | 同一团队完成 3+ 项任务或一次大型交付 |
| 重复问题类型 | 执行中反复出现同一类问题（如端口冲突、模拟数据） |
| 领域切换 | 任务类型从 A 领域转向 B 领域 |
| 用户主动要求 | 用户提出「评估团队」「优化角色」等指令 |
| 定期评估 | 每月初或每季度评估一次团队健康度 |

## 核心流程

### 阶段 1：执行数据收集

回顾近期完成的任务，按以下维度整理数据：

| 数据项 | 来源 | 收集方式 |
|:-------|:-----|:---------|
| 任务清单 | 今日执行的任务列表 | `session_search` 或会话记忆 |
| 每角色产出 | 各角色的具体交付物 | 项目文件、架构文档、代码、测试报告 |
| 执行问题 | 反复出现的问题类型 | 调试记录、修复轮次 |
| 中断/超时 | 子代理中断次数和原因 | 执行日志 |
| 用户反馈 | 用户明确指出的问题 | 对话记录 |

### 阶段 2：按角色评估贡献

对每个已设角色进行 5 维评估：

| 维度 | 评分标准 |
|:-----|:---------|
| 🎯 **参与度** | 在本批任务中是否频繁被调用？具体产出了什么？ |
| ✅ **质量** | 产出是否有重大问题？是否需要后续修复轮次？ |
| 🕐 **效率** | 是否按时完成？是否有超时、中断？ |
| 🧩 **覆盖度** | 是否覆盖了所有应该覆盖的方向？有无遗漏？ |
| 🆕 **创新性** | 是否提出了改善流程或产品的新思路？ |

**示例评估模板**：

```
角色名: 🧪 质量保证工程师
├─ 参与度:   ⭐⭐⭐   (参与测试策略评审 2 次)
├─ 质量:     ⭐⭐⭐⭐  (发现 3 个 P0 级问题，但漏掉 URL 路径错误)
├─ 效率:     ⭐⭐⭐⭐  (每次响应 < 2 分钟)
├─ 覆盖度:   ⭐⭐⭐   (API 测试充分，但缺页面渲染检查)
├─ 创新性:   ⭐⭐     (未提出新的测试方法)
└─ 总体:     ⭐⭐⭐⭐  (有价值的角色，但在前端测试方面需要加强)
```

### 阶段 3：识别能力缺口

基于执行问题数据，反向推断缺失角色：

```
执行中反复出现的问题 → 反推缺失的能力
├─ 模拟数据未替换 → 前端测试/验收能力缺失
├─ URL 路径错误 → 集成测试/联调能力缺失
├─ 参数校验缺失 → 边界测试能力缺失
├─ 端口冲突 → 基础设施/运维能力缺失
├─ 子代理文件冲突 → 并行任务协调能力缺失
└─ 某类知识匮乏（教育/财务/法律）→ 专业领域覆盖缺失
```

**角色扩展矩阵**（使用此表判断是否需要新增角色）：

| 缺失能力表现 | 建议补充角色 | 常驻/动态 |
|:------------|:------------|:----------|
| 前端渲染问题反复出现 | 用户体验/前端专家 | 常驻 |
| 测试遗漏 P0 级缺陷 | 质量保证工程师 | 常驻 |
| 无法评估团队能力配比 | 人力资源专家 | 常驻 |
| 成本/预算无法评估 | 财务顾问 | 动态 |
| 法律合规风险不确定 | 法律顾问 | 动态 |
| 安全漏洞反复出现 | 安全工程师 | 常驻/动态 |
| 性能瓶颈未识别 | 性能工程师 | 动态 |
| 特定领域知识匮乏 | 对应领域专家 | 按需加载 |

### 阶段 4：提出优化建议

将发现组织为结构化建议表：

| # | 建议类型 | 当前状态 | 建议变更 | 理由（含数据支撑） |
|:-|:--------|:--------|:---------|:------------------|
| 1 | 升格 | 动态→常驻 | 🎨 UX/前端专家 | 前端工作占比 40%+，反复出现模拟数据问题 |
| 2 | 新增 | — | 🧪 QA工程师 | 每个版本都有 P0 级修复，测试策略无人把关 |
| 3 | 降级 | 常驻→动态 | 👥 人力资源专家 | 本周无组织类任务，可按需加载 |
| 4 | 保留 | — | — | 其余角色表现符合预期 |

### 阶段 5：用户确认 → 实施

1. 将评估报告呈现给用户
2. 等待用户确认（C007 规范 — 禁止跳过确认）
3. 确认后更新：
   - 对应团队的 skill 文件（角色列表、评审流程、错误示例）
   - 记忆（memory）更新角色阵容描述
   - 版本号升级

## 团队自评视角参考

### 视角 A：顾问团评估自己（自我审视）

```
核心问题：我们（顾问团）现有的角色阵容，是否覆盖了今天实际执行的任务类型？
评估方法：回顾今日对话中实际调用了哪些角色的视角
         检查是否有需要但没有的角色视角
独特产出：角色升格/降级建议（动态↔常驻流动）
         角色新增/合并/拆分建议
         角色职责边界澄清建议
```

### 视角 B：顾问团评估开发团队

```
核心问题：开发团队（C005）在执行中的表现，哪些问题是由团队配置导致的？
评估方法：分析子代理超时率、重复交付物率、修复轮次数
         P0 修复的类型分布（代码质量/架构/测试/配置）
独特产出：子代理配置优化建议
         前置检查缺失项清单
         交付物格式改进建议
         负载等级边界调整建议
```

### 视角 C：开发团队评估自身

```
核心问题：当前的负载等级划分、子代理分配策略是否最优？
评估方法：统计每轮执行的时间分布（学习:设计:实现:测试）
         计算子代理超时率与文件复杂度的关系
         统计修复循环的触发率和耗时
独特产出：Tier 边界值调整建议
         子代理分区策略优化
         交付物模板改进建议
```

## 评估报告输出模板

最终产出应为结构化文档，包含：

```markdown
# [团队名称] 自我评估报告

## 概况
- 评估期间: YYYY-MM-DD
- 评估任务数: N 项
- 当前阵容: [角色列表]

## 角色评分矩阵
| 角色 | 参与度 | 质量 | 效率 | 覆盖度 | 创新 | 总分 |
|:----|:-----:|:----:|:----:|:-----:|:----:|:----:|

## 能力缺口分析
| 缺口 | 表现证据 | 根因 | 建议补充角色 |
|:----|:---------|:-----|:------------|

## 优化建议
| # | 类型 | 变动 | 理由 |
|:-|:----|:----|:-----|

## 变更实施清单
- [ ] skill 更新
- [ ] memory 更新
- [ ] 版本号升级
```

## 注意事项（从实战中总结）

1. **数据驱动而非直觉** — 每个优化建议必须有实际执行数据支撑，不能说「我觉得缺 XX 角色」而要说「XX 问题出现了 N 次，说明缺 XX 能力」
2. **避免角色膨胀** — 不是每个缺口都要新增常驻角色。能用现有角色吸收的缺口就吸收，能不常驻就动态
3. **升格容易降格难** — 把动态角色升为常驻要慎重，降回去会让用户困惑。建议先试运行一期
4. **常驻角色上限** — 建议不超过 10 人，超过 8 人时考虑合并职责（如 UX+前端合并为「交互工程师」）
5. **评估频率** — 正常每月一次，重大交付后可即时触发一次
6. **并行评估** — 评估顾问团的同时，也可以评估开发团队。但分两次询问用户，避免信息过载
