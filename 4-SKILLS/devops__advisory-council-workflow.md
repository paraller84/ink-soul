# C007 顾问团系统使用规范

> 顾问团的核心产出是**方案文档**，不是代码。

## 群感知启动规则（v3.6 新增）

> 当顾问团收到来自特定飞书群的消息时，无需用户额外声明角色 — 根据群 chat_id 自动匹配角色预设和启动模式。

### 检测逻辑

```python
# 系统自动注入当前消息来源的 chat_id（如 Feishu DM 则为 None）
SOURCE_GROUP = context.get("chat_id")

# 顾问团启动时自动检查
if SOURCE_GROUP in GROUP_ROLE_PRESETS:
    # 自动加载该群的角色阵容
    active_advisors = GROUP_ROLE_PRESETS[SOURCE_GROUP]["advisors"]
    start_mode = GROUP_ROLE_PRESETS[SOURCE_GROUP]["mode"]
    # 跳过 Socratic 询问（角色已由群主题确定）
```

### 使用指引

- 用户在顾问团群中**只需描述任务本身**，无需说「请教育专家评估」「请首席策略分析」
- 系统根据群 chat_id 自动识别要加载哪些角色
- 如果用户说的任务明确不属于该群预设范围，系统应提示「这个任务可能更适合在XX群处理」
- 如果用户需要临时补充预设之外的角色，可以在描述中指明（如「再加一个视觉设计师」）

## 使用时机

顾问团有五种工作模式，根据任务类型选择：

### 🔴 红线：方案文档交付后不得直接进入实施

**发现场景**（2026-05-14 AI家庭教师系统设计）：用户确认内部评审后，顾问团直接考虑进入 C005 编码实施。用户明确纠正：「现在都还不要执行搭建，请输出完整的设计文档到飞书云盘上。待我完全确认后再进入开发阶段」。

**执行铁律**：
1. 内部评审（4 角色评审）完成后，**必须输出完整设计文档到飞书云盘「顾问团报告」文件夹**
2. 文档包括：功能规格、页面交互路径、数据模型、技术架构、API设计、目录结构、分期计划、风险评估
3. **通知用户「文档已输出，请确认」**，不催促、不暗示「可以开工了」
4. 等待用户主动说「确认」或「可以了」后，才进入 C005
5. 用户可能隔几小时甚至隔天才回复——期间不得自行推进实施

**设计文档模板结构**（可在 Mode A 第2轮后预生成框架）：
| 章节 | 内容 |
|:----|:------|
| 1. 项目背景与定位 | 一句话定位 + 解决的核心问题 + 与现有系统的关系 |
| 2. 用户角色与使用场景 | 角色定义 + 典型使用场景 |
| 3. 功能规格 | 每页面的详细布局+交互逻辑+组件表 |
| 4. 页面交互路径 | 学生端/家长端的完整流转图 |
| 5. 数据模型设计 | 核心实体关系 + 存储设计 + 选型理由 |
| 6. 技术架构 | 架构图 + 核心选型 + AI引擎设计 |
| 7. 服务接口设计 | 所有API端点(方法/路径/说明) |
| 8. 与现有系统的借鉴对照 | 每个模块的借鉴内容+来源+实现方式 |
| 9. 目录结构 | 完整文件树 |
| 10. 分期交付计划 | 每期任务+工时+依赖 |
| 11. 技术风险与缓解 | 风险表+概率+影响+缓解措施 |
| 12. 内部评审意见 | 各角色意见（按优先级分组）|

**与 C005 的关系**：C007 产出设计文档后，C005 在启动 Step 0 时直接从 Step 2.5（骨架搭建）开始，跳过 Step 1（代码学习）和 Step 2（架构设计），详见 `code-development-workflow` 的「C007 加速路径」。

**例外**：用户说「直接执行」或「马上开始」— 按熔断机制执行。

### 模式A：新系统/新功能规划（Socratic 4轮询问）

- 用户提出一个模糊的、多步骤的新任务
- 用户说"我想搭建一个XX系统"
- 需要跨功能决策的任务（涉及技术选型、资源分配、分期规划）
- **涉及外部服务集成时，必须先做一轮API能力侦察（Pre-Step），再进入Socratic询问**

#### Pre-Step：外部服务API能力侦察（当任务涉及集成第三方服务时）

在进入Socratic 4轮询问之前，先独立评估外部服务的API能力，输出能力矩阵供顾问团参考：

```markdown
| 能力维度 | 服务A（如Get笔记） | 服务B（如Flomo） |
|:---------|:-----------------|:----------------|
| 读取笔记 | ✅ 列表+详情API   | ❌ 无读取API     |
| 搜索     | ✅ 语义搜索       | ❌ 无搜索API     |
| 写入笔记 | ✅ CRUD           | ✅ 仅写入        |
| 语音/音频| ✅ 录音→转写       | ❌ 不支持        |
| Webhook  | ❌ 不支持          | ❌ 不支持        |
| MCP Server | ✅ 26个工具     | ✅ 仅写入工具     |
| 速率限制 | 2 QPS, 5000次/天  | 无明确限制       |
```

**行动**：用web_search + web_extract调研外部服务的官方文档/API页面/GitHub仓库。重点确认：
1. 是否有读取API（list/search）— 决定了能否自动同步
2. 是否有Webhook/推送 — 决定了轮询还是推送
3. 是否有MCP Server — 决定了实时接入的便捷度
4. 特殊数据能力（语音转写、链接抓取、OCR等）— 决定了独有价值

**输出**：能力对比表 + 集成可行性评级（可行/有条件/不可行）+ 推荐的集成层次（L1 MCP/L2 自动化管道/L3 双向同步）

> **注意：模式A的特殊输出形态：持续服务（非一次性系统）**
>
> 顾问团Mode A的产出不仅是「搭建XX系统」，也可能是「由我在对话中持续交付的服务」。
> 例如：邮件回复顾问服务（用户把邮件发我，我拟草稿，用户微调后发送）。
>
> **这种场景下，必须增加"闭环反馈机制"**：
> 用户修改我输出的草稿后，应回传最终版，以便我通过差异比对隐式学习用户偏好。
> 直接问「你喜欢什么风格」效率远低于通过实际编辑比对来学习。
>
> 实战案例见 references/mode-a-email-advisory-service.md

### 模式B：外部提案集成评估

- 用户带来一个外部提案/模式/工作流，询问"这和我们已有的系统怎么整合？"
  → 参见 **`external-proposal-evaluation`** skill

### 模式C：已有系统策略优化（多角色链式评估）

- 用户说"请对现有的XX策略/方案进行评估"
- 用户已有一份完整的策略文档，希望获得专业意见
- 需要从多个专业视角（教育/产品/技术）评估可行性后再执行

#### 模式C 分型：Full Chain vs Lightweight

根据设计文档的成熟度，模式C有两种分型：

| 分型 | 适用场景 | 流程 | 典型触发 |
|:----|:---------|:-----|:---------|
| **Full Chain** | 冷启动系统设计，从零开始的方案评估 | 第1轮教育专家 → 第2轮产品经理 → 第3轮架构师 → 第4轮综合决策（完整4轮） | 「帮我设计一套XX系统」「从零规划这个功能」 |
| **Lightweight** | 已有设计文档的迭代优化，需求明确无歧义 | 跳过Socratic询问，直接3角色并行评估+参与设计产出 | v1.2→v1.3→v1.4这类已有基础的调整 |

**关键区分原则：**
- **有无不明确的点？** 如果设计方向、功能边界、约束条件都已确定（仅需补充和优化），走 Lightweight。
- **是否需要从零梳理需求？** 如果用户说「我想做一个XX但不确定具体怎么做」，走 Full Chain。
- **用户有明确的不确定项吗？** 如果用户已经给出了非常具体的修改要求（「补充拍照批改」「补充荣誉体系」），说明设计边界已清晰，走 Lightweight。

**Full Chain 流程（完整 4 轮）：**

```
第1轮 — 领域专家评估（如教育专家）
  评估维度：教学法合理性、认知规律匹配度、数据佐证
  输出：缺陷清单 + 优化建议（按优先级标注）

#### Pre-Step：厘清术语与系统边界

在进入多角色链式评估前，先做一轮**边界澄清**，避免我用自己的术语框架与用户的认知错位：

```markdown
# 向用户确认的问题（1-2个）

Q1：如何称呼目标对象？
  - 用户说"语文系统" → 核实：它是独立应用还是 Edu-Hub 下的一个模块？
  - 用户说"C003" → 核实：C003 是能力注册编号，用户是否也理解为"模块"还是"独立系统"？

Q2：评估范围是？
  - 代码工程边界 → 同一个 app.py 还是独立部署？
  - 数据共享边界 → 共享认证/共享学生管理/共享质量门禁？
```

**实战教训（2026-05-12）**：我在分析语文模块时直接按"独立系统"做了对比表，用户纠正说"这个和我记忆中的不同"。根因：C003/C008/C011 不是独立系统，而是共享同一个 `app.py`（Edu-Hub 统一教育平台）的学科模块。教训：先确认实体类型（系统/模块/管道），再选择分析框架。du-Hub）下的三个 Blueprint 模块。用户要求"先纠正体系内的表达标准"再继续讨论。

**输出**：在首轮对话中标注系统关系图（使用`architecture-diagram` skill），让用户确认后再进入评估。例如：

```
Edu-Hub (app.py :5002)
  ├── /math/       → 数学模块 (C003)
  ├── /english/    → 英语模块 (C008)
  └── /chinese/    → 语文模块 (C011)
  
模块间共享: auth认证 · Flask session · 质量门禁 C015 · 学生管理(english.v2.role_manager)
```

**模式C 流程**：

> ⚠️ **角色选择灵活化（2026-05-14 实战更新）：** 第2轮原为「产品经理」，对于教育/学习类系统，用户偏好替换为「🎨UX/前端专家」。选择依据：
> - 如果系统交互复杂或涉及儿童/老人等特殊用户 → UX专家替代产品经理
> - 如果是企业SaaS/商业系统 → 保留产品经理
> - 如果不确定 → 询问用户偏好

```
第1轮 — 领域专家评估（如教育专家）
  评估维度：教学法合理性、认知规律匹配度、数据佐证
  输出：缺陷清单 + 优化建议（按优先级标注）
  ⚠️ 专家可能会提出理论完全正确但落地困难的意见
       → 这正是下一轮存在的理由

第2轮 — 产品经理评审
  评估维度：用户体验、用户理解成本、动机/习惯养成
  任务：对第1轮建议做二次过滤
  输出：✅立即采纳 / ⚠️条件采纳 / ❌暂缓弃用 + 产品优先排序

第3轮 — 系统架构师评审
  评估维度：技术实现方案、改动量预估、风险、数据模型影响
  任务：对已通过的每一建议给出具体实现方案
  输出：P0/P1/P2 执行矩阵 + 代码改动位置 + 预估工时

第4轮 — 综合决策输出
  合并三轮评审为一张表：
    教育专家建议 | 产品判断 | 架构方案 | P0/P1/P2 | 执行/暂缓
  立即可执行项（R0）直接执行代码改动
  后续项作为 roadmap
```

**模式C 的关键设计决策记录规则**：
当教育专家、产品经理和系统架构师给出分歧意见时（如「独立入口 vs 合并」），
必须将**所有方案**呈现给用户选择，最终实现方案需记录**用户决策**及理由，
不得替用户拍板。

**模式C 的用户确认问题格式**：
在需要用户做决策时，将待确认的问题以结构化方式列出，**不超过3个**，
每个问题给出明确选项（A/B）和建议答案，让用户可以快速决策：

```
### 需要您确认的问题

Q1：方案A还是方案B？
  - A — 方案A（优点...）
  - B — 方案B（优点...）
  建议：选A

Q2：...
```

实战案例（2026-05-04 C008 学习币评估）：
用户提出了详细的学习币系统规格，产品经理和架构师对兑换比例的真实性有疑虑。
以3个结构化问题让用户确认后，用户选择「兑换=真实人民币」→ 确定了系统的
数据模型和风控设计方向。避免了替用户拍板的错误。

**模式C 的 P0/P1/P2 执行矩阵**：

在完成多角色链式评估后，将综合决策结果整理为执行矩阵：

```
评估报告
  ├── 教育专家：N 条建议（含教学法依据）
  ├── 产品经理：每条的可行性判断（立即采纳/条件采纳/暂缓）
  └── 系统架构师：每条的技术方案+预估工时+风险

综合决策表
  ┌────────┬────────┬────────┬────────┬──────────┐
  │ 建议    │ 产品   │ 架构   │ 优先级  │ 决策    │
  ├────────┼────────┼────────┼────────┼──────────┤
  │ 调整X  │ ✅     │ 5分钟  │ P0     │ 立即执行│
  │ 功能Y  │ ⚠️改进  │ 3小时  │ P1     │ 下迭代   │
  │ 架构Z  │ ❌暂缓  │ 风险中 │ -      │ Q2规划   │
  └────────┴────────┴────────┴────────┴──────────┘

P0（≤30分钟，今天执行）：
  - 纯配置变更（改参数/概率/文案）
  - 无风险的单行改动
  - 不超过 3-5 项

P1（2-6小时，下个迭代）：
  - 新增模块/页面
  - 涉及前后端但逻辑清晰
  - 不影响现有功能的核心路径

P2（Q2 规划）：
  - 涉及数据迁移
  - 架构级重构
  - 依赖其他系统或外部能力
```

**模式C 的执行双阶段协议**：评估报告产出后，不直接进入全面实施。执行**双阶段**流程：

```
阶段1 — 评估与选择（当前会话）
  ① 顾问团产出评估报告（按 P0/P1/P2 分级）
  ② 用户从报告中挑选可立即执行项（通常为 P0）
  ③ 用户确认「仅执行 P0」或「全部执行」
  ④ C005 编码工作流仅执行用户选中的项
  
阶段2 — 规划与跟踪（下个迭代）
  ① 未被选中的 P1/P2 项列入 roadmap
  ② 每次迭代开始时，用户决定是否从 roadmap 中选取新项
  ③ 选取的新项走 C005 编码工作流
```

**关键原则**：
- **不可替用户决定执行范围** — 评估报告中的 P0/P1/P2 是顾问团的建议，用户有权选择只执行其中一部分
- **P0 项可立即执行**（在同一会话中）— 因为 P0 项都是 ≤30 分钟的小改动，不影响现有功能
- **P1/P2 项必须进入 roadmap** — 即使架构师表示「改 3 行就行」，只要是 P1 级（涉及用户对新功能的感知变化），应留给用户下次决定
- **执行前必须用户确认** — 在开始 C005 实施前，用不超过 3 个问题确认执行范围

#### Lightweight 模式流程（迭代优化用）

**适用前提：** 已有明确的设计文档（v1.x+），用户的修改要求具体明确。核心区别是「没有不明确的地方」——如果用户说「补充拍照批改」「补充荣誉体系」等确切要求，就是Lightweight。不需要走Socratic询问。

**⚠️ 评估完整性红线：** 一旦确认进入Lightweight模式，**所有指定角色必须逐一完成评估，一个都不能少**。用户原话：「评估的完整性是必须的」。禁止仅做一个角色的评估就产出设计文档。如果时间或精力有限，宁可减少迭代次数也不要跳过角色。

**角色配比（针对教育系统）：**
| 角色 | 评估维度 | 设计输出 |
|:-----|:---------|:---------|
| 🎓 **教育专家** | 课标对齐、学习路径、练习设计合理性、记忆体系 | 题型表/记忆策略/出题策略建议 |
| 🎨 **UX/前端专家** | 触摸交互（儿童适配）、导航一致性、返回规范、防误操作 | 交互流程图、页面布局建议 |
| 🏗️ **系统架构师** | 数据模型扩展性、AI管线、预生成策略、风险 | 技术方案、数据模型变更、分期建议 |

> **注意：角色选择应基于任务领域。** 教育系统推荐上述3角色（教育专家/UX专家/架构师），替代Full Chain中的「产品经理」。如果是商业系统/企业系统，产品经理仍适用。根据任务特征从「角色阵容扩展机制」中选择最贴合的3个角色。

```
第0步 — 识别分型（必做）
  判断条件：
  a) 当前是否已有设计文档/方案？（v1.x 以上）
  b) 用户的修改要求是否具体明确？
  c) 是否存在需要从零梳理的不确定点？
  如果 a+b 满足而 c 不满足 → Lightweight。否则 Full Chain。

第1步 — 专业角色并行评估与设计（3角色，无Socratic询问）
  每个角色独立输出 评估意见 + 设计建议
  架构师标记改动量预估（P0/P1/P2）
  三个角色无串行依赖（可同步输出，但必须全部完成才能进入下一步）

第2步 — 综合决策输出
  合并3角色评估设计为一张综合决策表：
    | 功能点 | 教育专家建议 | UX建议 | 架构方案 | 优先级 | 决策 |
  将综合决策表呈现给用户 → 用户逐条反馈（✅执行/⏸️后续/❌忽略）

  用户反馈格式通常为编号清单（如36条）。必须将每条反馈归类为三种动作：
    ✅ 立即执行 — 修改方案中的文字/设计
    ⏸️ 二期 — 标记在方案中但不纳入当前实施
    ❌ 忽略 — 从方案中删除相关内容

  对用户说"不太明白"的项，先解释再执行，不要跳过。

第3步 — 基于反馈迭代设计文档（v1.N+1）
  将用户反馈按变更类型归类，逐一应用到设计文档：
  ・文字/章节描述变更 → 字符串替换或子代理文本补丁
  ・数据模型变更 → 更新数据模型表格
  ・页面说明变更 → 修改对应page-desc文本
  ・架构决策变更 → 替换/删除相关章节描述
  ❌ 不要修改页面原型的phone/pad HTML结构（只改文字描述）

  ⚠️ 子代理委托陷阱（2026-05-14 AI家庭教师v2.1→v2.2已验证）：
    委托子代理"生成vN+1"时，子代理容易：
    a) 只输出简略版本（65KB）而非完整版本（445KB）
    b) 不读取全量源文件，自己"概括"内容
    c) 引入不同CSS主题（暗色 vs 暖白）导致风格不统一
    → 修复：显式约束"读取源文件全文、只改文字、保持~400KB规模"

  合并后执行CSS主题一致性检查：
    # 检查是否混入不同主题的CSS变量名
    for var in ['--text-primary', '--bg-primary', '--bg-card']:
        if var in content and '--text,' not in content:
            print(f"⚠️ 混入暗色主题变量: {var}")
    # 如有，做完整映射替换
    theme_map = {'var(--bg-primary)': '#fdf8f0', ...}
    for old, new in theme_map.items():
        content = content.replace(old, new)

第4步 — 一致性审计（必做，用户2026-05-14明确要求）

  在方案上传交付前，执行**5维交叉审计**。用户会检查"清单页 VS 实际页面"的映射关系——这是用户发现问题的首要来源。

  5维审计清单：
  □ 清单 vs 原型 — 页面清单中的每项都有对应的page-card吗？编号一致吗？
  □ 删除覆盖 — 用户说"去掉X" → `content.count('X')` = 0（包括phone/pad frame内HTML）
  □ 编号一致性 — h3编号、清单编号、section引用编号三者一致
  □ CSS主题统一 — 全文无混入不同主题的CSS变量
  □ 页面原型完整性 — page-card数 = 清单数 + 非原型卡片数（知识点示例等）

  ```python
  # 快速审计脚本
  import re
  inv_pages = len(re.findall(r'<td>(\d+)</td>', content))
  page_h3 = [m.group(1) for m in re.finditer(r'<h3>(\d+)\.\s', content)]
  print(f"Inventory: {inv_pages}, h3: {len(page_h3)}")
  for kw in ['要删除的', '旧名称']:
      print(f"'{kw}'残留: {content.count(kw)}")
  ```

  发现不一致 → 修复后再上传。

第5步 — 交付
  以 HTML 格式为主交付物（含CSS + UI线框图），上传飞书云盘。
  每次迭代新建文件，不覆盖旧版本（v1.2 → v1.3 → 新文件）。
  上传后通知用户"文档已输出，请确认"，不催促进度。

设计文档的版本迭代：
  以 HTML 格式为主交付物（含CSS + UI线框图），上传飞书云盘。
  每次迭代新建文件，不覆盖旧版本（v1.2 → v1.3 → 新文件）。
```

> **实战案例**（2026-05-14 AI家庭教师v2.1→v2.2）：用户给出36条编号反馈，其中第1/3/7/26/31/32项确认后未正确应用到方案中——综合练习在phone frame内仍有残留、A-H字母编号未改、OCR章节引入暗色主题。用户主动指出这些不一致问题。教训：**不能假设子代理的输出完全正确**，一致性审计是独立且必须的步骤。

#### Pre-Implementation Step：架构比较分析

在方案文档得到用户确认后、进入 C005 编码实施之前，**必须额外执行一轮架构比较分析**。本轮分析的目标不是「做什么」，而是「怎么做更安全」。

**触发条件**：满足任一即执行：
1. 改造对象存在**两个以上代码分支**（如 chinese/ vs chinese-v2/）
2. 改造对象需要**与系统内已有模块对标**（如语文对标英语的 practice 架构）
3. 用户明确要求「先分析差异再实施」

**分析维度**：

```markdown
### 6 维架构比较

| # | 维度 | 检查方法 |
|:-:|:----|:---------|
| 1 | **Blueprint 注册** | 对比目标模块与参考模块的 Blueprint 数量、URL 前缀、目录独立性 |
| 2 | **数据持久化层** | state.json 是否有版本号？Session 是纯内存还是有磁盘持久化？有自动备份吗？ |
| 3 | **数据目录结构** | 单一数据目录还是分散多处？备份目录是否活跃？ |
| 4 | **模块内部组织** | 是单一文件包所有，还是分文件按职责隔离？ |
| 5 | **共享依赖** | 哪些模块被复用（role_manager / auth / 质量门禁）？哪些是本模块独有的？ |
| 6 | **模板/前端约定** | 模板前缀是否一致？CSS/JS 是否独立不会污染其他模块？ |
```

**决策输出**：架构差异对照表（新模块 vs 参考模块），标注「继承/改进/新增/舍弃」四类。

**实战案例**（2026-05-12 C011 v3 升级）：在方案文档确认后、实施前，用户要求做此分析。发现语文模块在 session 持久化、数据版本号、自动备份三方面比英语模块薄弱——这些被加入 v3 的必做项。

**模式C 的输出物**：
```
1. 联合评审报告（含三位角色各自的评估意见）
2. 综合决策表（P0/P1/P2 + 执行/暂缓/弃用）
3. 架构比较分析表（Pre-Implementation Step）
4. P0 项的实际代码修改
5. 策略文档同步更新（wiki）
```

## 模式E：多专家并行分析（Sub-Agent委派模式）

> 当用户提供**完整源材料**（邮件/文档/通知/方案）并明确要求从多个专业视角进行分析时，不走 Socratic 4轮询问，改为**直接委派子代理**并行分析。

### 与模式A/C的核心区别

| 维度 | 模式A/C | 模式E |
|:----|:--------|:------|
| 交互方式 | 与用户多轮问答 | 直接委派子代理分析，用户只看最终合成结果 |
| 角色启动 | 逐步加载，依次对话 | 一次性并行加载所有专家 |
| 用户参与度 | 用户每轮都需回应 | 用户只提供初始任务说明，产出合成结果后确认 |
| 适用前提 | 需求模糊/需逐步展开 | 用户已有完整源材料，目标明确 |
| 产出形态 | 方案文档/设计文档 | 多视角分析报告 + 合成行动方案 |

### 触发条件（全部满足）

1. 用户提供了**完整的原始材料**（邮件/通知/附件/会议纪要等）作为分析输入
2. 用户明确要求从 **多个专业视角** 进行分析（如「从EAST报送/监管统计/数据治理三个角度」）
3. 分析目标是**形成判断和建议**，而非从头设计一个系统
4. 各专家视角之间没有强依赖关系（可并行而非串行）

### 流程

```mermaid
第1步 — 角色定义
  ├─ 从用户要求中提取专家角色（通常3-4个）
  ├─ 每个角色定义需包含：角色名+专业背景+分析焦点
  └─ 确认角色间的分析维度互不重叠

第2步 — 上下文包准备
  ├─ 提取源材料核心内容（邮件正文要点、附件摘要等）
  ├─ 如有已有分析文档，也一并打包
  └─ 每个子代理获得相同的上下文包 + 各自的分析任务

第3步 — 并发委派（delegate_task）
  ├─ 每个专家作为一个独立task
  ├─ 设置清晰的输出格式约束（结构化分节拒绝散文）
  └─ 设置工具集约束（read_file用于读源材料）

第4步 — 合成交付
  ├─ 读取各子代理的输出
  ├─ 按逻辑主题合并（不要简单拼接三个报告）
  ├─ 提炼共识点和分歧点
  └─ 输出统一成品（邮件草稿/决策备忘录/行动清单）

第5步 — 用户确认
  └─ 将合成交付件呈现给用户确认或调整
```

### 角色定义模板

当用户说「增加XX专家共同参与分析」，用以下模板构造每个子代理的任务：

```markdown
你是一位有[N]年经验的[角色名]，熟悉[具体领域]。
你需要基于以下原始材料进行分析：

=== 原始材料 ===
[邮件正文/通知要点/文档摘要]

=== 你的分析任务 ===
从[角色名]的角度，重点分析：
1. [聚焦问题1]
2. [聚焦问题2]
3. [聚焦问题3]

输出格式：使用中文，结构化分节，每节3-5点，每条2-4句话。
不要写散文段落。不要输出HTML，直接输出结构化文本。
```

### 专家角色库（可扩展）

| 领域 | 常用角色 | 分析焦点 |
|:----|:---------|:---------|
| 金融监管 | EAST监管报送专家 | 对现有报送流程的影响、口径差异、规则复用性 |
| 金融监管 | 监管统计专家 | 新体系在监管版图中的定位、政策趋势、隐性需求 |
| 金融监管 | 数据治理专家 | 数据标准、质量管理、治理体系建设 |
| 教育科技 | 教育专家 | 教学法、课标对齐、认知规律 |

| 用户体验 | UX/前端专家 | 交互流程、儿童适配、可访问性 |

### 陷阱与注意事项

- **角色定义必须具体**：不要说「一个专家」，要说「有10年EAST报送经验的专家，熟悉IFRS 17和保险业务流程」
- **上下文包不能太大**：子代理的上下文窗口有限，源材料需提炼核心要点（3-8条为佳）
- **输出格式必须约束**：子代理容易输出散文/Markdown表格，要用「结构化分节，每节3-5点」约束
- **合成时不是简单拼接**：三个报告直接拼在一起等于没有合成。需按逻辑主题（如「对EAST的影响」「对数据治理的要求」「对业务部门的建议」）重新组织
- **不要遗漏用户原始需求**：合成时要回头检查是否回答了用户最初的问题，不能只呈现子代理的分析

## 模式D：Bug 修复监管流程（v3.3 新增）

> 所有系统 Bug 修复，必须先走 C007 顾问团前置审核，再进行 C005 实施。
>
> ⚠️ **禁止绕过**：如果因「修复很简单」或「之前修过类似」而直接跳入 C005 实施，
> 不仅违背用户明确指令（2026-05-06），更已在实战中验证会导致：
>   - 同类型 Bug 反复出现（C008 前端反馈问题连续 3 次修复才根除）
>   - 用户信任度下降
>   - 修复次数不断增多但质量不进反退
>
> **违反记录**：每次被检出跳过 Mode D 直接修复 → 模式 D 阶段 4（C005 前）
> 自动插入一次复盘：「这次跳过 Mode D 节省的 5 分钟，与后续重修的 30 分钟相比，值吗？」

**触发条件**：用户报告任何系统功能异常、数据不一致、渲染错误、性能问题。

**原则**：不直接接受「报什么修什么」——必须做根因分析，防止表层修复导致更多 Bug。

### 5 阶段流程

```mermaid
阶段1 ── 问题总结（首席策略）
  ├─ 记录用户报告的原始症状原文
  ├─ 要求用户提供：
  │   ├─ 复现步骤（精确到点击序列）
  │   ├─ 截图/浏览器控制台输出
  │   ├─ API 响应（如适用）
  │   └─ 服务端日志（tail -50 edu-hub.log）
  ├─ 评估影响范围：单个用户？所有用户？单个模块？跨系统？
  └─ 输出：问题清单 + 影响范围评估（1次clarify，不超过3个问题）

阶段2 ── 根因分析（领域分析 + 架构师）
  ├─ 数据流追踪：用户操作→前端→API→后端→持久化→返回路径
  ├─ 代码路径检查：涉及的文件→函数→数据流向
  ├─ L1/L2 缓存一致性检查（`_SESSIONS` 等内存缓存 vs 持久化）
  ├─ 判断问题类型：
  │   ├─ 表层症状（修复此处即可）
  │   ├─ 深层根因（症状只是表象，需修复底层）
  │   └─ 复合问题（一个表面现象掩盖多个根因）
  ├─ 关联模块标注：该 Bug 影响哪些系统/模块的回归测试
  └─ 输出：根因分析报告

  ⚠️ 根因深度升级机制（2026-05-06 重要教训）：
  
  当同一类 Bug 在本次 session 中第 2 次出现时 → **必须升级根因分析深度**：
  
  ```
  第1次出现: 分析"哪个函数导致了表面症状"
    → 修复: 改那个函数的输出
    ↓ (如果同类问题再次出现)
  第2次出现: 分析"这条数据流的完整路径是什么"
    → 修复: 追踪完整数据流，检查每个节点的语义
    ↓ (如果仍然出现)
  第3次出现: 分析"这个模块的数据模型设计是否有缺陷"
    → 修复: 检查数据字段是否被重载/覆盖/复用在不同语义上
  ```
  
  实战案例（C008 2026-05-06 同类 Bug 三次修复链）：
  
  | 轮次 | 分析深度 | 修复 | 结果 |
  |:----|:---------|:-----|:-----|
  | 第1次 | 函数级（get_fallback_prompt 输出有 {en}） | 去掉 {en} | ❌ 同类问题重现 |
  | 第2次 | 无关修复（语法题说明） | 加 cn_instruction | ❌ 不是同类修复 |
  | 第3次 | **数据流级（_enrich_question 覆写 cn 字段）** | 3 点位修复 | ✅ 彻底解决 |
  
  具体数据流追踪参考 `references/bug-fix-mode-d-worked-examples.md` 案例 3。
  
  **触发条件**：用户说"又出现之前的……"、"已经重复出现多次"、"请彻底解决"——这是质量信号，必须升级根因分析层。
  
  根因诊断检查清单（每次必查）：
  □ 数据流图是否完整？（输入→处理→输出→存储→下一次读取）
  □ 是否有双存储不一致？（内存缓存 vs 文件 vs DB）
  □ 修复后重启服务是否消失？（L1 缓存问题特征）
  □ 该问题是否曾在过去报告过？（搜索 session_search）
  □ 是否存在级联影响？（修 A 会不会影响 B/C/D？）

  实战根因追踪模式（从 C008 2026-05-06 两次修复中抽象）：

  **模式 A: 症状字符串溯源法**
  当用户报告一段可见的错误文字时，用 search_files 搜索该文字在代码库中的位置，从匹配行向上追溯函数定义，追踪该函数的调用链，确认是所有调用都受影响还是特定条件触发。实战案例（拼写题提示透露答案）：症状→search_files("请输入单词.*的正确拼写")→data_manager.py get_fallback_prompt() 返回 f"请输入单词 '{en}' 的正确拼写"→调用点 practice_engine.py 3处→根因: {en} 嵌入提示文字=把答案写在题目里。

  **模式 B: 问题类型三元分类**
  不是所有"用户报的问题"都是代码 Bug。先判断类型，再选择修复路径：

  | 类型 | 判断标准 | 行动 |
  |:-----|:---------|:-----|
  | 代码 Bug | 数据错误、逻辑错误、渲染异常、API 返回错误 | 代码修复（C005 Tier 0-2） |
  | 交互/UX 问题 | 功能正确但用户不知道要做什么、看不懂提示、操作路径不清晰 | 加中文说明/修改提示文案 |
  | 内容/课纲问题 | 题目正确但超出了学生当前年级/能力范围 | 告知用户知识点信息，由用户决定是否调整 |

  实战案例（语法题缺少中文说明）：用户报 "1/3 is read as ___" 不知道做什么。❌不是代码 Bug（题目格式正确、渲染正常）→✅是交互/UX问题（没有中文题型说明标签）→⚠️也是内容问题（分数读法是 G6 难度，孩子才 G3）。修复：加中文说明+告知用户该知识点超纲但由用户决定。

  **模式 D: 前端状态与后端不同步**\n  当用户报告「点击了 X 按钮但不生效」、「操作完成后状态未更新」、「下一题还是同一题」等问题时，检查该按钮的事件处理函数是否调用了后端 API。常见陷阱：前端只做了 UI 反馈（显示/隐藏元素）但未提交数据到后端，导致后端状态（session index、掌握度、记录等）停留在原地。\n\n  实战案例（C008 2026-05-06 "不知道"不翻题）：用户点击「不知道」按钮 → 前端只隐藏了提交/跳过按钮并显示正确答案 → 未调用后端 API → `current_index` 未被推进 → 点击「下一题」后后端返回同一题。修复：在 `skipQuestion()` 中追加 `fetch POST /api/v4/practice/submit` 提交空答案，触发后端索引推进。\n\n  **检查清单**：\n  □ 该按钮/操作有对应的后端 API 端点吗？\n  □ 前端事件处理函数末尾是只做了 DOM 操作还是也调了 API？\n  □ 如果调了 API，后端会更新状态（当前索引/掌握度/记录）吗？\n  □ 如果用户连续快速点击，是否有防重入保护？（`answerSubmitted` 标志）\n  □ 该操作导致的后端状态变化，在下一次请求中是否能被正确读取？
  当一个问题有多个修复层面时，用 clarify 向用户呈现选项，从轻到重排列，让用户选范围，只执行用户选中的。实战记录见 `references/bug-fix-mode-d-worked-examples.md`

阶段3 ── 修复方案设计（架构师）
  ├─ 设计最小化修复方案（尽量小改动，避免引入新 Bug）
  ├─ 标注关联模块（完整回归测试范围）
  ├─ 分配 C005 等级：
  │   ├─ Tier 0（<20 行，单文件，无依赖）：顾问团确认方案后直接执行
  │   └─ Tier 1（20-100 行，1-2 文件）：顾问团确认方案后走 C005 完整流程
  │   └─ Tier 2（100+ 行或跨模块）：顾问团确认方案后走 C005 Tier 2+
  ├─ 输出：修复方案文档（1-2 段描述）+ 测试策略

  ⚠️ 关键约束：同一 Bug 的修复不得 ≥ 3 次才正确
     → 如果某个 Bug 经过 3 次修复仍未解决，顾问团启动复盘：
       a) 之前的根因分析是否准确？
       b) C005 实施是否偏离了方案？
       c) 是否存在未发现的深层根因？

阶段4 ── C005 执行（实施团队）
  ├─ 按阶段3的方案执行修复
  ├─ 过程受 C007 监控（只需确认修复方向正确，无需全量重读）
  ├─ C005 Step 4 质量测试必须包含：
  │   ├─ 本次 Bug 的直接验证（确认修复生效）
  │   ├─ 受影响模块的回归测试（确认没有级联影响）
  │   └─ 质量门禁扫描（C015，新增/修改系统）
  └─ C005 Step 5 交付前，架构师做一次交付审计

  交付审计（C007 架构师，在 C005 Step 5 前执行）：
  □ 修复文件与问题匹配？（修对了地方吗？）
  □ 有注入新 Bug 的风险吗？
  □ 测试验证了原始复现步骤吗？
  □ 质量门禁通过了？

阶段5 ── 回归验证（质量保证）
  ├─ 执行阶段3标注的关联模块回归测试
  ├─ 重新模拟用户原始复现步骤（确保修复可感知）
  ├─ 检查服务端日志无新增异常
  ├─ 输出：修复确认报告（含：问题摘要 / 根因 / 修复方案 / 测试结果 / 有无回归）
  └─ 在 memory 中记录：Bug 摘要 + 根因 + 修复方案 + 日期

### Mode D 简化路径

对于非常明确的 Bug，可简化流程，跳过部分阶段：

```
明确条件（全部满足）：
  1. 用户报告了精确的复现步骤 + API 响应/日志
  2. 当前 session 刚做过该模块的代码审计（无需重新学习）
  3. 问题类型为「表层症状」（非深层根因）
  4. 修复范围 ≤ 3 行代码，单文件

简化路径：阶段1（记录确认）→ 阶段3（方案确认）→ 阶段4/5（实施+验证）
跳过：阶段2（根因分析）和阶段1 的完整影响评估
```

不满足任一条件 → 走完整 5 阶段流程。

## 用户交互命令系统（v3.0+）

在向顾问团描述任务时，可在输入中混入以下指令，无需离开当前对话：

| 指令 | 效果 | 示例 |
|:----|:-----|:------|
| `/skip` | 跳过剩余询问阶段，直接生成 Task Brief | `/skip 请设计一个登录页面` |
| `/fast` | 切换到快速模式（仅首席策略+领域分析，最多2轮）| `/fast 帮我分析这个需求` |
| `/reset` | 清空工作记忆和概要记忆，重新开始对话 | `/reset` |
| `/role add <名>` | 手动加载动态顾问（快速模式下拒绝）| `/role add tech_architect` |
| `/role remove <名>` | 卸载顾问（首席策略不可卸载）| `/role remove hr_specialist` |
| `/type <分类>` | 手动覆盖任务类型（tech/hr/finance/general）| `/type tech` |
| `/reject_last` | 撤销最近一次自动 apply | `/reject_last` |

### 组合使用

多条指令可在一轮输入中混用，按出现顺序执行：

```
/reset
/type tech
/fast
帮我设计一个用户认证系统
```

### 边界行为

| 场景 | 行为 |
|:----|:------|
| 纯指令无任务文本 | 返回 `command_only` 状态 + 指令反馈 |
| `/fast` 模式 + `/role add` | 拒绝，提示"快速模式下不支持加载动态顾问" |
| `/fast` 模式 + `/skip` | 输出快速模式的 skip Task Brief |
| `/role remove chief_strategy` | 拒绝，首席策略不可卸载 |
| `/type invalid` | 拒绝，列出合法选项 (tech/hr/finance/general) |

## ⚠️ 物理产品/硬件设计的特殊约束 — 材料/结构/布局三重假设验证

> 本节适用于模型飞机/机器人/工具架/3D打印件等**物理实物的设计任务**。软件系统的设计假设和经验法则不适用于物理世界。

### 核心教训

**教训1（2026-05-17 材料假设失败）**：用户在多个会话中描述「套材飞机竞速赛」，但从未明确说明机翼材料。我默认推断为「泡沫板」（航模常见材料），实际为「木板」。这个错误假设导致整份设计方案在翼载计算、制作工艺、CG预估上全部偏差，需要花一个完整的大版本（v2→v3）修正。

**教训2（2026-05-17 位置/布局约束失败）**：我基于「主翼在机身中部」的常规模型飞机布局完成了完整的多版本设计（v1-v8）。用户在v8后指出比赛规则要求「主翼只能和马达一起安装在飞机机身的最前面」——这是一个在多个会话中从未被确认的竞争规则约束。根因同样是「根本没有问」而不是「猜错了」。这说明物理产品的约束维度远多于软件系统。

**问题根因相同：默认推测而非主动询问。**

### 物理产品设计的三类假设验证清单

在进入Socratic 4轮询问的**第1轮（目标层澄清）**，必须主动确认以下三个维度：

#### 维度A：材料与结构

```
□ 机身材料是什么？（泡沫/木板/塑料管/金属/竹条/3D打印？）
□ 翼面（机翼/尾翼）材料是什么？（泡沫/轻木/层板/纸板？）
□ 翼面和机身的连接方式是什么？（胶粘/支架/插槽/绑扎？）
□ 各材料的大致厚度/截面尺寸？
□ 套材提供的木板有几块？尺寸如何？需要切割/拼接吗？
□ 电机/舵机/电子元件的安装方式？（螺丝/绑扎/热熔胶？）
□ 是否有预制的结构件（支架/加强片/连接件）？
```

#### 维度B：布局与位置约束

```
□ 主要部件的安装位置是否有限制？（竞赛规则/套材设计约束/安全要求？）
□ 主翼位置：必须在前？在中？在后？（如「主翼只能和马达一起安装在机头最前面」）
□ 尾部布局：尾翼/舵面在后部的空间是否有限制？
□ 重心范围：设计重心是否有特定的限制条件？（必需在机翼1/4弦附近？）
□ 电机安装位置与方向：能否调整右拉/下拉角？
□ 起落架安装位置：前三点还是后三点？轮距是否受限？
□ 其他部件的排布限制（电池盒、接收机舱等的位置约束）
```

#### 维度C：竞赛/使用规则约束

```
□ 是否有比赛规则文档？需要读取原文吗？
□ 尺寸限制：最大翼展/机长/翼面积？
□ 重量限制：起飞全重上限？
□ 动力限制：电机规格/桨叶尺寸/电池电压？
□ 材料限制：是否只能用套材提供的材料？
□ 是否允许对套材进行修改？哪些可以改哪些不能改？（切割、打磨、加装、减重）
```

### 实战流程修正

当设计对象为物理实物时：

1. **第1轮 Socratic询问** — 在问「目标是什么」之前，先问「材料和配件清单有什么」和「有什么位置/规则限制？」：
   - 让用户倒出所有配件、描述每件材料的外观和尺寸
   - 询问：「这些部件有固定的安装位置要求吗？」
   - 如果有竞赛/使用规则，请求提供原文或关键条款
   
2. **第2轮（约束展开）** — 确认以下物理约束：
   - 各个零件的重量（如果有秤）
   - 材料的刚度和脆性（会不会一摔就断）
   - 可加工性（能用什么工具切/钻/磨/粘）
   - 布局选项：还有多少挪动空间？哪些位置是固定的？

3. **设计中遇到不确定项** — 不要凭经验推断：
   - ❌ 「机翼在机身中部是标准布局，就这样吧」→ 还没确认规则是否允许
   - ❌ 「泡沫板应该会…」→ 用户还没说是什么材料
   - ✅ 「主翼位置在竞赛规则里有要求吗？我查下再出方案」
   - ✅ 「机翼的材料是套材里的木板还是额外购买的泡沫？」

4. **收到材料/位置修正后** — 全面级联更新（已验证流程）：
   - 更新材料/重量/刚度/位置假设
   - 重新计算CG、推力线、翼载、结构强度（布局变化会根本性改变这些参数）
   - 更新制作工艺（从泡沫热切割→木锉+砂纸打磨）
   - 更新所有三视图/装配图（部件位置变化需要重新绘制）
   - 更新所有示意图中的材料色标和标注
   - 生成新版本文档 → 上传飞书 → 发链接

### 与模式A（Socratic询问）的集成

在模式A的4轮询问中，如果是物理产品设计，将「材料与结构确认」作为**Pre-Step**，优先级高于Pre-Step的外部API侦察：

```
Pre-Step for Physical Products:
  ① 请用户列出所有套材/材料的完整清单（可用照片）
  ② 对每种材料问：可以改形状吗？可以切割、打磨、组合吗？
  ③ 确认连接方式：胶粘/螺丝/绑扎/插接/焊接？
  ④ 输出：材料与结构清单 → 进入Socratic 4轮
```

### 关键区分：纠正 ≠ 批评

用户提供材料/结构修正只是传递信息，不是对工作流程的批评。正确的回应方式是：
- ✅ 积极接受：「明白，我基于新信息重算所有参数」
- ✅ 全面更新所有受影响的决策（重量、CG、工艺、成本）
- ✅ 用新文档替换旧版本（不保留错误假设的内容）
- ❌ 不要道歉过多或自我批评——聚焦在「更新设计」而非「承认错误」

> - **触发条件**：任务涉及体检结果分析、健康指标解读、改善计划制定、疾病风险评估
> - **不替代**：专业医生诊断和建议。所有输出均标注「仅供参考，不构成医疗建议」
> - **典型场景**：年度体检报告解读、代谢综合征/慢性病风险分析、减重/运动/饮食改善计划、压力与睡眠管理
> - **评估维度**：① 指标异常优先级排序（从紧急到常规）② 各指标间的关联性分析（如脂肪肝+ALT升高+血糖升高的代谢综合征链）③ 生活方式干预建议的可执行性 ④ 随访/复查时间建议
> - **协同机制**：个人健康专家评估后，可将「改善计划」传递给首席策略/领域分析顾问，纳入整体方案执行

## 关键行为规则

1. **必须先和用户对话**，不能模拟顾问团输出
2. **严禁跳过讨论直接进入架构设计或系统搭建** — 顾问团的核心产出是方案文档，不是代码。即使你觉得"很明确可以直接做了"，也必须走完4轮询问+用户确认
3. **产出方案文档后停止** — 方案文档统一存放到飞书「我的系统/顾问团报告」文件夹
4. **实施交由 C005 编码工作流或其他团队执行**
5. 如果用户要求"直接执行"（说出明确的执行指令），熔断生效，直接执行
6. **方案产出后必须召开内部评审**，不可直接交付
7. **内部评审后需有用户确认环节**，用户说不清楚的问题不超过3个，集中一次确认

### 两种评审模式区分

顾问团评审需根据评估目标选择正确的模式：

| 模式 | 触发条件 | 评估焦点 | 典型输出 |
|:----|:---------|:---------|:---------|
| **Post-hoc Audit（事后审计）** | 用户报告系统问题（bug/数据不一致/功能缺失） | 7维检查清单 + 数据流追踪 + 深层根因发现 | 根因分析 + 修复方案 + 测试用例注册表 |
| **Pre-enhancement Audit（增强前评估）** | 用户要求对已有的新模块进行评审并提改进方案 | 功能完整性 + 教育/产品设计合理性 + 与现有系统集成度 | 增强方案 + 执行矩阵 + 集成路径 |

**选择判断**：用户说"系统有问题" → post-hoc audit。用户说"帮我评估这个新功能，看怎么做得更好" → pre-enhancement audit。

### Pre-enhancement Audit 评估维度和流程

当用户要求对**已完成的但尚未充分优化的新模块**进行评审时，走以下评估流程：

```
第1轮 — 领域专家评估（如教育专家）
  评估维度：
    a) 功能覆盖度 — 知识点体系是否完整？有无明显遗漏？
    b) 教学法合理性 — 练习方式、掌握度模型、反馈机制是否科学？
    c) 数据流审视 — 信息在模块内的流转是否有断点？（如：试卷分析结果是否被练习引擎使用？）
    d) 与系统内现有教育设计的对标 — 词汇系统已有的SM-2/错题本/星星币等，语法模块是否缺失？
  输出: 缺陷清单 + 优化建议

第2轮 — 产品经理评审
  评估维度：
    a) 用户体验 — 学习路径是否清晰？用户理解成本？
    b) 优先级过滤 — 哪些改进项是用户真实痛点，哪些是理论完美但实际低价值？
    c) 用户习惯 — 与现有模块（如词汇）的操作一致性
  任务：对第1轮建议做二次过滤
  输出: ✅立即采纳 / ⚠️条件采纳 / ❌暂缓弃用

第3轮 — 系统架构师评审
  评估维度：
    a) 数据架构一致性 — 新模块的数据存储方式是否与系统统一数据层一致？
    b) 集成点识别 — 需要对接哪些现有模块（星星币/错题本/角色管理等）？
    c) 改动量预估 — 每个改进项的文件修改范围、工时、风险
  输出: P0/P1/P2 执行矩阵 + 技术方案

第4轮 — 综合决策
  合并三轮评审 → 综合决策表
  用户确认执行范围
```

### 评估路线图：如何处理"用户要求全部P0一次到位"

本用户的典型模式偏好：当改进项在可控范围内（5-8个改进点，总工时≤6小时），用户倾向于**一次性全部执行**而非分期实施。在产品经理和架构师给出P0/P1/P2分级后，不要默认用户会选择分两期，而是主动询问用户的执行偏好。通常的对话流：

```
架构师输出执行矩阵 → 询问：先做P0观察效果再推二期，还是一次到位？
```

注意：不要假设用户总是选择分期或一次到位；基于本次任务的复杂度询问即可。

### 教育系统新模块评估的典型发现模式

从C008语法模块评估实战中抽象出的常见问题类型：

| 发现问题类型 | 典型例子 | 检查方法 |
|:------------|:---------|:---------|
| **数据流断点** | 试卷分析结果存入exam_analysis.json，但练习引擎只用预定义模板——分析结果从未被用于练习 | 追踪每个数据文件的写入方和读取方，画完整数据流图 |
| **教育设计未复用** | 词汇系统已有SM-2记忆曲线、错题本、星星币，但语法模块用线性掌握度模型、无错题本、无金币 | 对标系统内已有的教育设计组件，逐项检查新模块是否缺失 |
| **知识点体系稀疏** | 动词类只覆盖了be/can/must/do，缺have/has用法、like+doing、三单一致性 | 按年级大纲逐级核对知识点覆盖 |
| **UI翻新导航退化** | v4 门户丢失了 v3 老门户已有的积分规则/兑换/词库/分类说明等导航链接——新 UI 重写时只复制了核心功能链接，辅助入口被遗漏 | 打开旧版门户的 HTML，`grep -oP 'href=\"[^\"]+\"' old_portal.html | sort -u` 提取所有链接，与新版门户比对，找出缺失项 |
| **练习引擎与考试脱节** | 学生分析了一堆试卷，但练习时却用不到试卷中的原题 | 检查练习引擎的题目来源，是否包含试卷分析后的题目 |
| **导航死链与模板渲染失效** | 概览页导航菜单全部 `href="#"`、知识点总览页因模板变量名不匹配而永久空状态、缺少必需的模板文件导致403/404 | 逐页点击所有导航链接确认是否指向真实路由；用 Flask test_client 渲染每个页面并检查 HTML 内容（不仅仅是 HTTP 状态码），验证模板变量名与路由传参匹配；检查 `grep -n 'href=\"#'` 发现死链 |

**导航死链与模板渲染失效陷阱详解（C008 语法模块实战案例 2026-05-05）**：
一个刚完成 SM-2/题库/错题本/星星币/LLM进阶题全功能升级的模块，用户却说\"没有上传试卷功能\"。
```
用户报告: \"我找不到上传试卷功能\"
  → 检查路由: /parent/grammar/analyze 存在且有上传标签页 ✅
  → 检查导航: 从概览页到试卷分析页的链接全部是 href=\"#\" ❌
  → 用户尝试了3种方式都找不到，最终从 /english/parent/ 报告此问题
  → 更深层: 知识点总览页永远为空（模板用 knowledge_points，路由传 all_points）
            概览页掌握度永远0%（硬编码值）
            最近动态永远为空（硬编码空列表）
```

**根因模式（已从2次独立评估中验证）**：
Flask 新模块在快速开发中容易产生三类渲染问题，这三类问题经常组合出现，任何一个都会导致功能对用户不可见：
1. **导航死链** — 模板中的导航链接硬编码为 `#`，未绑定真实路由 URL。`grep -rn 'href=\"#' templates/` 可以快速定位。
2. **模板变量名不匹配** — 路由函数用 `render_template(..., posts=posts)` 但模板写 `{% for p in articles %}`。当模板 `{% if knowledge_points %}` 永不满足时，整个页面退化为空状态，不报错也不崩溃。
3. **必填模板缺失** — 路由 `render_template('exams_list.html')` 但该文件不存在，导致 HTTP 500 或 Jinja2 TemplateNotFound（在 try/except 中被吞噬时表现为空白页面）。

**诊断套路**：
```python
# 检查1: 死链
import re
html = c.get('/some-page/').data.decode()
dead_links = [l for l in re.findall(r'href="([^"]+)"', html) if l == '#']
print('死链:', dead_links)

# 检查2: 变量名匹配
# 找到 render_template 调用，对比模板内容
grep 'render_template.*html' routes.py
grep '{% for\|{% if\|{{\s*[a-z]' templates/xxx.html

# 检查3: 模板文件存在
ls -la templates/xxx.html

# 检查4: 渲染内容深度
html = c.get('/page/').data.decode()
# 不仅仅检查 HTTP 200，还要检查页面实际包含的内容
assert '预期的知识点名称' in html, '页面内容为空!'
assert len(html) > 5000, '页面太小，可能只渲染了骨架'
```

**修复套路**（验证可15分钟内完成）：
```python
# 修复导航：将 sidebar/navbar 中的 href="#" 替换为实际路由
sed 's|href=\"#\"|href=\"/real/url\"|' template.html

# 修复变量名：确保 render_template 的命名参数与模板中的变量名一致
# 如果拿不准，传两份（新旧变量名都传）
render_template('page.html', new_name=data, old_name=data)

# 补充缺失模板：从类似页面复制改造
cp similar_template.html missing_template.html
```

### 采用 clarify 工具时的陷阱预防

在模式C的多轮决策中，`clarify` 工具被频繁使用。以下陷阱已验证：

**陷阱1：选项文本方向不一致**
示例：`[语法用独立的SM-2参数（间隔更长，因为语法需更频繁练习）]`
→ "间隔更长" 与 "需更频繁练习" 方向相反，让用户困惑。
→ 修复：每个选项的描述必须是自洽的。如果选项有一条属性，确保其描述语义统一。

**陷阱2：选项数量超出认知负荷**
每次clarify不超过3个问题，每个问题2个选项。超过3个问题或3个选项以上时，用户选择质量下降。

**陷阱3：未确认选项语义即推进**
当选项文本有歧义时，不要等用户选完才追问——立即用follow-up clarify澄清，节省往返轮次。


### 核心原则

**表面症状 ≠ 根因**。用户提到的3个问题背后可能隐藏着更多问题。本节的 C008 实战案例中，用户报告了3个问题，但实际发现了4个问题（含一个隐藏的数据分裂问题）。

### 7 维评估检查清单

每次系统评审必须逐一检查以下7个维度，并在评审报告中记录每个维度的评估结果：

| # | 评估维度 | 检查方法 | C008 案例中发现的问题 |
|:-:|:---------|:---------|:---------------------|
| 1 | **数据流图完整性** | 追踪每个数据文件的写入路径和读取路径，画出完整的数据流图 | state.json 有2条写入路径（record_answer写顶层+迁移写students层），但只有1条读取路径 |
| 2 | **级联操作覆盖面** | 删除/更新操作是否覆盖了所有数据文件？逐一列出每个数据文件 | delete_student_cascade只清理3/7个文件，遗漏state.json和3个v3文件 |
| 3 | **读写路径一致性** | 迁移/重构后，所有旧读写函数是否更新到新路径？检查 import 依赖链 | v1→v2迁移增加了students维度，但record_answer()仍按v1格式写顶层 |
| 4 | **功能边界互斥性** | 每个模块/Blueprint的职责是否互斥？一个模块不应同时做两件事 | v2 Blueprint同时承载家长管理和学生功能，v3只有部分学生功能 |
| 5 | **数据存储冗余** | 同一概念（如听默记录）是否存储在多个数据文件中？ | state.json + v3/dictation_records.json 两套数据源，且不同步 |
| 6 | **路由命名语义** | URL 路径是否真实反映功能归属？用户能否从路径猜出功能域 | /v2/dictation/ 是学生功能却以v2为前缀，误导用户认为这是个通用接口 |
| 7 | **内容管理完备性** | CRUD 操作是否覆盖所有实体类型？是否存在「能添加但不能查看/删除」的实体 | 可以批量导入单词/词组/短句，但单词库页面只展示单词，无词组/短句标签页，无单条删除功能 |

### 深层问题发现技巧

**技巧1：数据追踪法**
```
用户报告: "删除学生后记录还在"
  → 只修级联删除是不够的
  → 先问: "state.json 有多个写入路径吗？"
  → 追踪 state.json 的所有写入操作（grep save_state / record_answer / record_dictation 等）
  → 发现: record_answer() 写顶层, v2 migration 写 students.stu_default
  → 根因: 迁移不完整 → 数据碎片化
```

**技巧2：重复数据检测法**
```bash
# 检查 state.json 中是否存在两个相同 key 在不同层级
python3 -c "
import json
s = json.load(open('data/state.json'))
top_keys = [k for k in s if k not in ('version','students','exercises')]
stu_keys = [k for k in s.get('students',{}).get('stu_default',{}) if k != 'exercises']
print(f'顶层批次: {top_keys}')
print(f'students下批次: {stu_keys}')
overlap = set(top_keys) & set(stu_keys)
if overlap:
    print(f'⚠️ 数据分裂: {overlap}')
    # 进一步比较数据是否一致
"
```

**技巧3：边界检测法**
```
用户说"3个问题" → 先假设还有第4、第5个
  → 检查新增功能的 API 是否完整（CRUD 缺哪个？）
  → 检查数据清理操作是否覆盖所有数据文件
  → 检查版本升级后的数据迁移是否幂等
  → (C008 案例: 发现了第4个隐藏问题 — state.json 数据分裂)
```

### C008 实战案例

> 以下案例记录了 C008 英语模块评审全过程，作为后续评审的参考模板。

```
用户报告: 3个问题
  ① 功能划分混乱（v2 同时含家长和学生功能）
  ② 数据不一致（删除学生后记录残留）
  ③ 内容管理不完整（无词组/短句显示，无单条删除）

执行7维检查后，额外发现:
  ④ state.json 数据分裂（顶层写新数据，students层存旧数据）
     → 根因: v1→v2 迁移不完整，record_answer() 仍写旧路径

解决方案:
  前置: 合并 state.json 分裂数据 + 修复 record_answer() 写入路径
  Task A: 路由重构 /v2/→/parent/, /v3/→/student/
  Task B: v2 学生功能代码剪切到 v3 Blueprint
  Task C: delete_student_cascade 补全7个数据文件清理
  Task D: 单词库增加词组/短句Tab + 单条删除API
  Task E: 测试用例注册表
  Task F: 本 skill 更新（即本文）

关键教训:
  - 用户报告的3个问题中，第②个（删除级联）是症状，第④个（数据分裂）是根因
  - 只修级联删除而不修数据分裂 → 后续操作继续在错误数据上叠加
  - 所有评估必须从「数据流追踪」开始，而不是从「用户描述的问题」开始
```

### 设计文档格式规范（用户偏好 — 硬约束）

> 用户夜雨对设计文档有非常具体的格式偏好。违反这些偏好会导致返工。

#### 核心原则：完整独立文档，非增量修改

- **设计文档必须是完整的独立文档**，包含所有此前已确认的内容 + 本次新增/修改内容。用户原话：「需要最终的完整设计方案（包括之前已经确认的）」。
- ❌ 不要只输出「本次修改的部分」或「增量差异」——即使改动很小，仍然需要呈现完整的方案。
- ✅ 合并旧版全部内容 + 新增内容，输出完整的 vN+1 版本。

#### 飞书文件上传流程

| 步骤 | 操作 | API | 关键参数 |
|:-----|:-----|:-----|:---------|
| 1. 生成本地 HTML | `write_file` 到 `~/.hermes/cache/` | — | 含完整 CSS + UI 线框图 |
| 2. 上传到 Drive | `POST /open-apis/drive/v1/files/upload_all` | multipart/form-data | `file_name`, `parent_type=explorer`, `parent_node=<folder_token>`, `size`, `file`(binary) |
| 3. 分享给用户 | `POST /open-apis/drive/v1/permissions/{file_token}/members` | JSON body | `member_type=openid`, `member_id=ou_50b21c92548fbb2173b049e57dfbdbec`, `perm=full_access` |
| 4. 分享类型 | 普通文件用 `?type=file`，文档用 `?type=docx` | — | 分享失败时切换尝试 |

⚠️ 不要用 feishu-md-writer.py 将 HTML 转为 Docx 内容提交——用户要的是**原始 HTML 文件**放在飞书云盘，不是 Docx 中的文本摘要。

### 格式铁律

| 规则 | 说明 |
|:-----|:------|
| **主交付物 = HTML 文件** | 设计文档必须以 **HTML 格式** 作为主交付物（含完整 CSS + 嵌入 UI 线框图），上传到飞书云盘作为文件。不要只用 Markdown 或 Feishu Docx 作为主要版本。 |
| **UI 线框图嵌入 inline** | 每个页面必须有对应的 UI 线框图（用 HTML/CSS 绘制嵌入文档中），而不是仅用文字描述布局。用户原话：「我需要的是html完整版」。 |
| **条条目 + 表格，避免散文** | 功能规格用表格和项目符号列表组织。整段散文段落会降低可读性。每页只说一件事。 |
| **无 AI 式套话** | 避免「赋能」「发力」「助力」「夯实」「抓手」等空洞词汇。用实打实的操作描述。 |
| **必有目录 + 版本号** | HTML 顶部包含目录和版本号（v1.0, v1.1, v1.2...）。 |
| **不写空话** | 每个功能点必须有具体实现方式，不做模糊承诺。 |

## 🔴 交付铁律（强制执行 — 涵盖所有模式 A/B/C/D）

> 此规则嵌入在所有 `产出` / `交付` / `输出` 步骤末尾。不允许将设计文档、方案文档留在本地或仅输出到对话中。

### 两步缺一不可

无论哪种模式，方案文档/HTML/设计说明书生成后，立即执行以下两步——中间不做其他事：

| 步骤 | 操作 | 关键参数 |
|:-----|:-----|:---------|
| **1. 上传飞书云盘** | 用 `upload_all` API 上传到对应文件夹（个人项目→「个人」、系统方案→「项目」、顾问团报告→「顾问团报告」） | `parent_node` = 目标文件夹 token |
| **2. 分享到对话** | 先调用 Permission API 给用户 `full_access`，再获取/构建链接，最后发送到当前对话 | `member_id = ou_50b21c92548fbb2173b049e57dfbdbec` |

**常见错误（需避免）：**
- ❌ 只输出对话中不存飞书 → 用户无法后续访问
- ❌ 只存本地 → 用户看不到
- ❌ 只链接不发 → 文档不在云盘

**链接获取方式：**
- 上传后从 `upload_all` 响应拿到 `file_token`
- 从同层级文件夹 `List files` API 中获取文件的 `url` 字段
- 飞书文件链接格式：`https://rcncjr4hag8s.feishu.cn/file/{file_token}`

违反此规则 → 主动道歉并立即补做。
