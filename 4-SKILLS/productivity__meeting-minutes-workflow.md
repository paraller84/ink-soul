# 会议文字稿处理工作流

## 使用场景
- 收到会议录音文字稿（原始转录或Get笔记AI纪要）
- **自动拉取Get笔记中的新会议记录并处理（API集成模式）**
- 需要从会议中提取待办事项并分配责任人
- 会议涉及关键领导（老苏、翟总等），需要分析其沟通风格
- 需要将领导风格观察沉淀到人生罗盘（C010）关键人画像中

## 工作流步骤

### Step 0: 说话人指认（必须先做，不可跳过）

在进入待办提取、风格分析等任何下游步骤之前，**必须先完成说话人身份映射**。不能假设已知映射关系。

**正确的指认方法（用户确认过的工作流）：**

1. 从会议**完整转写原文**（而非AI摘要）中，提取每个"说话人X"的 **最具识别性的一段话**
   - 每个说话人选2-4段早期发言，筛选条件：>20字，包含具体人名/项目/业务术语
   - 说话人短到只有"嗯"的，标注"仅X段回应型发言"
2. 整理成格式：
   ```
   【说话人1】— 共N段发言
     ▶ 具体发言段落1（包含人名/项目/数据等识别信息）
     ▶ 具体发言段落2
   
   【说话人2】— 共N段发言
     ▶ ...
   ```
3. **呈现给用户指认** → 等待用户回复映射关系
4. **用户确认后**，将映射关系持久化记录，供后续同会议复用

**陷阱：**
- ❌ 不要自行推断"说话人1=老苏"——用户的同事你可能认错
- ❌ 不要跳过指认直接进入待办提取——无标签的待办无法分配责任人
- ✅ 要从录音转写的**完整文本**（`## 录音转写原文`部分的逐字稿）中提取，不是AI摘要
- ✅ 用户明确说过的工作流："提供每个需要指认人说的最具备识别性的一段话,方便我指认"

### Step 1: 读取并理解会议内容
- 获取完整文字稿（docx→Python提取，或直接读取.md）
- 如为Get笔记格式，重点关注其结构化摘要（📑智能总结、📅章节概要、📋待办事项、✨金句精选）
- **说话人映射已由Step 0确认**，后续步骤直接使用
- 记录：会议时间、时长、参与人数、核心议题（用于📑会议概况）
- **陷阱**：docx无法直接read_file，必须用Python zipfile或pandoc提取

### Step 2: 提取待办事项（🎯核心输出）
按三级优先级分类：

| 级别 | 含义 | 时间要求 | 标记 |
|------|------|---------|------|
| 🟥 **P0** | 紧急且必须马上执行 | 天级别（3天内） | 加🔴 |
| 🟨 **P1** | 重点跟进 | 周级别（1-2周） | 加🔴如涉及用户本人 |
| 🟩 **P2** | 中期推进 | 月级别或持续 | 不加特殊标记 |

每项待办需包含：事项描述、目标/要求、截止时间、责任人、备注

**必须标记"用户重点关注"的场景：**
- 老苏或翟总直接点名用户负责的事项
- 用户本人自述"我来/我已经"的事项
- 跨部门需要用户亲自协调的事项
- 需要用户准备汇报材料的事项

### Step 3: 领导风格分析（仅限关键领导，如老苏）
从会议内容中观察并提炼：

**整体风格画像表：**
| 维度 | 特点 | 原文印证 |
|------|------|----------|
| 气场 | 强势/温和/引导型 | 可用原文佐证 |
| 语言 | 粗犷/严谨/幽默 | 脏话频率、比喻习惯 |
| 决策 | 当场拍板/集体商议/犹豫观望 | 关键决策时刻 |
| 思维 | 先宏观再微观/细节导向/战略导向 | 提问顺序和追问深度 |
| 问责 | 直接点名/含蓄提示/集体承担 | 分配责任的方式 |
| 给方案 | 亲自出方案/让团队自己想办法/给方向 | 问题解决模式 |
| 比喻 | 生动接地气/抽象理论/案例举例 | 习惯性比喻 |

**沟通方式特征：** 5-8条具体观察（开门见山、追根究底、现场算账、不接受借口、重视落地等）

**管理风格特征：** 3-5条（敢给资源也敢压担子、重视IT懂业务、抓大放小、要过程更要结果、有战略眼光等）

### Step 4: 更新关键人画像（C010生命罗盘）
- 将Step 3的风格观察更新到 `~/wiki/raw/systems/life-compass/portraits/workplace/<人物>.md`
- 更新内容：信息代谢模式、权力与政治地图、信任语法、时间权力场、恐惧与荣耀
- 每个更新标注 `⭐新发现` 来源为本次会议
- **置信度调整**：已有观点在本次会议中得到验证的，置信度⬆️；被推翻的⬇️
- **新增维度**：如本次会议发现的新特征，新增到对应维度

### Step 5: 提炼金句（💡）
- 从会议中摘录5-10条领导原话
- 标注类别：执行策略/战略洞见/思考启发/底线原则/问责风格/追问风格
- 简要标注使用场景（用于日后汇报时引用或说服协作方）
- 更新到关键人画像中新增的"金句集"章节

### Step 6: 输出结构化会议纪要
按以下模板输出：

```
📑 会议概况（时长、人数、核心议题）
🎯 待办事项清单
  🔴 P0 - 紧急待办（含您重点跟进标记）
  🟡 P1 - 重点跟进（含您重点跟进标记）
  🟢 P2 - 中期推进
  🔴 您的重点事项一览（汇总表）
👤 领导风格观察
💡 金句摘录
🔍 关键根因分析（数据问题类会议特有）
```

## Get笔记 API 集成（v2.0 2026-05-10）

> Get笔记已通过 MCP Server（iswalle/getnote-mcp v1.3.2）接入 Hermes，提供 26 个工具。
> 凭证存储在 `~/.hermes/.env.getnote`，MCP 配置在 `config.yaml mcp_servers.getnote`。
> 需 `/reset` 后 MCP 工具才出现在工具集中（否则手动调 REST API 代替）。

### 可用 MCP 工具（会议处理相关）

| 工具 | 用途 | 会议处理场景 |
|:-----|:-----|:------------|
| `recall` | 全局语义搜索（top_k ≤ 10） | 搜索相关会议：「查找上周关于EAST的会议」|
| `list_notes` | 浏览笔记列表（20条/页，cursor分页） | 拉取最新笔记，检测增量 |
| `get_note` | 获取笔记详情（含 audio.original 转写原文） | 获取完整的会议转写+AI摘要+待办 |
| `save_note` | 新建笔记（plain_text/link/img_text） | 写回结构化会议纪要 |
| `list_topics` | 查看知识库列表 | 按主题归类会议笔记 |

### API 直接调用（无需 MCP）

当 MCP 工具不可用（新会话前），直接调 REST API：

```bash
# 全局语义搜索（最常用）
curl -s -X POST "https://openapi.biji.com/open/api/v1/resource/recall" \
  -H "Authorization: gk_live_xxx" \
  -H "X-Client-ID: cli_xxx" \
  -H "Content-Type: application/json" \
  -d '{"query":"关键词","top_k":5}'

# 笔记列表（增量检测）
curl -s "https://openapi.biji.com/open/api/v1/resource/note/list?cursor=0" \
  -H "Authorization: gk_live_xxx" \
  -H "X-Client-ID: cli_xxx"

# 笔记详情（含音频转写原文）
curl -s "https://openapi.biji.com/open/api/v1/resource/note/detail?id=123456" \
  -H "Authorization: gk_live_xxx" \
  -H "X-Client-ID: cli_xxx"
```

### 会议笔记字段映射

Get笔记 API 返回的 `get_note` 详情中的关键字段：

| API 字段 | 含义 | 在会议处理中的使用 |
|:---------|:-----|:------------------|
| `content` | 笔记正文（含 AI 智能总结） | Step 1 阅读全文 |
| `audio.original` | 语音转写原文 | 获取完整逐字稿 |
| `audio.duration` | 录音时长（秒） | 会议时长 |
| `tags[]` | 标签列表 | 可用于识别会议类型 |
| `created_at` | 创建时间 | 会议时间 |
| `web_page.excerpt` | 链接笔记的 AI 摘要 | 若会议链接了参考资料 |

### 会议笔记特征检测

Get笔记中的会议笔记通常具有以下特征，用于自动化识别：

```python
# 判断一条笔记是否为会议笔记
def is_meeting_note(note):
    content = note.get('content', '')
    title = note.get('title', '')
    
    # 特征1：Get笔记的会议录音会自动生成结构化摘要
    if '时长' in content and '参与人数' in content:
        return True
    
    # 特征2：标题含会议关键词
    meeting_keywords = ['会议', '周会', '汇报', '讨论', '评审', '月会', '例会']
    if any(kw in title for kw in meeting_keywords):
        return True
    
    # 特征3：含章节概要（Get笔记会议录音的标志性结构）
    if '章节概要' in content and re.search(r'\[\d+:\d+:\d+\]', content):
        return True
    
    return False
```

## 会议跟进汇报的范围纪律（关键！）

### 仅限汇报对象亲自召集的会议

向某位领导做阶段性情况汇报时，**只纳入该领导亲自召集的会议中的交代事项**。其他同期进行的会议（即使是同一工作组的相关会议）不属于本次汇报范围。

**典型场景：** 用户说"基于老苏召开的专题会中重点交代事项" → 检索所有近期会议，逐一核对该领导是否作为会议召集人（而非仅参与）。

**执行步骤：**
1. 从历史会话/记忆收集近期所有相关会议清单
2. 逐一确认每场会的**召集人**（谁组织的会，而非谁参加了会）
3. 仅保留汇报对象亲自召集的会议及其交代事项
4. 如无法确认召集人，**先问用户**，不要自行推断

**反例（本次会话修正）：**
- ❌ 5/7手续费率排查会（老苏召集）+ 5/9周例会 + 5/9产险质量会 + 5/9五报四会
- ✅ 仅5/7手续费率排查会（仅此一场是老苏召集的）

### 排查发现问题 vs 交代事项的区分

当会议类型为**问题排查/调查/核查类会议**时，汇报内容分两类呈现：

| 类别 | 说明 | 示例 |
|------|------|------|
| **交代事项（task）** | 会议上明确分配的待办工作 | 手续费率异常数据核查清零、4月数据报送 |
| **排查发现（finding）** | 排查过程中新发现的问题，非会议明确交代，但领导需要知晓 | 保批单费率不一致、责任终止后批单操作、险种归类不匹配 |

**规则：** 排查发现必须单独呈现，不可混入交代事项中。排查发现的状态统一标注为"已发现并记录，待业务/技术侧确认处理方案"。

### 需领导协调事项的边界

- 仅标注**确实需要该领导层面推动**的事项（跨部门、外部资源、业务决策）
- 本方可直接推动的（内部跟进、确认进度）**不要**放入"需领导协调"清单
- 优先级标注：🔴 紧急/影响进度 / 🟡 重要但不紧急

### 产出结构模板（排查类会议跟进汇报）

```
收件人：汇报对象
主题：XX会议重点事项推进情况汇报（截至X月X日）

---

【汇报对象】，您好：

以下是X月X日您召集的XX会议中重点交代事项的推进情况。

（已完成的任务，按交代事项列表呈现）

### 二、排查中新发现的问题
（排查过程中发现的意外问题，不受限于原交代事项清单）

### 三、科技资源与外部协作
（涉及外部资源、工具采购、人员调配等事项）

---

1. 🔴 事项A — 原因说明
2. 🟡 事项B — 原因说明
```

## 输出内容原则（强制性）

### ⚠️ 会议转写只输出摘要，不输出完整文字记录

**这是底线规则，不可违反。**

收到会议录音转写/Get笔记转录后，**只输出结构化摘要**，绝不输出完整的逐字转写原文。摘要必须精炼，包含：

| 要素 | 说明 | 是否必须 |
|:----|:-----|:--------:|
| **会议主题** | 会议名称/时间/召集人 | ✅ |
| **核心议题** | 本次会议讨论的1-3个关键议题 | ✅ |
| **结论/决策** | 讨论后的结论、拍板事项 | ✅ |
| **待办事项** | 谁在什么时候前做什么（分级：P0/P1/P2） | ✅ |
| **需关注重点** | 需要你特别留意或亲自执行的事项 | ✅ |

**绝不包含：**
- ❌ 逐字转写 / 完整发言记录
- ❌ 会议过程描述 / 时间线流水账
- ❌ 无关紧要的对话片段

> 用户明确要求：「只需要会议的大致摘要，不需要完整的会议全部文字记录」

## 输出格式规范（关键！）

### 飞书中呈现待办清单
**Feishu 表格渲染陷阱**：复杂表格（多行列 + 单元格内换行）会被飞书截断，导致部分内容不可见。

✅ **推荐格式** — 分段式 bullet list：

```
### 🔴 P0 — 紧急待办

**T1** 事项标题 | 责任人 | 截止日
→ 具体描述第一行
→ 具体描述第二行

**T2** ...
```

❌ **避免格式** — 跨度过大的 pipe 表格：
```
| 事项 | 目标 | 截止 | 负责人 | 备注 |
| T1 | 很多内容... | 很长的时间... | 很多人... | 更多备注... |
```

### 微信汇报格式（结论前置）

向领导（如老苏）在微信群发进展汇报时，遵循以下结构：

```
【标题 — 会后第N天】

第一段：待办总览（一句话带出会议明确了多少项待办，今天启动了哪些）
→ 展示"我全面掌握了会议内容"

第二段：每项进展（结论先行，再展开）
① **已完成项** ✅ — 第一句就是结论，再讲细节
② **进行中** — 标明状态，说明下一步
③ **有发现** — 先说发现，再讲详情

第三段：明天重点（简洁收尾）
→ 让领导知道明天节奏
```

**关键原则**：
- 结论先行（老苏前30秒抓不住重点会失去耐心）
- 状态标记 ✅ / 进行中 / 有发现（一眼看清进度）
- 数字锚点（具体时间、具体数据）
- 完整性检查（必须覆盖会议所有关键待办，用户会逐条核对）
- 篇幅控制在手机一屏内（约500-800字）

## 附录 A: 跨会话恢复与多源交叉验证 (从 meeting-minutes-analysis 合并)

### A1. 跨会话恢复

当用户说"今天前面提供过"时，用 `session_search` 搜索关键字（"会议录音"、"文字稿"、"说话人"、人名等），找到历史缓存的文档文件（`~/.hermes/cache/documents/`）。

### A2. 多源交叉验证

使用三种数据源互验，避免遗漏：
1. **原始录音文字稿**（完整对话）— 获取原始细节和隐性待办
2. **AI纪要/Get笔记**（如有）— 获取结构化摘要（注意：AI可能遗漏用户发言，标注"未包含(我)的说话人ID"）
3. **用户自身笔记**（如有）— 获取用户视角

### A3. 六类异常场景框架

提取数据/技术问题时，按以下框架归类：

| 类别 | 优先级 | 说明 |
|------|--------|------|
| 技术类 | 高 | 程序错误、字段取错、逻辑缺陷 |
| 业务类 | 中 | 口径差异、业务规则导致、业务操作不规范 |
| 管理类 | 高 | 人力不足、流程缺失、系统管控缺乏 |

### A4. 隐式待办检测模式

老苏式领导的特点是不直接说"你去做XXX"，而是说：
- "你应该XXX" → 待办
- "你那个XXXX搞明白没有" → 需要跟进
- "我建议你XXX" → 必须执行
- 拍板定量的（"翻一翻"、"10个人"）→ 立即落实

### A5. 输出模板（表格版）

当用户明确要求表格格式而非 bullet list 时使用：

```markdown
# 📋 待办事项清单

## 🔴 P0 — 紧急待办

| # | 事项 | 要求/目标 | 截止日 | 责任人 | 备注 |
|---|------|-----------|--------|--------|------|

## 🔴 涉及您的重点事项（汇总）

| 优先级 | 事项 | 您需要做的 | 时间 |
|--------|------|-----------|------|
```

### A6. 老苏会议特别要点

- 对人名敏感——林星/林欣、宣鹏超/轩辕鹏超、张兆宇要确认正确写法
- 对数字敏感——汇报时数据要精准，不要"大概、可能"
- 当场拍板的事第二天就要看到进度
- "别的事99分无所谓，这个60分都没了"——抓重点

## 陷阱与注意事项
1. **不要只搬Get笔记的待办列表** — 它没有优先级和角色识别，必须人工补全
2. **不要等到会后问用户需要什么** — 直接分析后给出结构化输出，再问用户是否需要调整
3. **必须识别说话人3（🔵）是用户自己** — 这是标记重点事项的前提
4. **老苏金句要标注使用场景** — 方便日后在汇报中引用时知道在什么语境下说的
5. **人物画像更新后不需要通知用户** — 属于后台沉淀动作，只在用户提及相关话题时引用
6. **P0时间节点永远标注具体日期** — 老苏对deadline极敏感，不能模糊
7. **"用户重点关注"标记只针对用户亲自执行的事项** — 团队执行但用户监督的用"您盯进度"而非"您亲自"
8. **在飞书中输出待办清单不要用复杂 pipe 表格** — 用分段 bullet list 替代，飞书会截断长表格
9. **微信汇报发出前做一次完整性检查** — 逐条核对会议待办是否全部覆盖，遗漏会破坏领导对你全面掌控的印象
10. **微信汇报尾部固定"明天重点"** — 让领导知道下一步节奏，制造持续跟进的预期
11. **⚠️ 检查会议日期** — 用户说今天的会议，不要处理昨天的。确认当前日期（今天 = 对话时的日期），按日期过滤待办来源
12. **⚠️ 创建飞书待办前先去重** — C013管道已经自动同步了大部分带待办的会议笔记。在调用 feishu-task-manager meeting 前，先查 ~/.hermes/data/task-registry.json 确认哪些待办尚未被同步。详见 C012 skill 中的去重验证协议
