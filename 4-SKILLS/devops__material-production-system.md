# 材料制作系统 v1.0

> 材料制作是一项**独立于软件开发**的专业能力，不应混入 C005/C007 框架。
>
> 核心原则：
> - 顾问团专注策划（问需求、定框架、控风险）
> - 制作团队专注执行（写内容、讲故事、做视觉）
> - Hermes 一人分饰两阶段的全部角色，但换"布料"不换"人"

---

## 一、系统定位与核心价值

### 1.1 解决的问题

| 痛点 | 根源 | 本系统的解决方式 |
|:----|:-----|:----------------|
| 材料反复改、领导不满意 | 前期需求挖掘不足 | 4轮顾问团问询，先定方向再动手 |
| 写材料的人不知道领导偏好 | 缺乏用户画像 | 关键人画像逆向反馈回路持续喂养 |
| 材料质量参差不齐 | 无标准化质量门禁 | 5维度质量评分卡 + 异常熔断 |
| 多个版本改来改去无记录 | 无版本管理 | 飞书版本管理 + 版本演化分析 |
| 做了很多材料但没有沉淀 | 无知识复用 | 每份材料产出物入库，支持溯源 |

### 1.2 核心假设

```
一份优秀材料 = 精准的受众理解 × 清晰的逻辑框架 × 有力的叙事表达 × 一致的视觉呈现
                     ↑              ↑              ↑              ↑
                 顾问团阶段     顾问团阶段     制作阶段(2-3)   制作阶段(4)
```

**任何一个维度为0，整体质量归零**。

## 三、Pre-Step: 关键人画像预检

在进入顾问团4轮问询之前，必须先加载相关关键人的当前画像。

### 3.1 触发条件

当材料制作目标涉及以下场景时，必须执行预检：

| 场景 | 必查画像对象 | 预检优先级 |
|:-----|:------------|:----------|
| 向上汇报（翟总/老苏等领导场景） | 汇报对象画像 | 🔴 必查 |
| 跨部门协调 | 对方关键决策人画像 | 🟡 建议查 |
| 竞聘/述职 | 所有评委画像（已知） | 🔴 必查 |
| 团队内部材料 | 团队风格画像 | 🟢 可选 |

### 3.2 预检内容

从 C010 关键人画像系统获取以下信息：

```json
{
  "object": "翟总",
  "画像维度": {
    "思维风格": "全局型 / 细节型 / 战略型 / 执行型",
    "决策节奏": "果断 / 反复 / 授权型 / 拖延型",
    "沟通偏好": "数据驱动 / 叙事驱动 / 视觉驱动 / 文字驱动",
    "风险态度": "保守 / 平衡 / 激进",
    "信任半径": "高信任(结果导向) / 中信任(过程监督) / 低信任(亲自审阅)"
  },
  "近期行为信号": ["上一次汇报重点关注了第3页数据", "近期关注AI应用主题"],
  "禁忌区": ["不直接提竞争对手数据", "避免超过3个并列项"],
  "当前画像新鲜度": "2026-04-25 更新 (建议刷新)"
}
```

### 3.3 背景材料采集

当用户在顾问团第1轮中提到任何**附件文档**时（如「附件为《通知》《工作方案》等」），必须立即执行以下流程，不得继续问询：

1. 请用户发送文档（链接、token或文件）
2. 全部阅读完毕后再进入第2轮
3. 原因：文档可能定义了议程框架、已有分工方案或关键结论，跳过会导致提问用户已回答
4. 读完文档后，更新关键人画像和材料所需内容列表

**判断规则**：用户说「附件为……」或「这里有一份文档你看看」时，必须停下来读完再继续，不能跳过。

### 3.4 材料使用台账初始化

顾问团第1轮结束后、用户发来材料之前，必须**初始化材料使用台账**。台账是所有信息来源的追踪记录，格式如下：

```markdown
# 项目名 — 材料使用台账

> 任务ID: MP-YYYYMMDD-XXX

## 一、外部来源材料

| # | 材料名称 | 来源 | 格式 | 用途 | 入库KB | 归档路径 |

## 二、知识库引用材料

| # | 材料名称 | KB路径 | 引用用途 | 入库KB |

## 三、产出物清单

| # | 文件 | 状态 | 说明 |
```

**维护规则**：
- 每引用或收到一份新材料，立即追加一条记录
- 角色1~4的每次产出，立即更新产出物清单状态
- 从知识库引用的内容在「知识库引用材料」表中记录，**与外部来源分开**
- 材料制作完成后，所有源文件归档到 `~/wiki/raw/materials/<项目名>/`
- 台账文件本身也存于该目录下，命名为 `材料使用台账.md`

### 3.5 新鲜度检查

如果画像最后更新时间 > 30天，在顾问团问询第2轮中增加2个定向问题：

```
Q1：近期该领导的关注焦点有变化吗？（新战略？新业务？）
Q2：上一轮汇报的反馈中，有没有让你觉得「领导换了思路」的时刻？
```

## 四、阶段一：材料顾问团（4轮专属问询）

与C007软件顾问团的核心区别：

| 维度 | C007 软件咨询 | 材料顾问团 |
|:-----|:-------------|:----------|
| 问题焦点 | 功能/技术/数据/架构 | 受众/目标/叙事/风险 |
| 最终产出 | 系统方案文档 | 制作命令书 |
| 成功标准 | 系统能运行 | 领导满意、通过审批 |
| 检查机制 | Unit Test + 质量门禁 | 5维质量评分卡 |
| 回退机制 | 代码回滚 | 版本降级 + 熔断 |
| 关键风险 | 技术不可行 | 方向不对、用错叙事 |

### 第1轮 — 目标与场景分析（首席策略）

**问题集**（区别于C007的「目标层澄清」）：

```
Q1：这份材料的最终目标是什么？
  - 争取资源 / 汇报阶段性成果 / 申请审批 / 述职晋升 / 信息同步

Q2：材料的受众是谁？（单人或多人）
  - 单人汇报 → 锁定画像
  - 多人评审 → 列出关键决策人及其冲突点

Q3：受众看到这份材料后，你希望他们做什么？
  - 当场批准 / 给出反馈 / 记住某个点 / 信任你的判断

Q4：这份材料的上限和下限是什么？
  - 「最好能达成」 vs 「绝对不能搞砸」

Q5：汇报的时间和场合？
  - 正式汇报 / 会前沟通 / 邮件附送 / 口头配合
```

**输出**: 目标-受众-场景三元组

**⚠️ 重要陷阱 — 背景材料先行**

在继续第2轮之前，务必检查：
1. 用户在第1轮回答中是否提到了任何**已有文档**（通知、工作方案、过往纪要等）
2. 如有，必须先完成 **Pre-Step 3.3 背景材料采集**，阅读完毕后再进入第2轮
3. 原因：文档可能定义了议程框架、已有分工方案或关键结论，跳过会导致第2轮问出用户已回答过的问题

**✅ 正确做法模板**：
```
用户：Q2：附件为《成立EAST专项工作组的通知》《工作方案》……
助手：好的，请把这几份文档的链接或token发我，我读了再继续后面的问询。
```

  "goal": "争取AI应用专项预算审批",
  "audience": {

### 第2轮 — 受众画像与偏好映射（领域分析）

**问题集**：

```
Q6：该受众最吃哪套叙事？
  - 「数据对比」— 过去vs现在 vs 目标
  - 「故事线」— 挑战→行动→成果
  - 「问题驱动」— 痛点→方案→价值
  - 「战略叙事」— 趋势→定位→路径

Q7：该受众最反感什么？
  - 废话 / 不确定性 / 过度乐观 / 只有困难没有方案
  - 技术细节过多 / 战略空洞 / 3页才到重点

Q8：受众是否已经对该主题有先入为主的判断？
  - 支持 / 反对 / 尚未形成 / 被误导过

Q9：汇报时谁会先说、谁掌控节奏？
  - 领导引导型 / 汇报人主导型 / 评审提问型

Q10：以往类似材料被接受/拒绝的原因是什么？
  - 从历史版本演化分析提取
```

**输出**: 沟通策略

```json
{
  "narrative_preference": "数据对比型（翟总）",
  "avoid_list": [
    "无数据支撑的定性表述",
    "超过3个并列优先级",
    "只在最后一页写结论"
  ],
  "pre_existing_bias": "翟总认为AI落地风险高，需先打样试点",
  "flow_control": "汇报人主导型（15分钟=10min演讲+5min问答）",
  "historical_pattern": "上月AI专题报告因缺乏ROI测算被否，本次必须前置"
}
```

### 第3轮 — 框架与叙事选择（内容架构）

**问题集**：

```
Q11：材料的核心逻辑是什么？（选1个叙事框架）
  框架A — 「痛点-方案-价值」：问题驱动型
  框架B — 「目标-行动-成果」：成果驱动型
  框架C — 「趋势-定位-路径」：战略驱动型

Q12：需要突出哪些关键数据？（不超过3个）
  每个数据必须能回答「所以呢？」

Q13：材料的节奏感怎么设计？
  - 开头狠——第一页就抓住注意力
  - 中间实——每页只说一件事，配一个论据
  - 结尾收——最后一页是清晰的动作指令

Q14：有没有「必须说但不能直说」的内容？
  - 如：暗示竞争对手的劣势、委婉指出资源不足

Q15：是否需要预留「追问通道」？
  - 某页内容故意留有口子，引导受众追问你想让他们知道的事
```

**输出**: 叙事框架选择 + 页面节奏设计

**⚠️ 核心陷阱 — 叙事根基必须建立在事实复盘上，而非风险预测**

在确定关键信息的高潮页（通常是引出决策诉求的那一页）时，最常见的错误是**把材料的高潮建立在「未来风险预测」叙事上，而非「已发生的客观事实」推导上**。

| 错误案例（未来预测） | 正确案例（事实复盘） |
|:---------|:---------|
| 「EAST2.0升级在即，当前核心瓶颈是责任边界不清」→ 听起来是预判风险、制造焦虑 | 「二季度实施过程中暴露了X类突出问题 → 追根溯源，核心堵点是三方责任边界不清」→ 用事实推导结论，无法反驳 |

**判断规则**：
- 如果材料目标涉及 **推动决策 / 确认方案 / 争取资源** → 叙事根基必须是**复盘已发生的问题**，而不是预测未来风险
- 预测风险的叙事会让听众觉得「还没发生的事先不急」，缺乏紧迫感
- 复盘的叙事以已发生的事实为基础，具有**不可辩驳性**——事实已经发生，必须解决
- 模板：**「这X个月来暴露了A、B、C三类问题 → 透过现象看本质，根因都在边界不清 → 因此必须马上确定划分方案」**

**例外**：只有当材料类型明确是「战略规划」或「风控预警」时，才适用未来预测叙事。如果是「阶段性汇报+推动决策」类型的材料，必须用复盘叙事。

**⚠️ 次级陷阱 — 「协作框架」而非「责任界定」叙事**

当材料的高潮要提出一个**动作方案**（责任划分、分工调整、流程重构）时，叙事的语气差异决定了领导的第一反应：

| 错误表述（问责语气） | 正确表述（协作语气） |
|:---------|:---------|
| 「明确三方责任」/「启动问责」/「谁出问题谁担责」 | 「协作框架：让每件事都知道该找谁、怎么做」 |
| 领导听到的第一反应：防御心态、担心背锅 | 领导听到的第一反应：这个能解决问题、支持 |
| 落地时间线终点：四季度启动问责 | 落地时间线终点：三季度优化调整 |

**判断规则**：
- 用「协作框架」替代「责任界定」——定性不同，效果天差地别
- 用「案例推演」替代「规则宣读」——拿一个实际发生的问题，按新规则走一遍，比讲10条规则更有说服力
- 时间线终点用「优化调整」替代「启动问责」——给执行留出磨合空间，体现建设性而非惩戒性

### 第4轮 — 验证与确认（综合评审）

**此轮不做新提问，而是输出制作命令书草稿给用户确认。**

**验证清单**：

```
□ 目标-受众-场景三元组是否完整？
□ 叙事框架是否匹配受众偏好？
□ 关键数据是否不超过3个？
□ 风险熔断清单是否覆盖了所有「红线」？
□ 是否预留了「追问通道」？
□ 所有禁忌项是否已被排除？
```

**输出**: 用户确认后的制作命令书

**职责**：为每一页填充具体文字，提炼数据故事，设计「一句话记忆点」

**关键技法**：

```
1. 每页只说一件事 → 每页标题就是结论
2. 每个数据都要有「所以呢」 → 数据+洞察成对出现
3. 「三句话」写一个论点 → 现状→问题→建议
4. 首页有hook，末页有action
5. 善用「对比」→ 「转折」→ 「结果」三段式
```

### 角色3：视觉指挥（设计配色、排布局）

**职责**：制作实际交付物，兼顾专业度与飞书/PPT的渲染兼容性。

**设计原则**：

```
1. 深色科技风主基调（与Hermes体系一致）
2. 每页1个主视觉元素（图表/对比图/流程图）
3. 字体层级清晰：标题≈28pt / 正文≈16pt / 注释≈12pt
4. 留白 ≥ 30% 页面
5. 同一色系，不超过3色
```

**输出**：实际交付文件

- PPT: `.pptx` 文件（使用 PowerPoint API 生成）
- 飞书文档备选: `.md` → Feishu doc（当PPT不是强制选项时）

#### 角色3输出流程：PPT提示词先行审查

**重要规则**：在生成最终 `.pptx` 文件之前，必须先输出**PPT完整制作提示词**（文本格式），供用户审查内容后，再生成实际PPT文件。

**原因**：
- 用户需要先看到**全部内容**（每页标题、关键数据、口语脚本），再决定视觉呈现
- PPT生成后修改成本高，文本阶段可快速调整内容
- 用户可以借此机会补充缺失数据（用 `【待补充】` 标记）

#### 数据来源原则：优先使用客观数据

在为每一页填充关键数据时，按以下优先级选择数据来源：

| 优先级 | 来源类型 | 例子 |
|:-------|:---------|:-----|
| 🥇 首选 | **第三方/监管/审计数据** | 监管质检报告、审计结果、外部通报 |
| 🥈 次选 | **系统/工具自动生成的数据** | EAST月报、校验规则触发统计、异动监测记录 |
| 🥉 第三 | **会议纪要/工作记录中确认的数据** | 会议中各方确认的完成率、问题状态 |
| ❌ 避免 | **纯自评/内部推算的定性数据** | "我们认为改善显著"、"感觉上变好了" |

**案例**（来自实际项目）：汇报数据质量改善时，不用「异常完成率趋势」（自评指标），改用「监管质检报告中的异常数量变化」——这是第三方数据，更有说服力，且无法被质疑"自吹自擂"。

**判断规则**：如果某一页的关键论点要靠「自评数据」支撑，先问自己——有没有监管/审计/系统自动记录的数据可以替代？有则用，没有才退而求其次。

**PPT制作提示词文档结构**：

```markdown
# 材料标题 — PPT完整制作提示词

> 本文件是PPT生成的完整素材。缺失数据已用【待补充】标记。

## 【整体视觉规范】
- 风格、背景色、强调色、字体层级等

## P1 — 封面
【布局描述】【页面元素】【口语脚本】

## P2 — ...（逐页）

## 附：待替换数据清单
| 位置 | 替换字段 | 建议来源 |

## 附：口语脚本（每页引导语）
```

**审查流程**：
1. 角色3先输出**PPT制作提示词文档**（markdown格式）
2. 将文档上传至飞书云盘对应项目文件夹
3. 通知用户审查，等待确认
4. 用户确认后，再生成 `.pptx` 文件
5. 生成的 `.pptx` 同样上传至飞书云盘

**注意**：不要在用户确认提示词之前生成PPT文件。

### 角色4：质量门禁（评分卡校验、熔断检查、交付审计）

**职责**：最终交付前的质量把控

#### 5维度质量评分卡

| 维度 | 权重 | 检查项 | 计分规则 |
|:-----|:----|:------|:--------|
| **方向对齐** | 30% | 是否匹配制作命令书的目标、受众、叙事框架？ | 每偏离1项 -10% |
| **数据可信** | 20% | 每个数据有来源？数据故事完整（数据+洞察）？ | 缺来源 -5%/个 |
| **叙事有力** | 20% | 首页有hook？末页有action？每页标题就是结论？ | 缺hook -10% |
| **视觉一致** | 15% | 配色一致？留白充足？字体层级清晰？ | 每不一致 -5% |
| **简洁精炼** | 15% | 每页只说一件事？页数在范围内？无废话段落？ | 过密 -5%/页 |

**评分结果解释**：

```
≥ 90分 → 直接交付
80-89 → 标注改进建议后交付（非阻断）
70-79 → 打回第2/3角色修改（阻断）
< 70  → 触发熔断：返回第1轮顾问团重新讨论方向
```

#### 异常熔断机制

| 触发条件 | 动作 |
|:---------|:-----|
| 用户说「再想想」 | 自动熔断 → 回到顾问团第2轮，重新映射受众偏好 |
| 领导第2次看同一材料 | 自动熔断 → 提示「领导看了两次还没决策→建议切换叙事框架」 |
| 材料被要求大幅修改（>30%页面） | 自动熔断 → 回到第1轮验证目标是否偏移 |
| 交付后领导说「没说到点子上」 | 触发逆向反馈 → 关键人画像更新 + 顾问团复盘 |

从修改意见中提取沟通风格标签：

| 反馈原文 | 提取特征 | 建议更新画像维度 |
|:--------|:---------|:----------------|
| "这个数据来源可靠吗？" | 溯源癖、安全导向 | 沟通偏好→需附来源说明 |
| "少写篇幅，我要结论" | 结果导向、时间敏感 | 决策节奏→加速型 |

| "按这个方向推进" | 授权信任 | 信任半径→高信任 |

### 8.2 飞书云盘目录结构

每个材料制作项目完成后，必须在飞书云盘建立**独立项目文件夹**，结构如下：

```
Hermes生成文件夹 (FEISHU_LLM_FOLDER_TOKEN)
  └── 🗂 材料制作                                    ← 根目录
       ├── 🗂 项目名A（如：EAST二季度工作会）           ← 每期材料一个子目录
       │    ├── 📄 【PPT素材】制作提示词
       │    ├── 📄 【纲要】内容架构
       │    ├── 📄 【文稿】逐页口语脚本
       │    ├── 📄 材料使用台账
       │    └── 📄 v1_幻灯片.pptx
       └── 🗂 项目名B（如：苏总汇报）
            └── ...
```

**创建规则**：
1. 使用飞书API `POST /open-apis/drive/v1/files/create_folder` 创建文件夹
2. 文件夹取名规则：以项目名称命名
3. 所有交付物均以**飞书文档**形式存入（通过 `POST /open-apis/docx/v1/documents` + `folder_token`）
4. 文档分享给用户（`open_id`）
5. 每份文档的token登记到 `feishu-tokens.json` 注册表中

**文档命名规范**：
- `【PPT素材】` + 项目名 — 角色3的PPT制作提示词（先审）
- `【纲要】` + 项目名 — 角色1的内容架构
- `【文稿】` + 项目名 — 角色2的逐页文稿
- `材料使用台账` — 所有来源+产出记录
- `v1_项目名.pptx` — 最终PPT文件

```
用户说「帮我写一份XX材料」 → 自动进入材料制作系统

判断标准：
- 涉及「汇报」「材料」「PPT」「竞聘」「述职」「周报」等关键词
- 产出物是文档/PPT/汇报稿 → 走本系统
- 产出物是代码/系统 → 走C005/C007
```

### 多任务队列管理

当用户同时提出**多个材料需求**时（常见场景：一份汇报苏总、一份季度会材料），按以下规则处理：

#### 处理流程

1. **立即确认队列**：用表格列出所有材料任务，让用户一目了然
2. **用户指定优先级**：询问「先做哪份？」
3. **挂起任务登记**：未启动的任务标注为「待启动」，在后续会话中主动提醒
4. **逐份完成**：不建议并行（会污染上下文），一份做完再启动下一份

#### 用户典型表达

| 用户表达 | 含义 | 动作 |
|:---------|:-----|:------|
| "也纳入清单，先开始做X" | 队列管理 + 优先级指定 | 将Y挂起，立即启动X |
| "这两份都要做" | 全量队列 | 展示全部后确认优先级 |
| "先做X，Y后面再说" | 单步推进 | 只启动X，Y记忆到后续提醒 |

#### 例外规则

- 如果两份材料受众相同（如都是苏总）、时间相近（同一周），可考虑**合并为一份综合材料**
- 如果材料类型差异大（如一份周报、一份竞聘），必须独立制作

#### 飞书待办集成

如果材料来源是飞书待办任务（C012），先用 `feishu-task-manager.py list-user` 列出全部任务，从中筛选「汇报材料」类任务。然后将任务的GUID、标题、描述写入队列清单。

### 完整流程

```
① Pre-Step: 关键人画像预检（5分钟）
② 顾问团第1轮: 目标与场景（3-5轮对话）
③ 顾问团第2轮: 受众与偏好（3-5轮对话）
④ 顾问团第3轮: 框架与叙事（3-5轮对话）
⑤ 顾问团第4轮: 验证与确认（输出制作命令书）
⑥ 用户确认（必须）
┌── 如果用户大幅度修改方向 → 回到①或② ──┐
↓                                           │
⑦ 角色1: 内容架构（搭骨架，输出内容纲要）   │
⑧ 角色2: 故事大师（填内容，输出逐页文稿）    │
⑨ 角色3: 视觉指挥（做交付物，PPT/飞书文档）  │
⑩ 角色4: 质量门禁（评分≥80？→交付；<70→熔断返回⑤或②）
⑪ 交付（飞书归档 + 用户通知）
⑫ 逆向画像萃取（版本演化+语言特征+情绪锚点）
⑬ 关键人画像更新建议（→ C010）
⑭ 经验总结入库（→ wiki/raw/materials/）
```

### 熔断点汇总

| 检查点 | 条件 | 动作 |
|:-------|:-----|:-----|
| ① 用户确认 | 重大方向修正 | 返回顾问团①或② |
| ⑩ 质量门禁 | 评分≥90 | 直接交付 |
| ⑩ 质量门禁 | 评分80-89 | 标注改进后交付 |
| ⑩ 质量门禁 | 评分70-79 | 打回⑦或⑧修改 |
| ⑩ 质量门禁 | 评分<70 | 熔断返回⑤或② |
| ⑩ 质量门禁 | 用户说「再想想」 | 返回②重新映射受众 |
| ⑩ 质量门禁 | 领导第2次看同一材料 | 切换叙事框架 |
| ⑩ 质量门禁 | 修改>30%页面 | 返回①验证目标 |
