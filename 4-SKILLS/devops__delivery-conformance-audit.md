# 设计合规审定（Delivery Conformance Audit）

> 回答「交付物是否符合原设计」的结构化审计流程。

## 适用场景

- 用户问「请顾问团审定XX系统的交付情况是否符合原设计」
- 用户问「交付物和原方案有没有偏差」
- 需要验证已交付系统是否完整覆盖了设计规格中的所有需求
- 系统上线前或迭代交付后需要做合规检查

## 禁止场景

- 首次系统设计（走 C007 顾问团工作流）
- 单纯的 Bug 查找（走 `dogfood` 或 `product-quality-improvement-pipeline`）
- 开发实施（走 C005 编码工作流）

## 核心流程

```
Phase 1 ── 设计规格收集（强制前置步骤）

**⚡ 此阶段不可跳过。没有原型文件=没有审计基准=审计结论无效。**

```
前置检查：
  ├─ ① 确认本次审计涉及哪些设计文档/原型（从项目文档注册表获取文档ID清单）
  │
  ├─ ② 对每个文档ID，检查本地是否有已下载的 HTML 原型文件
  │   ├─ 有 → 验证文件完整性（大小≥5KB，且不为飞书包装页的 94-96KB）
  │   └─ 无 → 必须下载（用飞书 Drive API 下载端点，不可用 weblink）
  │
  ├─ ③ 验证下载正确性：
  │   ├─ 打开 HTML 文件检查内容是否包含完整页面结构
  │   ├─ 检查文件大小：纯 HTML 原型通常 5-25KB，不是 94-96KB 的飞书包装页
  │   └─ ⚠️ 所有文件下载后大小完全一致（如 94659 bytes）→ 说明下载的是飞书包装页而非原始文件
  │
  └─ ④ **在原型文件全部下载并验证前，不得进入 Phase 2**
  
向下执行条件：全部文档ID的原型文件已下载且验证通过
```

具体操作：
- 从会话历史搜索原设计方案（session_search）
- 从飞书/本地查找设计文档（search_files）
- **从飞书云盘下载原型文件**（用 Drive API，传 file_token → download 端点）
  - ⚠️ 不要用 web 链接（nofile.feishu.cn/file/xxx）下载——会返回飞书页面包装而非原始 HTML
  - ✅ 正确方式：`GET /drive/v1/files/{file_token}/download` + `Authorization: Bearer {tenant_token}`
- **原型文件验证**：下载后校验文件大小（纯 HTML 原型通常 5-25KB，非 95KB 的飞书页面包装）
  - ⚠️ 常见错误：所有文件下载后大小完全一致（如 94659 bytes）→ 说明下载的是飞书包装页而非原始文件
- 从 C005 artifacts 查找交付物
- 输出: 结构化设计需求清单（P0/P1/P2 优先级保留）

Phase 2 ── 实际系统探测
  ├─ 2A: API 层测试（全部端点 curl）
  ├─ 2B: 数据库一致性审计
  ├─ 2C: curl 端到端全链路测试
  ├─ 2D: 浏览器页面渲染测试（browser_navigate + snapshot）
  ├─ 2E: 边界条件测试（空数据/错误状态）
  └─ 输出: 功能实测 + 数据一致性 + 全链路结果清单

Phase 2B ── 数据库一致性审计
  ├─ 检查所有表的行数（是否为空/有意义）
  ├─ 检查外键引用完整性（孤立记录检测）
  ├─ 检查表间数据量一致性（如每个 schedule 的 question_count 与实际 practice_records 数）
  ├─ 检查跨表字段一致性（如 schedule.status = 'completed' 应有对应 answer 记录）
  ├─ 检查引用完整性：
  │   └─ SELECT * FROM records WHERE fk_id NOT IN (SELECT id FROM parent)
  ├─ 检查旧系统遗留数据（迁移后的数据污染）
  ├─ ⭐ 检查"下游数据消费者"是否连接
  │   ├─ 不仅仅是表存在、字段正确——要验证数据是否真正能流入这些表
  │   ├─ 追踪"写路径"：用户交互动作（答题/打卡/切换）→ 哪些表被 INSERT/UPDATE？
  │   └─ 常见缺失：
  │       1. wrong_questions 表 schema 正确但 schedule_answer 路由从未写入（答错不记录）
  │       2. practice_schedule.status 始终 'assigned' 从未被修改为 'in_progress'/'completed'
  │       3. daily_checkins 表存在但没有任何路由触发 INSERT
  │       → 每个数据表至少有一个写入点被生产端动作触发，不满足即为 Bug
  └─ 输出: 数据库一致性状态报告（含孤立记录数 + 消费者连通性）

Phase 2C ── curl 端到端全链路测试
  ├─ 构建结构化测试套件，按阶段组织（推荐10-100+用例）
  ├─ 每阶段测试相关页面的 HTTP 状态 + 响应内容关键词
  ├─ 测试流程完整性（登录→练习→答题→完成→报告→历史）
  ├─ 使用 curl cookie jar 模拟浏览器会话
  ├─ 处理蓝图 URL 前缀中的动态路径参数（如 /parent/chinese/<int:student_id>/...）
  ├─ 使用 session_search 或 session_start 保持 session 测试
  ├─ 同时对同一路由测试多重关键响应条件（200/302 + 内容 + 数据更新）
  ├─ 创建独立测试脚本文件，便于重复运行
  ├─ 参考: references/curl-full-link-testing.md
  └─ 输出: 完整通过的测试套件（推荐目标：100% 通过）

Phase 3 ── Bug 定位与修复（如发现时）
  ├─ 定位根因（文件+行号）
  ├─ 评估修复范围
  ├─ 执行修复（patch 或 C005 Tier 0/1）
  ├─ 验证修复生效
  └─ 输出: Bug 修复记录

Phase 4 ── 规格对照矩阵
  ├─ 逐行对照：设计规格 → 实测结果 → 是否符合
  ├─ 分类记录：
  │   ├─ ✅ 达标 — 完全符合设计
  │   ├─ ⚠️ 部分达标 — 功能存在但未完全按设计实现
  │   ├─ ❌ 未达标 — 功能缺失或与设计冲突
  │   └─ ➕ 超额交付 — 超出设计范围
  ├─ 注意：不要只做 API 测对照。对每张数据库表、每个路由、每个核心模板都做实现 vs 设计文档的逐项比较
  └─ 输出: 完整对照矩阵表（表层+路由层+模板层）
```

## 对照矩阵模板

| # | 原设计规格 | 优先级 | 实测情况 | 结论 |
|:-:|:----------|:------:|:---------|:----|
| 1 | 规格描述 | 🔴/🟡/🟢 | 测试结果 | ✅/⚠️/❌ |
| ... | ... | ... | ... | ... |

**字段说明**：
- `#`：编号，与设计文档中的需求编号对应
- **原设计规格**：从设计文档中摘录的原始描述（逐字）
- **优先级**：保留原设计中的 P0(🔴)/P1(🟡)/P2(🟢)
- **实测情况**：Phase 2 的实际测试结果描述
- **结论**：
  - ✅ 达标 — 功能正常且符合设计
  - ⚠️ 部分达标 — 功能有但未完全覆盖设计
  - ❌ 未达标 — 功能缺失或实现错误

## 质量评分体系

| 维度 | 权重 | 评分标准 |
|:----|:----:|:---------|
| P0 功能覆盖率 | 40% | 全部达标=10分，每缺1项-2分 |
| P1 功能覆盖率 | 25% | 全部达标=10分，每缺1项-2分 |
| P2 功能覆盖率 | 10% | 全部达标=10分，每缺1项-2分 |
| 技术架构合规 | 15% | 完全符合设计=10分，有偏差-3分 |
| Bug 密度 | 10% | 0个P0 Bug=10分, 1个-5分, 2+个-10分 |

**综合评分**：∑(维度分 × 权重)

**结论判定**：
- ≥ 8.5 分：✅ 通过审定
- 6.0 ~ 8.4 分：⚠️ 有条件通过（需整改关键项）
- < 6.0 分：❌ 需整改后重新审定

## Phase 6（选做）：领域功能完整性评估 — 「做对了，但做对了吗？」

> 本阶段在 Phase 1-5 的合规审定（「做对了没有」）完成后，由用户明确提出需要「从领域功能完整性的角度评估」时启动。
>
> **核心问题不同**：
> - Phase 1-5：交付物是否符合原设计规格？（设计→实现）
> - Phase 6：这个系统从领域角度看是否完整？（领域→设计→实现）

### 适用条件

- Phase 1-5 已通过（合规审定通过）
- 用户明确要求从功能完整性角度再评估（如「请以教育专家/领域专家的角度评估」）
- 该系统涉及特定领域知识（教育、医疗、金融、工程等）

### 核心流程

```
Phase 6 ── 领域功能完整性评估
  ├─ ① 领域基准定义
  │   ├─ 该领域的「理想完整系统」应该具备什么功能？
  │   ├─ 按能力层级分层（核心层/支撑层/扩展层）
  │   └─ 注意：这不是设计规格，是领域本身的客观需求
  │
  ├─ ② 当前系统映射
  │   ├─ 将实际功能映射到领域基准的每一层
  │   ├─ 标注：✅ 已覆盖 / ⚠️ 部分覆盖 / ❌ 未覆盖
  │   └─ 对「未覆盖」项判断：是设计遗漏还是刻意排除？
  │
  ├─ ③ 缺口影响评估
  │   ├─ 每个缺口对领域价值的影响程度（高/中/低）
  │   ├─ 区分：必须修复 → 架构级改造？还是增量优化？
  │   └─ 输出：缺口矩阵（领域层 | 功能 | 当前状态 | 影响 | 修复方式）
  │
  ├─ ④ 架构级 vs 功能级判别
  │   ├─ 功能级缺口：可增量添加（如增加一个新题型模板）
  │   ├─ 架构级缺口：需重新设计分层（如知识库与Web的数据分离）
  │   └─ 关键判断标准：能否在不改核心数据流的前提下修复？
  │
  └─ ⑤ 用户确认
      ├─ 将缺口按优先级分组（🔴架构重构 / 🟡功能新增 / 🟢优化）
      ├─ 列出需用户决策的问题（不超过3个）
      └─ 用户确认后 → 转为C005实施计划或新方案文档
```

### 领域专家角色激活

Phase 6 需要扩展 C007 顾问团阵容，按任务领域激活对应专家：

| 领域 | 核心专家 | 评估焦点 |
|:----|:---------|:---------|
| 🎓 教育/学习 | 资深教育专家 | 学习路径、认知规律、练习设计、间隔重复 |
| 🏗️ 软件系统 | 资深系统架构师 | 数据流、扩展性、层分离、技术债 |
| 🎨 UI/交互 | 用户体验专家 | 交互完整性、状态覆盖、任务流 |
| 🧪 质量 | QA工程师 | 测试完备性、边界覆盖 |
| 🏦 金融/合规 | 领域风控专家 | 合规要求、审计跟踪、数据完整性 |

### 关键分析框架：「按认知复杂度分层」

当 Phase 6 识别出系统存在「功能够用但架构不够优雅」的问题时，可使用以下分析框架评估是否需要重构：

```
操作类型               认知密度    推荐位置
──────────────────────────────────────
LLM/推理               🧠 极高     Backend
算法/知识图谱          🧠 高       Backend
统计分析              📊 中高      Backend
确定性值比对          ➕ 极低      Frontend（预置答案）
简单IO写入            📝 低        Frontend
纯展示                👁️ 最低      Frontend
```

**判断准则**：如果某个操作不需要大模型、不需要上下文、不需要算法决策，它就不应该放在后端。向前端移动简单操作可以消除网络延迟、降低后端压力、提供离线工作能力。

**实战案例**（来自 C003 数学模块重构）：在线批改原本是每次提交→后端查答案→返回，重构后后端在生成题目时预置答案 → 前端本地比对 → 零网络调用。

### 领域评估模板：以教育系统为例

当评估一个教育/学习类系统时，使用以下三层模型检查出题逻辑的完整性：

```python
三层出题模型（教育领域基准）：
  Layer 1 — 知识点选择：基于 Ebbinghaus 间隔 × 错误类型 × 掌握度 × 教学进度
  Layer 2 — 题目类型与难度：每知识点 ≥3 题型 × 3 级难度（基础/标准/拓展）
  Layer 3 — 具体生成：模板参数随机化 + 错题变式 + 解题步骤输出
```

**缺失检测清单**（示例）：
- 每知识点是否只有1个题型模板？→ 机械记忆而非真掌握 ❌
- 难度是否对所有学生固定？→ 无自适应 ❌
- 错题是否只重做不换角度？→ 背诵答案 ❌
- 有无间隔复习排期？→ 学过的知识点过期即忘 ❌
- 有无前驱依赖检查？→ 薄弱点可能根在更基础的知识 ❌

### 实战陷阱

1. **不要混淆「未实现」和「不需要」** — 领域功能完整性评估不是要求系统实现一切。有些功能可能是用户有意排除的（如过于复杂或超出需求范围）。关键是区分「设计遗漏」和「刻意排除」
2. **领域基准不是固定模板** — 每个领域的「完整系统」基准不同。教育系统需要三层出题模型，CRM 系统需要客户旅程管理——不要生搬硬套教育系统的模板到其他领域
3. **架构级缺口不可通过增量修复解决** — 如果发现「数据源不统一」「前后端职责错位」等架构问题，不要试图在现有架构上打补丁，应该先做架构方案再做实施
4. **领域评估后可能推翻原设计** — 这是正常的。如果原设计本身在领域层面就有缺陷，重构原设计比在原设计上叠加修改更可持续

### 模板-上下文变量验证（提交前必做）

**问题**：重新撰写路由（迁移 DB 查询）后，`render_template()` 可能传入正确的数据但使用了错误的变量名，模板不报错但无内容显示。

**信号**：模板中 `{% if selected_student and report %}` 没有匹配的分支，或内容区域为空。

**检查方法**：
1. 读取模板，提取所有模板变量引用（`{{ var }}`、`{% if var %}`、`{% for x in var %}`）
2. 读取路由的 `render_template()` 调用，逐项比对
3. 注意：模板变量可能来自父模板（`extends`）、继承块（`block`）、或模板中的 `set` 语句——不要只查当前路由，还要查 `{% extends %}` 中的父模板
4. 验证每个分支条件都有对应的双侧渲染可能性检查

**实战案例**（AI家庭教师 `reports_detail` 路由）：
```python
# ❌ 错误：传递了 student, schedules, subject_stats
return render_template('parent/reports.html', student=student, schedules=schedules, subject_stats=subject_stats)

# ✅ 正确：模板期望 selected_student 和 report（含各科统计+错题+建议）
return render_template('parent/reports.html', selected_student=student, students=all_students, report=report)
```

### SubAgent 可发现跨层不一致

当使用 SubAgent（`delegate_task`）执行数据迁移时，SubAgent 在处理迁移过程中可能发现调用链上下游的参数不一致（如路由与模板的变量名不匹配），这是 SubAgent 作为「不知情的局外人」天然具有的审查优势。

处理方法：
- SubAgent 发现的模板/路由不匹配 → 记录为**跨层不一致 Bug**，由主 Agent 决策修正
- SubAgent 完成迁移后，应向报告中明确指出此类问题（例如 SubAgent 发现 `reports_detail` 传 `student` 但模板用 `selected_student`）

## 实战陷阱（从 AI家庭教师家长端全量审计总结）

> ⚠️ **致命陷阱：审计不载入参考文件，仅凭代码分析判断** — 2026-05-20 用户纠正
>
> **错误做法**：接到"检查所有页面是否正确"的任务后，只做静态代码分析（检查 url_for 引用、旧表名、语法问题），然后宣告"已完成审计"。
>
> **正确做法**：
> 1. 第一步：**下载所有参考原型/设计文件**（从飞书 Drive API，不依赖 web 链接）
> 2. 第二步：**提取每份原型的关键特征**（页面结构、功能元素、数据展示方式）
> 3. 第三步：**逐一对照**（每份原型 → 对应路由/模板 → 差异记录）
> 4. 第四步：**只对逐页验证过的部分宣告完成**
>
> **根源**：静态代码分析只能发现"语法错误"和"明显的旧引用"，发现不了"功能不对、布局不对、数据展示方式不对"等设计偏差。只有逐页打开原型 HTML 文件对比才能发现这些问题。

## 实战陷阱（从 C003 Web v1.0 审定总结）

1. **设计文档可能不存在于文件系统中** — 设计可能在会话历史中而非文件中。要先 `session_search` 再 `search_files`
2. **不要猜测 API 端点路径** — 先检查实际路由注册（`grep -rn 'route\|blueprint'`），再逐一测试。猜测会导致误报"端点缺失"
3. **浏览器测试比 API 测试更能发现渲染问题** — API 返回 200 不代表页面可用。实际填写并提交才能验证完整流程
4. **发现 Bug 当场修复比后续再修复成本低得多** — 审定过程中发现的 Bug 可即时用 patch 修复，重新验证后纳入报告
5. **修复后必须清除缓存/重启服务** — 否则测试看到的是旧数据
6. **对照矩阵要逐字对照** — 不要概括设计规格，保留原文中的措辞细节，避免"近似匹配"导致漏判
7. **不要单独依赖 HTTP 200 作为页面可用性的唯一验证** — 200 不代表内容正确。增加内容关键词检查（`in r.text`）确认预期内容渲染
8. **数据库一致性要检查孤立记录** — 系统迁移后旧表遗留数据可能引用不存在的父记录（schedule_id 指向已删除的 schedule）。用 `SELECT * FROM child WHERE fk_id NOT IN (SELECT id FROM parent)` 检测
9. **蓝图 URL 前缀可能含动态路径参数** — 如 `url_prefix='/parent/chinese/<int:student_id>'`，测试 URL 必须携带该参数，否则返回 404。先检查路由注册（`grep -rn 'route\\|blueprint\\|url_prefix'`）
10. **新旧系统共存时要分别测试两套路由** — 旧系统路由（如 `/practice/<session_id>/...`）和新系统路由（如 `/practice/schedule/<id>/...`）可能同时运行，确保都覆盖
11. **设计对照不只是路由对照** — 逐张数据库表、逐条路由、每个核心模板都需要对照设计文档。常见遗漏：设计文档中的表存在但无数据、模板硬编码了本该动态计算的字段值
12. **测试脚本本身也有 Bug** — URL 格式错误、cookie 未传递、关键词匹配过严都会产生假阳性失败。先手动验证几个端点的真实行为再信任脚本输出
13. **⭐ 下游消费者连通性检验（常见缺失）** — 不要只检查表是否存在。要验证每个表的*写入路径*是否被用户交互动作触发。典型缺失：
    - `wrong_questions` 表 schema 正确但 `schedule_answer` 路由从不 INSERT（答错不记录）
    - `daily_checkins` 存在但无任何路由触发 INSERT（打卡功能不完整）
    - `practice_schedule.status` 始终 'assigned' 从未被修改为 'in_progress'（状态机不完整）
    - 审计方法：对每张用户交互涉及的表，执行"交互→DB快照"两步确认
14. **⭐ "隐藏功能"检测** — 后端计算了数据并传递到模板，但前端没有对应的 UI 触发入口。典型表现：模板中 `{% if X %}` 只有单分支（X=True 时显示，X=False 时无任何 UI）。审计所有条件分支的双侧完整性
