# 邮件→知识库分类管道

## 概述

企业邮件（以 **CC位为主**的管理角色）→ 13条规则 → 价值判定 → 选择性入库 → Phase2回复任务分发+风格学习。

```
邮件源 → 13条规则标记 → 价值判定 → 入库 → [Phase2] 需回复邮件→📬对外联络部 / 自身体件→风格学习
```

## 触发条件

当需要处理以下场景时加载此 skill：
- 新邮件进入需要判断是否进KB
- 需要对存量邮件进行回溯分类
- 需要从KB中检索需回复邮件并分发任务
- 需要分析自身邮件风格并更新画像
- 需要提取某时间段的关键事件时间线
- 需要判断转发链/回复链的保留策略

## 13条规则 → 结构化标记 + 价值判定

采用**自上而下优先匹配**的决策树，命中即终止，不继续判断后续规则。

所有邮件均执行 13 条规则的结构化标记。然后按价值判定决定是否入 KB。

### 规则表

| # | 条件 | 关注度 | 标记字段 | 价值 | 去向 |
|:-:|------|:------:|---------|:---:|:----:|
| ① | 发件人不属于正常人名的（系统通知/自动提醒等，**不含EAST数据治理专项工作组**） | **低** | `sender_type: "system"` | ❌ | 不入KB |
| **②** | **发件人是我（叶郁/yeyu-004）** | **低** | **`is_from_me: true`, `status: "replied"`** | **✅** | **入KB留存** |
| ③ | 发件人是 **苏金华 或 翟玉婷** | **极高** | `is_key_sender: true` | ✅ | 入KB |
| ④ | 收件人 + 抄送人 **总计 ≥ 20 人** | **低** | `is_broadcast: true` | ❌ | 不入KB |
| ⑤ | 邮件名含会议通知等特征，且**我在抄送人中** | **低** | `is_meeting: true`, `my_role: "cc"` | ❌ | 不入KB + 记会议时间表 |
| ⑥ | 邮件名含会议通知等特征，且**我在收件人中** | **中** | `is_meeting: true`, `my_role: "to"` | ❌ | 不入KB + 记会议时间表 |
| ⑦ | 发件人是部门内部人员，且**我在收件人中** | **高** | `sender_type: "internal"`, `my_role: "to"` | ❌ | 不入KB |
| ⑧ | 发件人是部门内部人员，且**我在抄送人中** | **中** | `sender_type: "internal"`, `my_role: "cc"` | ❌ | 不入KB |
| ⑨ | 收件人中**仅有我一人** | **极高** | `my_role: "sole_to"` | ✅ | 入KB |
| ⑩ | 收件人中**我是多收件人之一** | **高** | `my_role: "multi_to"` | ✅ | 入KB |
| ⑪ | 抄送中有我，且**收件人或抄送人中有翟玉婷或苏金华** | **中** | `my_role: "cc_with_leader"` | ✅ | 入KB |
| ⑫ | 抄送中有我，且收件人和抄送人中**均无**翟玉婷和苏金华 | **中** | `my_role: "cc_no_leader"` | ✅ | 入KB |
| ⑬ | **其余所有邮件** | **低** | 无特殊标记 | ❌ | 不入KB |

会议关键词匹配模式：`会议通知`、`会议邀请`、`会议安排`、`开会通知`、`关于召开`

### 决策树优先级总览

```
┌─ 预过滤：发件人非 @cpic.com.cn 域名？ → ❌ 不处理
│
① 发件人非正常人名的（系统/自动）？        → 低，不入KB
② 发件人是我？                             → 低，入KB
③ 发件人是苏金华/翟玉婷？                  → 极高，入KB
④ 收/抄送人总数 ≥ 20 人？                  → 低，不入KB
⑤ 会议通知+我在抄送？                       → 低，不入KB+记时间表
⑥ 会议通知+我在收件人？                     → 中，不入KB+记时间表
⑦ 部门内部+我在收件人？                     → 高，不入KB
⑧ 部门内部+我在抄送人？                     → 中，不入KB
⑨ 收件人仅我一人？                          → 极高，入KB
⑩ 收件人中我是多收件人之一？                 → 高，入KB
⑪ 抄送中有我+收件人/抄送人有领导？          → 中，入KB
⑫ 抄送中有我+无领导？                       → 中，入KB
⑬ 其余邮件                                  → 低，不入KB
```

### 关键约束

| 约束 | 说明 |
|------|------|
| CC邮件不需要回复 | 我在抄送人中的邮件 → Phase 2 不触发回复任务分发 |
| 非cpic邮箱直接忽略 | 发件人所在域名非 `cpic.com.cn` → 完全丢弃，不进入任何处理流程 |
| EAST数据治理专项工作组 | 视为部门内部同事，不被分类为系统发件人 |

## 全流水线（Phase 1 + Phase 2）

```
.eml 文件（Foxmail导出/WPS同步）
   ↓
13条规则 → 结构化标记（所有邮件都做）
   ↓
附件处理（所有邮件都做）：
  ├─ 文件名 → 存入邮件元数据（做钩子/关联索引）
  └─ 内容 → 向量化入 ChromaDB
   ↓
价值判定（哪些入 KB）
   ↓
生成邮件概况 → 发送 🗓️ 办公室群
   ↓
你确认
   ↓
按分类入库：
  ├─ ✅ 有价值 → ChromaDB KB
  ├─ ❌ 无价值 → 本地元数据（state.json）
  └─ 📅 会议通知 → 不入KB + 本地记会议时间表
      ↓
      === Phase 2（自动触发）===
      ├─ ⚡ 任务A：从KB查需回复邮件 → 发任务到 📬 对外联络部群
      └─ ⚡ 任务B（每周）：从KB查我发出的邮件 → 学习风格 → 更新画像
```

## 附件处理策略

每个附件同时做两件事：

```
附件文件
  ├─ 文件名 → 存入邮件元数据（attachments[] 字段，供索引钩用）
  └─ 内容 → 向量化入 ChromaDB（.xlsx/.docx/.pdf/.pptx 均解析）
```

检索时可双向勾兑：搜到邮件 → 附带文件名列表；搜到附件内容 → 追溯源邮件。

### 各区块详细度

| 区块 | 内容 | 详细度 |
|------|------|:------:|
| 🔥 需重点关注 | 发件人 + 时间(HH:MM) + 主题 + 📝 规则引擎提取2-3句关键内容（纯规则，零LLM） | **详版** |
| 🟡 值得关注 | 发件人 + 时间 + 主题，一行一封 | 简版 |
| ℹ️ 已回复 | 发件人 + 时间 + 主题（缩进，不自带编号） | 简版 |
| 📅 会议记录 | 时间 + 会议名称（缩进，不自带编号） | 简版 |
| 📌 知悉类 | 发件人 + 时间 + 主题 | 简版 |
| 📋 本批次全部 | 图标 + 发件人 + 时间 + 主题，按时间排序，含所有邮件 | **完整清单** |
| ❌ 不入KB邮件 | 仅在本批次全部清单中显示为 ⚪其他 | 完整清单中显示 |

### 确认流程

1. 每30分钟：扫描新邮件 → 13条规则分类（纯规则引擎，零LLM调用）
2. 有新邮件时，自动生成概况并投递到 🗓️ 办公室群
3. 无新邮件时保持静默（不投递）
4. 你在群内确认（或指出需修改项）
5. 确认后按价值判定规则入库
6. 手动调用 phase2_reply_dispatch.py 触发 Phase 2 回复分发

## ⚡ Phase 2：入库后自动处理

Phase 2 在每批邮件入库后自动触发，无需人工干预。

### 任务A：需回复邮件分发到 📬 对外联络部群

**触发时机**：每批邮件入库后自动执行。

**触发来源**：从 ChromaDB 查询最新批次中标记为"可能需回复"的邮件。

**判定逻辑**：

| 规则 | 条件 | 是否触发任务 | 回复对象→风格 |
|:----:|------|:-----------:|:------------:|
| ③ | 领导发件 | ✅ | 上级 → 强势结论型 |
| ⑨ | 仅我一人收件 | ✅ | 视发件人匹配风格 |
| ⑩ | 多收件人之一 | 🟡 LLM判断是否需回复 | 视发件人匹配 |
| — | CC邮件（规则⑪⑫） | ❌ **不需要回复** | — |

**发送到 📬 对外联络部群的格式**：

```
📮 邮件回复任务
━━━━━━━━━━━━━━━━━━
发件人：[姓名+职务]
主题：[邮件名]
时间：[YYYY-MM-DD]
━━━━━━━━━━━━━━━━━━
📝 邮件要点：
[2-3句话提炼核心内容及需回复的问题]
━━━━━━━━━━━━━━━━━━
🎯 回复对象：[(上级/跨部门/同事)]
📋 建议风格：[强势结论型/稳妥专业型/指令清晰型]
━━━━━━━━━━━━━━━━━━
💡 请拟回复内容（不需直接发送），完成后回复此消息
```

**风格匹配规则**（引用自 企业邮件起草·三风格矩阵）：

| 回复对象 | 选用风格 | 特点 |
|---------|:-------:|------|
| 苏金华、翟玉婷 | 强势结论型 | 先结论后原因，干脆果断，少修饰 |
| 跨部门/外部单位 | 稳妥专业型 | 结构完整，语气温和，分工明确 |
| 同事/部门内部 | 指令清晰型 | 简洁直接，行动项分明 |

**对外联络部产出**：只需提供回复内容草稿，不直接发送邮件。

### 任务B：每周自身体件风格学习

**触发时机**：每周一次，与你确认的频次一致，减少干扰。

**分析源**：从 ChromaDB 中查询 `is_from_me: true`（规则②）的最新批次邮件。

**分析维度**：

| 维度 | 具体指标 |
|------|---------|
| 称谓习惯 | 直呼姓名/带职务/用"您" |
| 开头模式 | 结论先行/背景铺垫/直接指令 |
| 段落结构 | 数字序号/自然段落/列表 |
| 数据呈现 | 嵌入正文/列表/表格（禁用） |
| 语气倾向 | 强势/稳妥/温和/指令性 |
| 附件标注 | 正文中标注/正文末说明/不说明 |
| 结尾公式 | 固定落款/行动项/后续安排 |

**输出更新**：更新 `style_profile.yaml`（位于 `~/.hermes_tools/email_classifier/style_profile.yaml`）。

**三风格矩阵（基准，来自用户经验）**：

| 场景 | 风格 | 特点 |
|:----|:-----|:-----|
| 翟总/强势领导 | 强势结论型 | 先结论后原因，干脆果断，少修饰 |
| 跨部门协作 | 稳妥专业型 | 结构完整，语气温和，分工明确 |
| 同事/平级 | 指令清晰型 | 简洁直接，行动项分明 |

**风格规范（基准，来自用户校正）**：

- 称谓：直呼姓名+职务（"窦总、晋武"），不用"各位同仁"等通用开头
- 段落标记：用数字序号（1、2、3）而非中文数字
- 子项：`- ` 短横线列表，不用编号嵌套
- 数据嵌入：关键数据直接嵌入文字，不另列表格
- 表格：禁用表格
- 语气：协作而非追责（"建议按以下分工"而非"请你们负责"）
- 附件说明：在正文明确标注附件内容

**风格画像文件格式**（`references/style_profile.yaml`）详见参考文件。

每次分析只更新 `evolution` 记录和 `global_traits` 中的新发现，不动已有基准。

## 会议时间线提取（辅助系统）

即使会议通知本身不进KB正文索引，也必须提取结构化时间线记录：

```yaml
- date: 2026-03-10
  type: 会议通知/会议纪要/工作安排/阶段性汇报
  subject: 关于召开分公司EAST报送工作会议的通知
  participants: 全国分公司财务/信息技术对接人(100+)
  attachments: []
  significance: EAST报送工作从财务部划转至数据管理部的宣导大会
  related_meeting_minutes: <doc_token>
```

**用途**：用于「某时间段关键事件回溯」查询，串起问题发现→方案决策→落地执行的时间链。

## 转发链保留规则

### 默认：保留链末

转发链（Fw_ / Fw_Fw_ / 转发_ 系列）一律保留**最后一封**。

### 替代触发

当链末满足任一条件 → 回退到最近一封含完整内容的邮件：
1. 链末剥除了附件 / 正文被截断
2. 链末仅为转发动作（"请查收""转发"等无实质内容）
3. 链中某封含重要新附件

### 回复链规则

Re_ 系列默认保留最后一封（结论/汇总）。
**替代触发**：中间某封含新附件/新信息 → 多封同时保留。

### 混合链规则

Fw_ + Re_ 混合 → 默认保留链末 → 替代按「最先含原附件」原则保留。

## 特殊情况处理

### 数据质量报告的多版本保留

**策略**：全部保留（作为后续报告对比素材），**新版不替代旧版**。
每封邮件记录元数据：月份、表数量、异常规则数、环比异动数。

### 操作手册类文档

**策略**：邮件正文可丢弃，**附件（操作手册）提取入库**。

### 附件云盘链接

发现邮件仅含云盘链接（无附件）→ 额外下载并归入知识库。

## 邮件分类速查

全部邮件通过 13 条规则决策树自动分类。详见 [13条规则表](#13条规则-→-结构化标记--价值判定)。

**入KB判定速查**：
- ✅ **入KB**: 自己发出的、领导发来的、仅我收件、多收件人之一、抄送有/无领导
- ❌ **不入KB**: 系统发件人、非cpic域名邮件、大群发(≥20人)、会议通知、部门内部、其余兜底
- 📅 **仅记时间表**: 会议通知（不入KB）

**回复触发判定速查**：
- 🔥 **触发回复**: 领导发件、仅我收件、多收件人之一（且我在TO位）
- ❌ **不触发回复**: CC邮件（含领导发来的CC邮件）、其余邮件

## 实现工具

### Foxmail 自动导出（公司电脑 Windows 端）

**脚本**：`~/.hermes/scripts/email/foxmail-auto-export.py`

从 Foxmail 本地存储（MBOX 格式 `.BOX` 文件）增量导出 `.eml` 到 WPS 云盘目录。

```bash
# 公司电脑 (Windows) - 嵌入式 Python，无需安装
C:\tools\python3\python.exe C:\tools\email-export\foxmail-auto-export.py          # 正常导出
C:\tools\python3\python.exe C:\tools\email-export\foxmail-auto-export.py --dry-run  # 探测路径
C:\tools\python3\python.exe C:\tools\email-export\foxmail-auto-export.py --status   # 查看状态
```

**部署**：Windows 任务计划程序 → 每 30 分钟触发（详见 `~/.hermes/scripts/email/README-WINDOWS-DEPLOY.md`）

**技术要点**：
- Foxmail In.BOX 是标准 MBOX 格式，Python `mailbox.mbox()` 直接解析
- 读取前先 `shutil.copy2()` 到临时文件，避免 Foxmail 写入冲突
- 按 Message-ID 增量追踪（存储在 `_export_state.json`）
- 文件名格式：`YYYY-MM-DD_HHMM_主题.eml`

## 故障排查指南

### 邮件"未被扫描"的第一检查项：路径对齐

当用户报告「邮件放入了但没被检查到」时，**第一步永远检查路径对齐**：

```
Foxmail 导出目的地         → 分类器扫描路径
E:\WPS云盘\邮件\*          ？  →  /mnt/g/WPSDocument/WPS云盘/邮件/自动导出/
```

**常见故障模式**：
- Foxmail 导出到 `邮件/` 根目录，但分类器只扫描 `邮件/自动导出/` 子目录 → 邮件静默堆积
- 用户手动将 .eml 文件拖入根目录而非子目录
- WPS云盘同步延迟导致文件虽在Windows端但未挂载到WSL

**诊断命令**：
```bash
# 1. 检查各目录中有多少 .eml 文件
ls /mnt/g/WPSDocument/WPS云盘/邮件/*.eml 2>/dev/null | wc -l
ls /mnt/g/WPSDocument/WPS云盘/邮件/自动导出/*.eml 2>/dev/null | wc -l

# 2. 检查分类器状态
python3 email_classifier.py stats

# 3. 检查 Foxmail 导出日志
cat /mnt/g/WPSDocument/WPS云盘/邮件/_export_log.txt

# 4. 检查 cron 是否正常运行
crontab -l | grep email
```

**修复**：如果邮件放在错误目录，移动到扫描路径即可。如需要修改扫描路径，调整 `email_classifier.py` 中的 `EMAIL_INBOX` 变量。

### 常见问题

| 症状 | 根因 | 修复 |
|------|------|------|
| 邮件文件存在但分类数为0 | 路径不匹配 | 检查EMAIL_INBOX配置 |
| 分类后数量不增加 | Message-ID已存在state.json | 正常，跳过重复 |
| _export_log.txt显示0封 | Foxmail .box文件路径变化 | 更新foxmail-auto-export.py路径 |
| cron扫描后state.json不更新 | 权限/路径问题 | 手动跑一次看报错 |

## 👤 管理者邮件分流与响应协议

### 场景定位

用户（数据管理部管理者）查看**已分类邮件**后的决策和响应动作。此阶段是入站分类管道的**下游环节**——邮件已完成五级分类和 Feishu 通知，现在需要呈现给用户做最终判断和响应。

```
邮件源 → 分类引擎 → Feishu通知 → [← 你在这里] 管理者分流 → 回复草稿/仅知晓
```

### 前提条件

- 已通过 `email_classifier.py scan` 完成邮件五级分类
- 通过 `state.json` 可查询每封邮件的 `role`（To/CC）、收件人列表、分类标签
- 通过 `.eml` 文件（在 WPS 云盘目录 `/mnt/g/WPSDocument/WPS云盘/邮件/`）可读取完整正文
- 部门同事名单（TO 中含此人即按"CC模式"处理）：曹爽、张照宇、张子越、庄经纬、王佳怡、孙蒨茹、芦新永、兰轶东、李显翠

### 双轨分流规则

| 你的角色 | 部门同事是否在TO中 | 处理方式 |
|---------|------------------|---------|
| **CC（抄送）** | 是，且邮件需回复 | 只需告知「作为部门管理者需关注什么方向」，不需要点摘录，给出方向性判断 |
| **TO（收件人）** | — | 告知邮件要点 + 如需回复则拟好回复草稿（纯通知类除外） |

### 第一步：角色分流（TO / CC）

从 `state.json` 查询：
```python
classified = data.get('classified', {})
for key, entry in sorted(classified.items(), key=lambda x: x[1].get('date',''), reverse=True):
    role = entry.get('role')  # 'To' 或 'CC'
```

### 第二步：CC 模式处理

1. 检查收件人(TO)中是否含部门同事
2. 若不涉及部门同事或无有价值内容 → 跳过或标注纯知悉
3. 若含部门同事 + 邮件需回复 → 提取邮件背景，判断「用户作为管理者需要盯什么方向」
4. 输出格式：邮件名 + 发件人 + 收件人中的部门同事 + 截止时间 + 1-2句需关注的方向

**典型输出**：
```
**集团主数据管理办法征求意见** — 曹爽(To)，夜雨CC
→ 5/21截止。关注曹爽汇总的部门修改意见是否覆盖EAST相关痛点
```

### 第三步：TO 模式处理

1. **排除纯通知类**（系统提醒、公告转发、定期周报等），纯通知类标注"无需回复"
2. **对需处理的邮件**：分析要点 + 判断是否需要回复
3. **如需回复** → 按三风格矩阵拟草稿，附在邮件要点后面
4. **如无需回复（只需知晓）** → 提取要点，标注"仅供知悉"

### 第四步：整体呈现

将两栏结果合并为一个结构化报告，末尾必须加 **「今日动作项汇总」**，按优先级排序（🔴 紧急 / 🟡 重要 / 🟢 常规 / ⚪ 备忘）：

```
--------------------------
🅰 我是抄送(CC)，收件人中有部门同事，需关注方向
--------------------------
[邮件A] — [关注方向]
[邮件B] — [关注方向]

--------------------------
🅱 我是收件人(TO)，要点+回复草稿
--------------------------
[需回复类]
[需回复类]
[纯知悉类]

--------------------------
⚙ 今日动作项汇总（按优先级）
--------------------------
🔴 事项1 — 动作
🟡 事项2 — 动作
🟢 事项3 — 动作
```

### 回复草稿风格匹配

| 发件人类型 | 风格 | 示例 |
|-----------|------|------|
| 部门内部同事（照宇/曹爽/蒨茹） | **简洁确认型** — 确认收到 + 给出下步安排 | "收到，我先看一下，下午碰个头。" |
| 跨部门/平级协作 | **稳妥专业型** — 结构完整，明确时间节点 | "收到。我们已安排排查，预计周五前反馈。" |
| 上级/需要批示 | **严谨汇报型** — 先说结论再汇报进度 | 直接引用三风格矩阵中的强势结论型 |

### 关键边界

- **什么算「纯通知类」**：系统自动提醒（IT云平台任务提醒、垃圾邮件隔离）、党务群团通知（学习通知、参观安排）、周期性统计（已读周报）
- **什么不算**：招标文件、审计反馈、数据修正、监管要求——即使看起来是通知也要打开确认内容
- **CC+部门同事但无实质回复需求**：标注为"仅知悉"不展开
- **同一发件人同一主题多封**：合并处理，以最新一封为准

### 与入站分类的关系（已升级为13条规则体系）

| 关注度标签 | 本协议如何映射 |
|-----------|--------------|
| 🔥 需重点关注（规则②③⑨⑩） | 自动触发 TO 模式「需回复类」→ 拟回复草稿 |
| 🟡 值得关注（规则⑪⑫） | 自动触发 CC 模式「需关注方向」→ 给出方向性判断 |
| ℹ️ 已回复（规则②自身发件） | 仅标注已回复，不触发分流 |
| 📅 会议记录（规则⑤⑥） | 仅记时间表，不触发分流 |
| ❌ 不入KB（规则①④⑦⑧⑬） | 跳过 |

## ✉️ 企业邮件起草（出站邮件）

> 本节整合自 `business-email-drafting` skill（已归档）。覆盖企业邮件/公函起草工作流——转发材料、协调分工、汇报进展、回复咨询。与入站五级分类构成邮件的**完整生命周期管理**（入→出）。

### 三风格矩阵（来自用户经验）

| 场景 | 风格 | 特点 |
|:----|:-----|:-----|
| 翟总/强势领导 | 强势结论型 | 先结论后原因，干脆果断，少修饰 |
| 跨部门协作 | 稳妥专业型 | 结构完整，语气温和，分工明确 |
| 同事/平级 | 指令清晰型 | 简洁直接，行动项分明 |

### 核心结构模板（跨部门协调/转发类）

```
主题：[动作]+[材料名称]+[补充说明]

收件人：[姓名+职务/直接对接人]
抄送：[上级领导/相关方]

---

[一、背景铺垫——1-2句话]
[何时何事]+[根据什么安排]+[做了什么]+[获得什么材料]。现将原文转发供参考。

[二、正文——按需要分段]
[问题/背景概述——用数据说话，避免主观评价]
[具体说明——按序排列，每条嵌入关键数据]

[三、行动方案——分责任方]
[责任方A]：[做什么]。[具体子项：]

[四、后续安排]
[正在做什么]，[预计何时]，[后续通知方式]。
```

### 风格规范（基于用户校正）

- **称谓**：直呼姓名+职务（"窦总、晋武"），不用"各位同仁"等通用开头
- **段落标记**：用**数字序号**（1、2、3）而非中文数字
- **子项**：`- ` 短横线列表，不用编号嵌套
- **数据嵌入**：关键数据直接嵌入文字，不另列表格
- **表格**：**禁用表格**——用户偏好文字列表
- **语气**：协作而非追责（"建议按以下分工"而非"请你们负责"）
- **附件说明**：在正文明确标注附件内容（"详见附件"）

### 流程步骤

1. **理解目的和受众** — 发什么？转什么？发给谁？抄送谁？
2. **确认关键信息要素** — 背景、核心数据、分工方案、后续安排
3. **按核心结构起草初稿**
4. **用户微调 → 定稿** — 记录用户改动偏好供下次复用

### 产出物格式

```
**主题**：[完整主题行]

**收件人**：[人名+职务]
**抄送**：[人名]

---

[完整的正文内容，最终版本]
```

用户确认后自行通过公司邮件系统发送。

### 常见陷阱

| 陷阱 | 正确做法 |
|:-----|:---------|
| 不知道具体人名 | 用 `[职务]` 占位，等用户填写 |
